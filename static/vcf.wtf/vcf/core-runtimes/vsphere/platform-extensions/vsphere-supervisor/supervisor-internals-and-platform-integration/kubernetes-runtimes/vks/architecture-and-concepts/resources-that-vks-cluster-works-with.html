Resources that VKS Cluster Works With

VCF Automation

 and 

VKS

 cluster management provide a hierarchy of objects to help you manage your resources. The four core resources that 

VKS

 cluster management uses are the organization, project, cluster, and the 

VKS

 workload cluster namespace.

Resource Hierarchy in 

VKS

 Cluster Management

 

VCF Automation

 Organizations

An organization is a top-level entity used to group and manage resources, users, policies, IaaS services, and catalog entities, while maintaining a secure boundary from other organizations. Organizations offer a centralized structure for billing, access control, and resource sharing, facilitating collaboration and governance across teams and departments. The organization typically correlates to a business or possibly a line of business within a large enterprise. Under the organization are grouping objects that allow you to separate the concerns of the people that use your resources.

VCF Automation

 Projects

Projects are logical representations of application teams and consist of users, groups, and resource pools they are entitled to. Projects enable you to organize your 

VKS

 clusters into logical groupings, for example, to align with business units.

However, projects are also tied to the underlying infrastructure. The resource pools that projects use are called 

vSphere Namespaces

. You can add or remove namespaces to a project at any time. Clusters within the same 

vSphere Namespace

 cannot be in different projects. Also, you cannot move a cluster between different 

vSphere Namespaces

.

By combining your resources into projects, you can simplify management. For example, you can apply policies or perform batch operations at the project level instead of working with separate clusters.

Since you can change the name of a project at any time, 

VKS

 cluster management uses the project ID as the unique identifier for all project-scoped resources.

You cannot delete a project if it contains a 

VKS

 cluster.

VKS

 Clusters

A 

VKS

 cluster is a vSphere-based Kubernetes cluster that runs your workloads.

For more information about 

VKS

 and 

VKS

 clusters, see the 

Managing vSphere Kubernetes Service with vSphere Supervisor

 documentation.

VKS

 Cluster Namespaces

Each 

VKS

 cluster can have one or multiple Kubernetes namespaces that you can view with 

VKS

 cluster management. Your 

VKS

 clusters can also have unmanaged namespaces. Unmanaged namespaces are namespaces that are created externally and that you do not manage with 

VKS

 cluster management.

VKS

 cluster namespaces are different from 

vSphere Namespaces

. A 

vSphere Namespace

 is a resource pool that comprises CPU, memory, and storage resources that the provider administrator allocates to organizations and projects in 

VCF Automation

. You do not manage 

vSphere Namespaces

 through 

VKS

 cluster management.

Synchronization of Organizations and Projects From 

VCF Automation

 to 

VKS

 Cluster Management

Upon deployment, 

VKS

 cluster management subscribes to the 

VCF Automation

 event broker service to synchronize organizations and projects. After subscribing to event broker, you get an event every time an organization or a project changes. This way 

VKS

 cluster management remains in sync with 

VCF Automation

. For any events that might get missed, there is also a regular periodic re-synchronization.

VKS

 cluster management is not available for VM Apps organizations in 

VCF Automation

. Those organizations and the projects in them do not get sychronized with 

VKS

 cluster management. For more information about VM Apps organizations and All Apps organizations in 

VCF Automation

, see Organization Management.

1) VKS Cluster Management: Hierarchy & ‚ÄúOrganization/Project/Cluster‚Äù

Yes ‚Äî VCF Automation introduces a hierarchical resource model that organizes VKS clusters under:

  • Organizations

  • Projects

  • VKS Clusters

This hierarchy exists in the VCF control plane and is exposed via the VCF Automation APIs/UI to help platform teams:

  • allocate quotas,

  • enforce policy,

  • view/manage clusters centrally

This is done as part of the VCF Automation resource model, not as native Kubernetes objects inside the VKS workload clusters themselves. Broadcom TechDocs

What this implies:

  • Organizational boundaries exist above the Kubernetes API level.

  • They are part of the VCF Automation control plane, not part of kube-apiserver for the VKS clusters.

  • This means these constructs are not visible via kubectl against a VKS cluster as CRs like PlatformMetaData or similar.

Analogy:
Think of Organization/Project as resource containers in a cloud control plane (like AWS Organizations + Accounts). Kubernetes doesn‚Äôt know or enforce them ‚Äî but the cloud control plane does. Similarly here: VCF Automation knows about Organization and Projects, but the VKS clusters themselves do not expose that model in their own Kubernetes API surface.

üß± 2) Kubernetes Namespaces inside Supervisor & VKS Workload Clusters

It‚Äôs important to separate three different namespace domains:

A) VKS Workload Cluster Namespaces (normal Kubernetes)

These are the namespaces you see when you do:

kubectl get ns

inside a VKS workload cluster ‚Äî this is ‚Äúreal native Kubernetes‚Äù and works exactly like any other cluster namespace (pods, services, roles, etc.). WilliamLam.com

These do not represent Organization/Project boundaries from VCF Automation ‚Äî they are just Kubernetes namespaces in the workload cluster.

B) Supervisor Cluster Namespaces

The vSphere Supervisor cluster also has namespaces ‚Äî these are used to scope:

  • System components (for VKS, VM Service, etc.)

  • Resource allocation pools

  • Workload provisioning contexts

You can see Supervisor namespaces via:

kubectl get namespaces

when connected to the Supervisor cluster‚Äôs control plane API endpoint. mpoore.io

These Supervisor namespaces are used as a container for VKS cluster objects ‚Äî but again, this is not the same as the VCF Automation Organization/Project construct. They exist inside the Supervisor‚Äôs Kubernetes API.

C) VCF Automation Organization/Project Context (external control plane)

This is a higher-level grouping used by VCF to track:

  • VKS clusters

  • quotas

  • policies applied at Org/Project level

These are not Kubernetes namespaces ‚Äî they are external catalog/metadata constructs and are only visible through VCF Automation APIs/UI and related governance surfaces. Broadcom TechDocs

So:

üß± 3) So Yes ‚Äî There Is a Synchronization / Event Mechanism‚Ä¶

VCF Automation coordinates:

  • cluster creation

  • cluster status changes

  • policies and quota enforcement

Between the Automation control plane and the Supervisor/VKS cluster APIs.
That‚Äôs why you see a mention of ‚Äúsynchronization / event brokerage‚Äù in the document: it‚Äôs the VCF Automation service syncing state between its own internal model (Organization ‚Üí Project ‚Üí Cluster) and the underlying Supervisor/VKS API objects. Broadcom TechDocs

But these sync events do not create new Kubernetes CRs inside a VKS cluster that represent Organization/Project names ‚Äî they exist in the Automation control plane.

üß± 4) Does ‚ÄúOrganization/Project‚Äù Exist in Supervisor Itself?

No, at least not as directly exposed organizational CRs in the Supervisor‚Äôs Kubernetes API.
What you do get in the Supervisor cluster:

  • Namespaces scoped for system services

  • Namespace/context for VKS clusters

  • Standard K8s objects and CRDs used internally by VKS (Cluster API, controllers), but nothing labeled as ‚ÄúOrg/Project‚Äù natively in kube-apiserver.

That is an out-of-band governance metadata layer, managed by VCF Automation, not in Supervisor‚Äôs Kubernetes API. Broadcom TechDocs

To reiterate:

  • Supervisor cluster = special embedded Kubernetes cluster for vSphere control plane and VKS services. Medium

  • VCF Automation Org/Proj = constructs in the VCF control plane to manage cluster fleets, quotas, policies. Broadcom TechDocs

  • Neither of these is a native namespace in VKS workload clusters.

üß± 5) Why It *Doesn‚Äôt Show Up via kubectl (Under Normal Circumstances)

There are three APIs involved:

  1. VCF Automation API/UI:

    • Exposes Org/Project ‚Üí VKS clusters

    • Does not live in Kubernetes API

    • You use the VCF automation UI or automation API to see and manage this.

  2. Supervisor Kubernetes API:

    • Exposes Supervisor cluster resources

    • Supervisor namespaces, system services, VKS cluster objects

    • You can use kubectl here if you point at Supervisor API.

  3. VKS Workload Kubernetes API:

    • The actual workload cluster that developers interact with

    • Standard Kubernetes namespace API

None of these expose an API object like:

kind: PlatformMetaData

that carries the higher-level Organization/Project mapping ‚Äî that mapping is external to the Kubernetes APIs, part of VCF Automation governance. Broadcom TechDocs

üåü Summary of Key Points

  • VCF Automation introduces a hierarchical model (Org ‚Üí Project ‚Üí VKS clusters) that is a control plane construct, not native Kubernetes. Broadcom TechDocs

  • Supervisor clusters have Kubernetes namespaces you can see via kubectl, but these are for system and resource scoping ‚Äî not the high-level Org/Project model. mpoore.io

  • Workload clusters have their own native Kubernetes namespaces ‚Äî standard kube namespaces separate from Supervisor. WilliamLam.com

  • Organization/Project metadata is not visible in the Kubernetes API of workload or supervisor clusters; instead, it lives in VCF Automation service APIs/UI

Short answer: a VKS Cluster does not directly ‚Äúconsume‚Äù or reason about that out-of-band metadata at runtime.
That metadata is used upstream, by the control plane, to decide how, when, and whether a VKS cluster is created, configured, governed, or changed.

Let‚Äôs unpack this precisely and mechanically.

The three layers involved (critical to keep straight)

1. VMware Cloud Foundation Automation

This is where Organization / Project / Cluster metadata lives.

  • Org, Project, ownership

  • Quotas

  • Policies

  • Approval flows

  • RBAC mappings

  • Lifecycle intent (‚Äúthis cluster belongs to Project X in Org Y‚Äù)

This layer is authoritative for that metadata.

2. vSphere Kubernetes Service (VKS)

This is the service that provisions and manages Kubernetes clusters on the Supervisor.

VKS:

  • receives instructions

  • does not reason about ‚Äúorganizations‚Äù as a first-class concept

  • enforces what it is told via concrete config

3. vSphere Supervisor

This is the execution substrate:

  • namespaces

  • resource pools

  • networking

  • storage

  • controllers

It enforces mechanics, not business structure.

So what does VKS actually do with that metadata?

Think of the metadata as compile-time inputs, not runtime signals

The Org / Project info is used to derive concrete, enforceable artifacts before or around the cluster ‚Äî not inside it.

Here‚Äôs how that plays out.

1. Cluster creation & placement (primary use)

When you say (implicitly or explicitly):

‚ÄúCreate a VKS cluster in Project A of Organization B‚Äù

VCF Automation uses that metadata to decide:

  • Which Supervisor Namespace the cluster is created under

  • Which resource pools (CPU/mem/storage) it may consume

  • Which networks it may attach to

  • Which policies apply by default

By the time VKS sees the request, it looks more like:

‚ÄúCreate a cluster with these limits, in this namespace, using these credentials‚Äù

So VKS never sees ‚ÄúOrganization = FooCorp‚Äù ‚Äî
it sees resolved, concrete constraints.

2. Policy materialization (guardrails ‚Üí config)

Org / Project metadata drives policies such as:

  • max node count

  • allowed VM classes

  • allowed storage policies

  • network isolation rules

  • upgrade windows

These are compiled down into:

  • admission constraints

  • Supervisor namespace settings

  • cluster spec defaults

  • lifecycle rules enforced externally

Once compiled, VKS enforces the result, not the intent.

3. RBAC mapping (humans, not pods)

Org / Project membership is used to decide:

  • who can create / delete clusters

  • who can resize them

  • who can see them in UI / API

But inside the cluster:

  • Kubernetes RBAC is still Kubernetes RBAC

  • there is no native concept of ‚ÄúOrg Admin‚Äù or ‚ÄúProject Owner‚Äù

Those roles are translated at provisioning time into:

  • kubeconfigs

  • initial role bindings

  • access paths

Again: translation ‚Üí enforcement ‚Üí forget the original label.

4. Lifecycle control (this is subtle but important)

Because the Org / Project model is authoritative:

  • deleting a Project can trigger cluster deletion

  • quota exhaustion can block scale-out

  • org-level policy changes can trigger reconciliation

But:

  • the decision happens in VCF Automation

  • VKS just executes lifecycle operations it is told to execute

This is why you see eventing / synchronization mentioned in the docs:
it‚Äôs about state reconciliation, not metadata propagation.

5. What VKS explicitly does not do

This is the part you were circling around ‚Äî and your instinct was right.

Inside a VKS workload cluster:

  • ‚ùå No Organization CR

  • ‚ùå No Project CR

  • ‚ùå No PlatformMetadata object

  • ‚ùå No way to kubectl get and discover:

    ‚ÄúWhich Org do I belong to?‚Äù

From a DevOps or in-cluster perspective:

The cluster is sovereign.

It has:

  • nodes

  • namespaces

  • workloads

  • policies

But no awareness of who ‚Äúowns‚Äù it at the platform governance layer.

This is very intentional.

6. Why this design is deliberate (and smart)

If Org / Project metadata were injected into clusters:

  • cluster portability would suffer

  • backup / restore would leak tenancy info

  • federation would get messy

  • GitOps would become environment-coupled

By keeping the metadata out-of-band:

  • clusters remain generic Kubernetes

  • governance stays centralized

  • enforcement happens at the edges (creation, mutation, deletion)

This mirrors how:

  • AWS Organizations ‚â† Kubernetes namespaces

  • vCenter folders ‚â† VM guest OS awareness

  • Cloud billing ‚â† workload runtime

Mental model (use this)

VCF Automation owns intent.
VKS enforces outcome.
Supervisor executes mechanics.

Once the outcome is materialized, the intent is no longer visible downstream.

Direct answers to your implicit questions

  • Does the metadata exist?
    Yes ‚Äî in VCF Automation.

  • Is it synchronized into the cluster?
    No ‚Äî only its effects are.

  • Can DevOps see it via kubectl?
    No ‚Äî by design.

  • Does the Supervisor know about Orgs/Projects?
    No ‚Äî only namespaces, pools, and policies.

TL;DR

VKS does not ‚Äúuse‚Äù Org / Project metadata at runtime.
It receives decisions derived from that metadata and enforces them.
The metadata itself never enters the Kubernetes API surface.
