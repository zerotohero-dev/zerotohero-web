VKS

 Cluster Management Policy Types

Policies allow you to provide a set of rules that govern your organization and all the objects it contains.

Five policy types exist in

VKS

 cluster management.

  • Security policy

    Security policies enable you to manage the security context in which deployed pods operate in your clusters by imposing constraints on your clusters that define what pods can do and which resources they have access to. For more information about security policies, see Managing Pod Security with VKS Cluster Management Security Policies.

  • Image registry policy

    Image registry policies enable you to specify the source registries from which an image can be pulled. For more information about image registry policies, see VKS Cluster Management Image Registry Policies.

  • Quota policy

    Quota policies enable you to constrain the resources used in your clusters, as aggregate quantities across specified namespaces, using preconfigured and custom templates. For more information about quota policies, see VKS Cluster Management Quota Policies

  • Mutation policy

    Mutation policies enable you to alter resources before they are admitted to the cluster. For more information about mutation policies, see VKS Cluster Management Mutation Policies.

  • Custom policy

    Custom policies enable you to implement additional business rules, using templates that you define, to enforce policies that are not already addressed using the other built-in policy types. For more information about custom policies, see VKS Cluster Management Custom Policies.

Out of these five policies, only the quota policy type is implemented by using the Kubernetes-native 

ResourceQuota

 objects,. The rest of the policy types use the open-source OPA Gatekeeper as the policy engine.

You can apply policies at any level of the hierarchy - the organization, the project, or cluster levels.

Policy Inheritance

In the 

VKS

 cluster management resource hierarchy, there are three levels at which you can create policies:

  • Organization

  • Project

  • VKS

     cluster

You can create direct policies at any level. But policies are also propagated down the hierarchy tree. Every object inherits the policies defined for its parent objects. For example, when you create a security policy at the project level, all clusters within that project inherit the applicable policy. Additionally, the project also inherits policy definitions from its parent organization, if there are policies created and applied to that organization.

VKS

 Cluster Management Policies and Gatekeeper

Some of the policy types in 

VKS

 cluster management are implemented using OPA Gatekeeper. These policy types include security policy, image registry policy, and mutation policy. When you implement one of these policies, 

VKS

 cluster management installs Gatekeeper on the cluster and maintains that installation while you have 

VKS

 cluster management policies applied to that cluster.

If there is an installation of Gatekeeper on the cluster that was not installed by 

VKS

 cluster management , then 

VKS

 cluster management does not alter or maintain that installation. In many cases, this situation does not impact the functionality of the policy. However, because Gatekeeper was not installed and configured by 

VKS

 cluster management , Gatekeeper-based policies implemented through 

VKS

 cluster management might not function as expected. This situation raises an 

Incompatibility

 insight that can be viewed on the 

Policy Insights

 page.

For best results, if you are using 

VKS

 cluster management policies on a cluster, remove external installations of Gatekeeper and let 

VKS

 cluster management manage Gatekeeper for you. For more information, see the official Gatekeeper documentation at https://open-policy-agent.github.io/gatekeeper/website/docs/.
