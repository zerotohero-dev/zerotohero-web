NSX Segment

NSX Segment in VMware Cloud Foundation

Short definition

An NSX Segment is a logical Layer-2 network in VMware NSX.
It is where workloads (VMs, containers) attach their NICs, just like plugging into a virtual switch—except it’s policy-driven, distributed, and decoupled from physical VLANs.

Where an NSX Segment sits in the stack

VCF
├── vSphere (compute)
│   └── VMs / Pods
│       └── vNIC
│           └── NSX Segment (L2)
│               └── Tier-1 Gateway (L3, optional)
│                   └── Tier-0 Gateway → Physical network
└── NSX Manager (policy & control)

  • Defined in: NSX Manager

  • Enforced on: ESXi hosts (distributed dataplane)

  • Consumes: Overlay networking (GENEVE)

What an NSX Segment actually does

Core responsibilities

  • Provides L2 connectivity between attached workloads

  • Enables east–west traffic within the segment

  • Acts as the policy anchor for:

    • Distributed Firewall (DFW)

    • DHCP (optional)

    • MAC/IP learning & isolation

Think of it as:

“A virtual Ethernet network with cloud-native policy.”

Segment vs VLAN (why segments exist)

Aspect VLAN (classic vSphere) NSX Segment
Scope Physical fabric–bound Fabric-independent
Creation Network admin Policy / API driven
Scale Limited by VLAN IDs Large overlay scale
Security Perimeter-centric Microsegmentation (DFW)
Mobility Constrained Portable across hosts

Segments remove VLAN sprawl and enable fine-grained security.

Attachment models (important nuance)

1. Segment with no gateway

  • Pure L2 broadcast domain

  • No routing out

  • Common for isolated tiers

2. Segment attached to a Tier-1 Gateway

  • L3 routing enabled

  • North–south possible via Tier-0

  • Most common in VCF

Segments and security (key point)

  • DFW rules are applied to vNICs on segments

  • Security is identity- and policy-based, not subnet-based

  • Zero-trust patterns become practical

The segment is not the security boundary — the workload is.

NSX Segment vs related constructs (avoid confusion)

Thing Is it a Segment? Why
Port Group (vDS) ❌ VLAN-backed, host-centric
Supervisor Namespace ❌ Kubernetes tenancy
Resource Pool ❌ CPU/memory scheduling
Subnet ❌ IP concept; segments may have one
Tier-1 Gateway ❌ L3 router, not L2 network

How VCF uses NSX Segments

In VCF:

  • Segments are first-class networking objects

  • Used by:

    • VM workloads

    • Kubernetes (via NSX CNI)

    • Automation blueprints

  • Lifecycle managed and validated by VCF

You don’t hand-craft switch configs; you declare intent.

Mental model

If vSphere Port Group is:

“A virtual switch tied to a VLAN”

Then NSX Segment is:

“A logical network with built-in policy, security, and mobility”

What an NSX Segment is not

  • ❌ Not a physical network

  • ❌ Not a tenant boundary

  • ❌ Not a router

  • ❌ Not Kubernetes-specific (but integrates cleanly)

Key takeaways

  • NSX Segment = logical L2 network

  • Attaches workloads and anchors security policy

  • Decoupled from physical VLANs

  • Foundational to zero-trust networking in VCF
