Managing Blueprints in VCF Automation

A blueprint is a declarative template used to define and automate the deployment of infrastructure, applications, and services. It allows users to model complex cloud resources, such as VMs, networks, storage, and software components. Blueprints orchestrate the deployment of multiple resource in the VCF platform. It acts as a reusable design for deploying resources and services consistently. A design canvas is used to create a blueprint. YAML code is automatically generated. It describes infrastructure as code. The code editor also allows to type code directly or enter values.

Blueprint Features

  • Blueprint designer.

     The Blueprint Designer offers a single interface to create blueprints for various resource types, including single machines, multi-machi deployments, and multi-tiered applications.

  • Declarative Infrastructure-as-Code.

     Blueprints define the desired state of the infrastructure using YAML or a visual designer, rather than specifying step-by-step instructions.

  • Reusability and versioning.

     Blueprints can be versioned, imported, exported (YAML format), and shared across projects or even tenants, promoting reusability and collaboration.

  • Self-Service provisioning.

     

    VCF Automation

     provides self-service portals where users can deploy and manage resources based on pre-defined blueprints.

  • Inputs and Customization.

     Blueprint inputs enable customization during deployment, allowing users to select options like machine size or image.

  • Governance.

     To ensure consistency, reusability, and governance across environments.

  • Day 2 operations.

     Blueprints support day 2 operations, allowing for updates, modifications, and management of existing deployments.

Blueprint Concepts

  • Versioning.

     Blueprints can be versioned. At deployment time, you can select any of your versions to deploy.

  • Resource objects.

     Resource objects represent the infrastructure components and services that are provisioned and managed in a blueprint.

  • Dependencies between resources in blueprints.

     When you deploy a 

    VCF Automation

     blueprint, one resource might need another resource to be available first.

  • Instances.

     Instances are created when blueprints are deployed from the catalog.

  • Day 2 actions.

     Actions that can be performed on the provisioned resource objects.

  • Property groups.

     Instead of defining the same properties repeatedly across multiple blueprints, property groups enable you to define them once and then link them to various resources. This promotes consistency and simplifies management, as updates to a property group are automatically reflected in all linked templates.

  • Property bindings.

     Sometimes, a resource property needs a value found in a property of another resource. For example, a backup server might need the operating system image of the database server that is being backed up, so the database server must exist first.

  • Custom forms.

     Allows users to design and customize request forms for catalog items, providing a tailored user experience for submitting requests. They can be designed with a drag-and-drop interface and configured with various properties, including conditional logic and external data sources.

  • Custom resources.

     

    VCF Automation

     supports the use of custom resources within blueprints to extend automation capabilities beyond native resource types. These custom resources are built upon 

    VCF Operations orchestrator

     workflows and can represent a resource.

When to Use Blueprints

Blueprints allow you to streamline and automate the deployment process, ensuring consistency and reducing manual effort.

With blueprints, you can model complex IT services by visually designing infrastructure components like VMs, networks, and storage. You can use blueprints when you want to hide the complexities of the infrastructure provisioning from the end user.

You can publish blueprints to the self-service 

VCF Automation

 catalog which allows users to request and deploy services based on predefined templates.

Best Practices for Designing Blueprints

  • Leverage input properties in blueprints to allow users to customize deployments while maintaining a degree of control. Consider providing default values for required fields to improve user experience and prevent errors.

  • Establish clear and consistent naming conventions for blueprints, catalog items, and other resources to improve manageability and organization.

  • Using an iterative method to build and version blueprints helps teams adopt Infrastructure-as-Code (IaC) practices. Start with a minimum viable blueprint (MVB) and add features iteratively. Additionally, use tools like Git for version vontrol.

Sharing Blueprints with Projects and Organizations

When you first create a blueprint, you associate it with a project so that when you publish the blueprint to the catalog, users in the selected project can see request and deploy the blueprint. You have the option of making the published blueprint available to other projects in the organization either when you create the blueprint or at a later time.

If you are managing a Provider Consumption organization, you can share blueprints with all other organizations under the same provider administrator or you can select specific organizations. After the blueprint is shared, whenever PCO administrator publishes a blueprint to the catalog, the catalog item is shared with the selected organizations. That means one blueprint backs multiple catalog items, one in each organization.

When sharing blueprints with other organizations, consider the following limitations.

  • Blueprints that contain any of the following resource types cannot be shared:

    • property groups;

    • custom resources;

    • secrets;

    • namespaces;

    • VM images from provider tenant content libraries.

  • If a blueprint is shared with an organization, a blueprint version with restrictions cannot be published to the catalog.

  • If a blueprint is not shared with any organizations yet, and it has versions that have restrictions and have been published to the catalog, the blueprint cannot be updated and shared with any organization.
