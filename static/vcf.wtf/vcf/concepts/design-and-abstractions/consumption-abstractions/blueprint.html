Blueprint

A Blueprint is a declarative deployment template that defines what can be requested, how it is configured, where it may land, and under which policies and approvals.

Blueprints live in VCF Automation.

Blueprints in VCF Automation are governed deployment templates that define allowed workload patterns, configuration inputs, placement constraints, and approval policies. They orchestrate resource creation but are not part of the runtime execution layer.

Once a deployment is created, the Blueprint can be deleted and the workload keeps running: It’s not part of the runtime layer.

User
  ↓
Blueprint   ← governance + intent
  ↓
Project     ← who / where / policy
  ↓
Supervisor Namespace
  ↓
vSphere / Kubernetes runtime

Kubernetes YAML answers:

“How do I create this thing?”
Blueprints answer:
“Who is allowed to create which thing, under what rules, and with whose approval?”

How Blueprints relate to Projects and Tenants

  • Organization: business boundary (e.g. Verizon)
  • Project: governance scope (e.g. US-East)
  • Blueprint: allowed patterns inside that scope
Organization: Verizon
└── Project: US-East
    ├── Blueprint: Standard App VM
    ├── Blueprint: GPU Pod
    └── Blueprint: 3-Node K8s Cluster

Each Blueprint:

  • targets specific Supervisor Namespace(s)
  • enforces constraints
  • hides complexity from requesters

What Does a Blueprint Define

Typically it defines:

Resources to create

Examples:

  • A VM
  • A vSphere Pod
  • A Guest Kubernetes cluster
  • A multi-tier app (VM + LB + storage)

Configuration inputs

  • CPU / memory sizes
  • Environment variables
  • Image versions
  • Feature toggles
  • Secret references (not secret values)

Constraints & policies

  • Which Supervisor Namespace(s) are allowed
  • Which storage policies are allowed
  • Which networks are allowed
  • Size limits
  • Placement rules

Governance hooks

  • Who can request it
  • Who must approve it
  • Lease duration
  • Cost visibility / chargeback tags
