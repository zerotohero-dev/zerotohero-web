VM Workload and Pod Workload

In VCF, a workload is:

A unit of compute that consumes CPU, memory, storage, and networking, and is subject to lifecycle, policy, and governance.

VCF doesn’t care what the workload is at a philosophical level. It cares about:

  • How it is instantiated
  • How it is isolated
  • How it is scheduled
  • How it is governed
  • Which control plane owns it

VM Workload

one machine per workload

A VM workload is a traditional virtual machine:

  • Runs on ESXi
  • Has its own guest OS
  • Is managed by vCenter
  • Uses vSphere constructs directly

Control plane

  • vCenter Server is the authoritative control plane
  • VCF Automation wraps vCenter, but does not replace it

Resource model

  • CPU / memory allocated per VM
  • Strong isolation via hypervisor boundaries
  • Overcommitment controlled at ESXi level

Lifecycle

  • Create VM
  • Power on/off
  • Snapshot
  • Clone
  • Patch guest OS
  • Backup VM image

Governance style

  • VM policies
  • Placement rules
  • Affinity / anti-affinity
  • HA / DRS
  • Backup & DR policies

Pod Workload

many disposable units forming a service.

A pod workload is a Kubernetes Pod running on the vSphere Supervisor Cluster.

A pod on the vSphere Supervisor Cluster is not a traditional Kubernetes pod.
It is a vSphere Pod — Kubernetes-shaped, but vSphere-implemented.

Key point:

A pod workload is not a VM pretending to be a container.

Supervisor pods are VMs implementing container semantics, not containers implemented as VMs.

A Supervisor pod is a Kubernetes Pod whose execution unit is a micro-VM, where the VM boundary replaces the Linux container boundary.

  • The VM is the isolation primitive
  • The Pod is the semantic primitive
  • The container image is payload, not a process model
    So the VM is not “pretending to be a container” —
    it is implementing the Pod abstraction using a VM instead of a process.
Pod (Kubernetes API object)
 └─ Backed by
     └─ Ultra-lightweight VM
         └─ ESXi
             └─ Hardware

It is:

  • A Kubernetes-native unit
  • Scheduled by Kubernetes
  • Governed by Kubernetes APIs
  • Backed by vSphere under the hood

Control plane

  • Kubernetes API (Supervisor cluster) is authoritative
  • vCenter supplies infrastructure, not lifecycle semantics
  • VCF Automation governs Kubernetes intent, not individual pods

Resource model

  • CPU / memory requested via Kubernetes requests & limits
  • Scheduler decides placement
  • Isolation via:
    • Namespaces
    • RBAC
    • Pod security
    • (Optionally) VM-backed pod runtimes

Lifecycle

  • Declarative (YAML)
  • Reconciliation-based
  • Pods are ephemeral
  • Restart ≠ failure
  • Replacement is normal

Governance style

  • Namespace-level quotas
  • Admission control
  • Image policies
  • Network policies

Side-by-Side Comparison

Dimension VM Workload Pod Workload
Primary abstraction Virtual Machine Pod
Control plane vCenter Kubernetes (Supervisor)
Unit of scheduling VM Pod
OS ownership Per workload Shared / abstracted
Lifecycle style Imperative Declarative
Failure model Avoid failure Expect & replace
Scaling Clone VMs ReplicaSets / HPA
Identity Host / VM-centric Workload identity
Governance granularity VM-level Namespace / policy-level
  • VM workloads → legacy, stateful, pet-like systems
  • Pod workloads → cloud-native, stateless, cattle-like systems

Separate but Coordinated

VCF’s value is:

Running both on the same physical infrastructure, with unified policy and automation, without forcing one model onto the other.

Trying to manage pods like VMs breaks Kubernetes.
Trying to manage VMs like pods breaks operational reality.

VCF keeps them separate but coordinated.

A pod workload in VCF is a Kubernetes-native execution unit scheduled and governed by the vSphere Supervisor Kubernetes control plane, consuming vSphere resources indirectly through Kubernetes abstractions—unlike a VM workload, which is a hypervisor-managed, OS-bearing unit directly governed by vCenter.

Takeaways

  • “Pod workload” ≠ “small VM”
  • Pod workloads are Kubernetes-first, not vSphere-first
  • VMs are stateful machines
  • Pods are replaceable execution instances
  • VCF supports both because enterprises need both
