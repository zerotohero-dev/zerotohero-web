VMware Cloud Foundation

At a High Level

VMware Cloud Foundation (VCF) is a unified platform to design, plan, deploy, and manage your SDDC by using automation capabilities.

A Workload is a Workload is a Workload

That is one of the core tenents of VCF, and thus, it requires clarification before anything else.

In VCF-speak, the phrase “a workload is a workload” is not a tautology—it’s a deliberate abstraction boundary.

It means that VCF intentionally collapses many very different execution models into a single management concept, and refuses to care about their internal form unless it must.

Below is the precise meaning, layered from philosophy, to architecture, to consequences.

From VCF’s point of view, a workload is simply “something that consumes resources and is governed by policy.”

It is not:

  • a VM
  • a Container
  • a Pod VM
  • a Kubernetes pod
  • a Guest Cluster
  • a Helm release

Those are implementation details.

VCF asks only:

  • Where does it run?
  • Who owns it?
  • What resources does it consume?
  • What policies apply?
  • What lifecycle rules govern it?

If those questions can be answered, VCF is satisfied.

But, Why Does VCF Need This Abstraction

VCF exists above:

  • ESX
  • vSphere Clusters
  • vSphere Supervisor
  • Kubernetes
  • VKS
  • VMs
  • Containers

If VCF tried to manage each kind of workload differently, it would:

  • explode in complexity
  • duplicate logic
  • couple business policy to runtime mechanics

So instead, VCF defines:

Workload = Policy + Placement + Lifecycle + Resources

Everything else is delegated downward.

Examples of Workloads

All of the following are equally valid workloads in VCF:

Example Internally From VCF’s view
Traditional VM vSphere VM Workload
Pod Workload vSphere Pod on Supervisor Workload
VKS Guest Cluster Kubernetes cluster of nodes Workload
Stateful app Pods + PVCs Workload
Stateless service ReplicaSets Workload
Helm-installed app CRDs + Pods Workload

VCF deliberately does not rank or privilege any of these.

In VCF, a workload is anything that consumes platform resources and is subject to platform policy—its internal shape is intentionally irrelevant.

  • VCF intentionally erases runtime distinctions
  • Governance > implementation
  • Supervisor is the abstraction bridge
  • If two things consume resources and obey policy, VCF treats them identically

But, How is that Possible

vSphere Supervisor makes it possible.

vSphere Supervisor converts:

  • Pods to lightweight VMs
  • Kubernetes objects to vSphere-managed entities
  • Namespaces to policy and tenancy units
  • From below, everything is Kubernetes.
  • From above, everything is vSphere resources.

VCF sits above Supervisor, so it sees:

  • Resource Pools
  • Namespaces
  • Quotas
  • Policies

The kind or form of the workload becomes irrelevant.

VCF does not care:

  • whether the workload uses containerd or not
  • whether it’s a PodVM or classic VM
  • whether it runs Go, Java, or Bash
  • whether it was deployed via Helm, kubectl, or Terraform
  • whether it is “cloud-native” or “legacy”

Those are runtime-layer concerns, not platform-layer concerns.

A power grid doesn’t care if you plug in a toaster or a server rack—only how much power you draw and what safety rules apply.

This is why statements like these are true:

  • “Pod workloads are not special.”
  • “VM workloads are not special.”
  • “Guest clusters are just workloads.”
  • “Infrastructure doesn’t care about your app model.”

So, you don’t fight against the platform to fit the workload in; you design the workload to fit the platform’s (opinionated) invariants.

To internalize VCF, don’t think in:

  • Pods vs VMs
  • Containers vs Processes
  • Control Planes vs Data Planes

Instead, think in Governance:

  • Who owns this?
  • What are its limits?
  • How is it created and destroyed?
  • How does it fail?
  • How is it audited?

Don’t anchor your designs to what runs. Anchor them to how they are governed.

vSphere

At its core, vSphere is a compute, storage, and network abstraction layer: A distributed operating system for data centers.

It manages large collections of infrastructure, such as CPUs, storage, networking, GPUs, and DPUs as a seamless and dynamic operating environment, and also manages the complexity of a data center.

 
It virtualizes physical infrastructure into managed CPU, storage, and networking resources.

vSphere
├── ESX (data plane)
├── vCenter Server (control plane)
└── Integrated services (HA, DRS, vMotion, etc.)

vSphere is a distributed system that provides the following:

Primitive What it abstracts
Compute CPU scheduling, memory isolation
Storage Policy-driven persistence
Networking Virtual L2–L7 constructs
Identity RBAC, tenancy
Lifecycle Placement, HA, DRS

ESX and vCenter

The two core components of VMware vSphere are ESX and vCenter.
Whereas vSphere Client is the Web UI that interacts with the APIs of vSphere components.

vSphere is a product contract, not a system.
ESX and vCenter are the systems that implement that contract.

Lifecycle Management

Here is how vSphere facilitates LCM:

  • Lifecycle management via pre-staging ESX images.
  • Near-zero-downtime live patching for ESX.

Workload Performance

vSphere as a platform, manages workload performance in several dimensions:

  • With its GPU support, vSphere improves workload performance for modern AI workloads
  • DRS enables automatic load balancing of resources allocated to workloads in a vSphere Cluster.
  • Storage DRS optimizes VM data placement as the VM is created and used over time.

Business Continuity

Moving workloads seamlessly across the fabric, and ensuring things are always ship-shape, is what vSphere excels at:

  • High Availability with vSphere automatically restarts your VMs following physical machine failure.
  • Fault Tolerance provides continuous availability of any application in the event of a hardware failure with no data loss or downtime.
  • vMotion enables live migration of virtual machines with no disruption to users or loss of service, eliminating the need to schedule application downtime for planned server maintenance.
    • Storage vMotion avoids downtime for planned storage maintenance.

Security

At vSphere security is by default, not an afterthought:

  • Identity Federation with Microsoft Entra ID, ADFS and Okta.
  • Virtual Machine Encryption: Data-at-rest encryption for virtual machine data and disks.
  • vSphere Trust Authority: Remote attestation for sensitive workloads.

Use **

NSX

** as a network administrator for networking and automation for virtual machines, containers, or bare metal applications.

NSX enables network connectivity, operations, and scale. It uses a software-defined approach to networking that spans data centers, clouds and application frameworks.

NSX brings networking services closer to the application, whether running on virtual machines (VMs), containers or physical servers. This allows for a cloud operating model where networks can be provisioned and managed independently of the underlying hardware.

NSX abstracts network services from the underlying hardware, enabling programmable infrastructure through APIs. You can implement policies and workflows to automate tasks like network provisioning, security enforcement, load balancing, streamlining operations, and improving efficiency.

Layer 2--Layer 7 Network Stack and Overlay Services

NSX provides a complete set of logical networking capabilities and services, including logical switching, routing, distributed load balancing, virtual private network (VPN), quality of service (QoS), and monitoring.

Switching

Enable logical Layer 2 overlay extensions across a routed (Layer 3) fabric within and across data center boundaries.

Routing

Dynamic routing between virtual networks that is performed in a distributed manner in the hypervisor kernel, and scale-out routing with active-active failover with physical routers. Static routing and dynamic routing protocols are supported, including support for IPv6.

Virtual routing and forwarding (VRF)

Complete data plane isolation among tenants with a separate routing table, network address translation (NAT), and edge firewall support in each VRF on the 

NSX Tier-0 gateway

Support for bridging between VLANs configured on the physical network and NSX overlay networks, for seamless connectivity between virtual and physical workloads.

VPN

Site-to-site and unmanaged VPN for cloud gateway services.

Multi-Site Networking and Disaster Recovery

By using NSX, you can move workloads between racks and data centers by decoupling them from the underlying networks.

Federation

Centralized policy configuration and enforcement across multiple locations from a single pane of glass, enabling network-wide consistent policy, operational simplicity, and simplified disaster recovery architecture.

Multi-cloud networking

Enable consistent networking and security across data center sites, and across private cloud boundaries, irrespective of underlying physical topology or cloud platform.

Network Automation with Declarative APIs

NSX can manage the provisioning, deployment, operations and retirement of networking infrastructure and applications from a central control pane. You can automate infrastructure operations and offload manual tasks with workflows and agile templating using tools such as VCF Automation, Terraform, and Ansible. 

Self-Service Networking with VPCs and Multi-Tenancy

VPCs provide isolated network environments within a shared infrastructure. You can use Virtual Private Cloud (VPC) constructs to provision and manage network resources to business units or project teams independently, with granular control over networking policies and resource allocation.

By creating projects, you can isolate security and networking objects across tenants in a single NSX deployment. You enable multi-tenant deployment and consumption of NSX services for the provider administrator and the tenants or organizations.

The provider administrator can create projects, assign users and groups, and allocate quotas to restrict the configurations available to the tenants.

SDDC Manager

What is SDDC Manager in VMware Cloud Foundation (VCF)?

SDDC Manager is the foundational lifecycle-management and orchestration engine of VMware Cloud Foundation.
It is responsible for deploying, configuring, upgrading, and maintaining the entire VCF stack as a single, validated system rather than as independent products.

If you think of VCF as a factory-built private cloud, SDDC Manager is the factory control system that ensures every component is installed correctly, stays in a supported state, and evolves safely over time.

Also: in VCF 9.0, Broadcom’s docs state that SDDC Manager UI is being deprecated and that SDDC Manager workflows can now be found in VCF Operations and vSphere Client—so many “platform lifecycle” workflows show up there now. (TechDocs)

Core Responsibilities

1. Lifecycle Management (LCM) — its primary job

SDDC Manager performs end-to-end lifecycle operations across the VCF stack:

  • ESXi (hypervisor)

  • vCenter Server

  • NSX

  • vSAN

  • Optional components (Aria / Operations, Automation, etc.)

Key characteristics:

  • Uses validated upgrade bundles

  • Enforces component compatibility

  • Executes sequenced, automated upgrades

  • Provides rollback and precheck logic

This is not just patching — it’s system-level lifecycle orchestration.

2. Bring-Up and Expansion

SDDC Manager is responsible for:

  • Initial VCF bring-up

  • Creating:

    • Management Domain

    • VI (Workload) Domains

  • Adding:

    • Hosts

    • Clusters

    • Workload domains

  • Ensuring:

    • Correct networking

    • Correct certificate trust

    • Correct inventory alignment

Without SDDC Manager, you do not have VCF — only loosely assembled VMware products.

3. Inventory & State Authority

SDDC Manager acts as the system-of-record for:

  • What components exist

  • Their versions

  • Their relationships

  • Their lifecycle state

vCenter manages virtualization.
SDDC Manager manages the cloud system itself.

4. Guardrails & Drift Prevention

SDDC Manager enforces:

  • Supported topologies

  • Supported versions

  • Supported upgrade paths

If you try to:

  • Upgrade NSX independently

  • Modify components outside supported flows

You risk breaking VCF supportability.

This is intentional — VCF favors predictability over freedom.

What SDDC Manager Is Not

It is not:

  • A tenant portal

  • A developer platform

  • A policy engine

  • A workload orchestrator

  • A replacement for vCenter, NSX Manager, or Kubernetes APIs

Those functions live elsewhere (VCF Automation, Supervisor, vCenter, NSX, etc.).

Relationship to Other VCF Components

Component Role
SDDC Manager System lifecycle, topology, version control
vCenter Server Virtualization & cluster management
NSX Manager Networking & security
vSAN Storage
VCF Automation Consumption, policy, approvals
Supervisor / Kubernetes Workload execution

A useful mental model:

SDDC Manager manages the cloud itself; everything else manages what runs inside that cloud.

Why SDDC Manager Exists (Strategically)

Without SDDC Manager:

  • Enterprises would manually assemble components

  • Version skew would be common

  • Upgrades would be risky and expensive

  • Supportability would collapse

With SDDC Manager:

  • VCF becomes repeatable

  • Lifecycle becomes deterministic

  • VMware/Broadcom can guarantee outcomes

This is why SDDC Manager is non-optional in VCF.

Analogy (Concrete)

  • SDDC Manager → Kubernetes control plane for the cloud platform itself

  • vCenter / NSX / vSAN → Kubernetes nodes & CNI & CSI

  • VCF Automation → Higher-level API / UX for consumers

Different layers, different responsibilities.

Key Takeaways

  • SDDC Manager is the backbone of VCF

  • It owns lifecycle, topology, and compatibility

  • It enforces validated, supported states

  • It is infrastructure-centric, not tenant-centric

  • Without it, VCF does not exist as a product

Governance

#tenancy

Governance in VCF is the set of mechanisms that decide who is allowed to request which platform capabilities, under what constraints, with what approvals, and for how long — before anything is created.

VCF Automation governs entitlement and lifecycle decisions, including lease expiry and grace periods, but vSphere Supervisor remains the execution authority and only acts when Automation explicitly requests a change.

  • It’s about entitlement, approval, and lifecycle

  • It is not a Terraform-style reconciler
  • It is exactly what service providers and large enterprises need to avoid chaos

VCF Day 0 Tasks

VMware Cloud Foundation Day 0 tasks are everything required to prepare and bootstrap the platform before any tenant, workload, or higher-level automation exists.

Think of Day 0 as laying the concrete and pouring the foundation—after this, you can build floors (VCF Day 1 Tasks) and move people in (VCF Day 2 Tasks).

  • Day 0: Make the platform exist
  • Day 1: Configure and extend the platform
  • Day 2: Operate workloads on the platform

VCF is very opinionated: nothing meaningful can happen until Day 0 is clean and correct.

Core Categories of VCF Day 0 Tasks

1. Physical and Infrastructure Prerequisites

These are outside VCF but mandatory.

  • Rack & cable hosts
  • Power, cooling, BIOS/firmware
  • NTP, DNS, IPAM
  • VLANs / MTU (often jumbo frames)
  • Out-of-band management (iDRAC / iLO)

If this is wrong, everything later becomes “mysteriously flaky.”

2. ESX Installation and Host Preparation

Before VCF can orchestrate anything, ESX must exist.

  • Install ESX on all hosts
  • Apply compatible firmware and drivers
  • Validate hardware against VCF BOM
  • Configure basic networking (vmnics, MTU)

These hosts are raw compute, not yet part of any SDDC.

3. SDDC Manager Deployment

This is the true start of VCF.

  • Deploy SDDC Manager Appliance
  • Connect it to:
    • DNS
    • NTP
    • Identity source
  • Verify reachability to all ESX hosts

From this point on, VCF owns lifecycle.

4. Initial Management Domain Bring-Up

This creates the core control plane SDDC.

Includes automated deployment of:

  • vCenter Server
  • NSX Manager
  • vSAN
  • ESX Management Cluster

This is where VCF becomes a self-hosting platform.

After this, you no longer treat components individually.

5. Network and Security Baseline

Multi-Tenancy

#tenancy

In a VCF-based platform, tenant isolation is realized through vSphere Supervisor Namespaces, which serve as the enforced execution boundary for workloads.

A single tenant may be associated with one or more Supervisor Namespaces, typically aligned to Projects within an Organization.

Supervisor Namespaces provide hard isolation for compute, networking, storage, and identity, while VCF Automation operates above this layer to deliver governance, policy enforcement, and self-service without weakening those isolation guarantees.

What Multi-Tenancy Means in VCF

In VMware Cloud Foundation, multi-tenancy does not mean soft separation within a shared runtime. Instead, it is defined by:

  • Explicit tenant boundaries
  • Platform-enforced isolation
  • No cross-tenant visibility or control
  • Lifecycle independence between tenants

This design prioritizes predictability, security, and supportability over maximal flexibility.

Administrative Boundaries vs Isolation Boundaries

VCF intentionally separates administrative constructs from enforcement points.

  • Organizations and Projects define who can do what
  • Supervisor Namespaces define what can actually run and where

While Organizations and Projects are essential for governance and delegation, isolation becomes real only at the Supervisor Namespace layer.

At this boundary, VCF enforces:

  • Resource allocation and limits
  • Network segmentation
  • Storage policy enforcement
  • Identity and access scoping

Tenant-to-Namespace Mapping

A common tenant layout in VCF looks like this:

  • One Organization represents a tenant (i.e. Coca Cola)
  • One or more Project exists within that Organization (i.e. Dev / Staging / Prod)
  • Each Project (typically) maps to one Supervisor Namespace (but can map to more than one)
  • A tenant may therefore own multiple namespaces, often separated by:
    • Environment (dev / stage / prod)
    • Application domain
    • Regulatory or compliance boundary

This model allows tenants to scale horizontally without eroding isolation.

The Role of VCF Automation

VCF Automation provides the control plane for consumption, not the enforcement layer.

It is responsible for:

  • Self-service access to platform capabilities
  • Policy-driven approvals and quotas
  • Governance and auditability
  • Abstraction of underlying platform complexity

VCF Automation:

  • Does not bypass Supervisor Namespace isolation
  • Does not weaken platform guardrails
  • Does not act as the isolation mechanism itself

It governs access to platform-enforced boundaries—it does not replace them.

It Is a Layered Cake

If we think in layers:

  • Infrastructure layer: Hosts, clusters, networking, storage (operated outside VCF Automation)
  • Isolation layer: Supervisor Namespaces (hard tenant boundary)
  • Governance layer: Organizations and Projects (who can consume what)
  • Workload layer: VMs, Kubernetes clusters, and services

Multi-tenancy in VCF emerges from the composition of these layers, not from any single construct.

Summary

  • Supervisor Namespaces are the hard isolation boundary in VCF
  • Organizations and Projects provide governance, not isolation
  • Tenants may span multiple namespaces
  • VCF Automation governs consumption without compromising isolation
  • This layered design is foundational to VCF’s security and supportability

Policy

#tenancy

A Policy in VCF is a Governance Rule that defines what is allowed, constrained, or enforced across infrastructure and workloads.
It exists across multiple layers of VCF, but it becomes a first-class, user-visible construct in VCF Automation.

If you think of VMware Cloud Foundation as a city:

  • Infrastructure → roads, power, zoning
  • Policies → zoning laws, speed limits, building codes
  • VCF Automation → city planning office that enforces them before you build
  • VCF Operations → inspectors that check compliance after things exist

1. Infrastructure-Level Policies

These are foundational, implicit, and exist below VCF Automation, often enforced automatically.

Examples:

  • Storage policies (SPBM): RAID, availability, performance tiers
  • Network policies: NSX security groups, microsegmentation rules
  • Host policies: DRS, HA, admission control
  • These are:
  • Defined in vSphere / NSX
  • Consumed transparently by higher layers
  • Not tenant-facing by default

2. **VCF Automation Policies

This is usually what people mean when they say “policy”.

This is where Policy becomes a named, explicit, governable object.

In VCF Automation, policies define constraints and guardrails for consumers.

Examples:

  • Who can deploy what
  • Where workloads can land
  • What sizes, images, or networks are allowed
  • Lease durations and approval requirements

Typical policy categories:

  • Placement policies: Which clusters, regions, zones
  • Sizing policies: CPU, memory, storage limits
  • Image policies: Approved OS images only
  • Network policies: allowed networks / isolation
  • Approval policies: manual gates for certain actions
  • Lease policies: TTL and reclamation rules

These policies are evaluated before provisioning.

3. Organization and Project Scoping

In VCF Automation, policies are usually scoped to:
- Organizations
- or Projects

They act as guardrails, not implementation details

Example:

“In Project Prod, only Gold storage, max 16 vCPU, and approval required.”

This is where policy intersects with consumption abstraction.

4. VCF Operations

In **VCF Operations, policy ≠ enforcement.

VCF Operations uses policy indirectly.

In VCF Operations:

  • Policies become:
    • Alerts
    • Compliance checks
    • Drift detection
    • Optimization rules

VCF Operations answers:

“Is reality still compliant with the policy?”

VCF Automation answers:

“Should this action be allowed at all, in the first place?”

Summary

  • A Policy in VMware Cloud Foundation is a governance rule, not a single product feature
  • VCF Automation is where policies are:
    • Explicit
    • Named
    • Scoped
    • Evaluated before action
  • VCF Operations observes and reports compliance
  • Infrastructure layers actually enforce the low-level mechanics

Blueprint

A Blueprint is a declarative deployment template that defines what can be requested, how it is configured, where it may land, and under which policies and approvals.

A VMware Cloud Foundation Design Blueprint is a prescriptive end-to-end design for a VCF Fleet in a commonly requested architecture topology, created to help accelerate time to value when building a new VMware Cloud Foundation platform.

A Blueprint has a prescriptive selection of deployment models chosen from the design library to conform to the design profile of the Blueprint.

A Blueprint an be deployed as is, with no alterations, or can be used as template where any of the deployment models can be substituted for alternatives from the same category within the design library. 

A Blueprint reflects the initial deployment of a VCF Fleet and supports expansion in a variety of ways to support the needs of your organization.

A Blueprint provides a set of planning, implementation and design elements (requirements and recommendations) based on the chosen deployment models.

Blueprints in VCF Automation are governed deployment templates that define allowed workload patterns, configuration inputs, placement constraints, and approval policies. They orchestrate resource creation but are not part of the runtime execution layer.

Once a deployment is created, the Blueprint can be deleted and the workload keeps running: It’s not part of the runtime layer.

I Heard Something Called a “Template”—Is that a “Blueprint”

In VMware Cloud Foundation, the terms template and blueprint are related but not interchangeable. They operate at different abstraction layers and serve different purposes in the lifecycle of infrastructure and workloads.

Concept What it is Primary Scope Think of it as…
Template A technical artifact used to create resources Infrastructure / VM / Image layer A golden image or base mold
Blueprint A declarative application or environment definition Application / platform layer A deployment recipe or infrastructure-as-code document

A template defines what a single resource looks like
A blueprint defines how multiple resources come together.

Templates in VCF

A template is a reusable, preconfigured artifact that represents one resource, typically:

Common Examples

  • VM Template
    • A golden VM image (OS + patches + base software)
    • Used to rapidly deploy identical VMs
  • OVF/OVA Templates
    • Portable packaged virtual appliances
  • Content Library Templates
    • Shared across vCenter instances and VCF Domains

Characteristics

  • Infrastructure-level
  • Static
  • Low abstraction
  • No logic, no dependencies
  • Consumed by automation tools (not orchestrating anything themselves)

A Template says:

“This is what a machine looks like when it’s born.”

In contrast, a single Blueprint can define these following components:

  • 2 application VMs
  • 1 database VM
  • A load balancer
  • A network segment
  • Placement rules
  • Lifecycle actions

That entire topology is deployed as one logical unit.

Blueprints consume templates.

  1. Template:
    • Ubuntu 22.04 VM with hardened baseline
  2. Blueprint:
    • Uses that template
    • Defines how many instances
    • Attaches networks
    • Applies policies
    • Adds lifecycle logic
      So:
  • Templates = building blocks
  • Blueprints = architectural plans

Workflow

Workflows consist of a schema, workflow variables, and input and output parameters.

The workflow schema is the main component of a workflow as it defines all the workflow elements and the logical connections between them. The workflow variables and parameters are the properties that workflows use to transfer data.

Who Creates and Executes Workflows

In VMware Cloud Foundation, there isn’t a single “workflow engine” owned by only one product. Different parts of VCF create/execute different kinds of workflows:

VCF Automation is the place where you typically design and run “day-1” style workflows tied to requests (catalog items, provisioning, configuration, approvals, extensibility, etc.). Its value prop explicitly includes orchestration and extensibility.

Under the hood, these workflows commonly leverage an Orchestrator (formerly vRealize Orchestrator) integrated/embedded with the automation stack. It is executed by VCF Operations Orchestrator which is a part of the VCF Operations stack.

VCF Operations is primarily the ops brain: monitoring, alerting, capacity, compliance, etc.. It can run actions (and sometimes chain remediation) in response to alerts.

Actual workflow automation is done through VCF Operations Orchestrator, which is explicitly described as a “workflow automation solution” and can create and run workflows.

VCF Operations Orchestrator

VCF Operations Orchestrator saves a workflow token every time a workflow runs, recording the details of that specific workflow run.

Workflow Parameters

Workflows receive input parameters and generate output parameters when they run.

Input Parameters

Most workflows require a certain set of input parameters to run. An input parameter is an argument that the workflow processes when it runs. The user, an application, another workflow, or an action passes input parameters to a workflow for the workflow to process when it runs.

For example, if a workflow resets a virtual machine, the workflow requires the name of the virtual machine as an input parameter.

Output Parameters

A workflow's output parameters represent the result from the workflow run. Output parameters can change when a workflow or a workflow element runs. During a workflow run, the workflow can receive the output parameters of other workflows as input parameters.

For example, if a workflow creates a snapshot of a virtual machine, the output parameter for the workflow is the resulting snapshot.

Workflow Variables

Workflow elements process data that they receive as input parameters, and set the resulting data as workflow variables or output parameters.

Read-only workflow variables act as global constants for a workflow. Writable variables act as a workflow’s global variables.

You can use variables to transfer data between the elements of a workflow.

You can obtain variables in the following ways:

  • Define variables when you create a workflow.
  • Set the output parameter of a workflow element as a workflow variable.
  • Inherit variables from a configuration element.

Workflow Schema

A workflow schema is a graphical representation that shows the workflow as a flow diagram of interconnected workflow elements. The workflow schema is the most important element of a workflow as it determines its logic.

Workflow Presentation

When users run a workflow, they provide the values for the input parameters of the workflow in the workflow presentation.

Workflow Tokens

A workflow token represents a workflow that is running or has finished running.

A workflow is an abstract description of a process that defines a generic sequence of steps and a generic set of required input parameters. When you run a workflow with a set of input parameters, you receive an instance of this abstract workflow that behaves according to the specific input parameters you give it. This specific instance of a finished or a running workflow is called a workflow token.

Workflow Token Attributes

Workflow Token Attributes are the specific parameters with which a workflow token runs.

The workflow token attributes are an aggregation of the workflow's global variables and the specific input and output parameters with which you run the workflow token.

Within VMware Cloud Foundation, an SDDC is the foundational architectural model that virtualizes all core data-center resources (compute, storage, networking, and management) and delivers them as software services.

Think of SDDC as the “what”, while VCF is the “productized implementation” of that model.

1. What “Software-Defined” means here

In an SDDC:

Resource How it’s defined
Compute Software-defined via vSphere
Storage Software-defined via vSAN
Networking and Security Software-defined via NSX
Lifecycle and Ops Software-defined via VCF Operations
Automation and Consumption Software-defined via VCF Automation

No hardware-specific configuration is required by tenants. Everything is expressed as policies, APIs, and declarative intent.

2. SDDC vs VCF

If you think of SDDC as an interface, VCF is an implementation of that interface.

Concept What it is
SDDC An architecture / operating model
VCF A curated, integrated, opinionated implementation of SDDC

Or in mor simpler terms, You cannot “install an SDDC”.
You deploy VCF, and the result is an SDDC.

As a very simplistic analogy:

  • SDDC → “Cloud computing”
  • VCF → “AWS (but for private cloud)”

3. What an SDDC gives you in practice

Inside VCF, SDDC enables:

  • Uniform resource abstraction
    • A VM workload and a Kubernetes workload are both “workloads”
  • Policy-driven behavior
    • Placement, HA, security, backups are rules, not scripts
  • Lifecycle coherence
    • Upgrade ESX, NSX, vSAN together, safely and repeatably
  • Multi-tenancy
    • Organizations, Projects, namespaces, quotas, RBAC
  • Cloud-like operations on-prem
    • Self-service, APIs, Governance, chargeback

4. What is not SDDC

SDDC is not:

  • A single product
  • Just virtualization
  • Just Kubernetes
  • A UI
  • A vCenter feature

It is the system-level design philosophy that VCF enforces.

VCF’s value proposition is:

“You don’t assemble an SDDC yourself anymore: We ship one.”

That’s why:

  • VCF has strict BOMs
  • VCF enforces lifecycle orchestration
  • VCF tightly integrates ops + automation + infra

This is intentional: freedom at the top, rigidity at the bottom.

Summary

  • SDDC = architecture
  • VCF = implementation of that architecture
  • In VCF, everything (compute, storage, network, ops) is software-defined
  • SDDC enables cloud-like behavior on private infrastructure
  • You deploy VCF, and you get an SDDC: That simple.

In VMware Cloud Foundation, a Project is a logical, tenant-scoped construct used for workload organization, Governance, and self-service consumption.

It exists at the automation and policy layer, not the virtualization layer.

Projects group workloads, users, quotas, policies, and lifecycle controls under a common administrative boundary. It is “a governed slice of cloud capacity” exposed to consumers.

Projects live primarily in VCF Automation (formerly vRealize Automation / Aria Automation).

It is not:

  • A vSphere cluster
  • A Kubernetes cluster
  • A resource pool
  • A Supervisor Namespace (though it may map to one)

Instead, it is a control-plane abstraction that binds together multiple lower-level constructs.

VCF Instance
 └── Organization
      └── Project
           ├── Users & Roles
           ├── Policies & Quotas
           ├── Cloud Zones
           ├── Workload Deployments
           └── Supervisor Namespaces / VKS Clusters

What Does a Project Provide

1. Multi-Tenancy Boundary

A project represents a tenant or sub-tenant:

  • Different teams
  • Different applications
  • Different environments (prod, staging, dev)

Each project is isolated from others by policy.

2. Access Control and Roles

Projects define who can do what:

  • Project Administrator
  • Project User
  • Project Advanced User
  • Project Auditor

These roles govern:

  • Who can deploy workloads
  • Who can modify Blueprints
  • Who can see usage or costs

3. Resource Consumption Boundaries

Projects define where workloads can land:

  • Which Cloud Zones they can use
  • Resource quotas (CPU, memory, storage)
  • Network constraints
  • Storage policies

This prevents a single tenant from exhausting shared infrastructure.

4. Workload Placement and Lifecycle

Projects are the context in which users deploy:

  • Virtual Machines
  • Kubernetes clusters (e.g., Supervisor-backed)
  • Application blueprints
  • Day-2 operations (scale, update, destroy)

Everything a user deploys is always within a project.

Relationship to Kubernetes / Supervisor

A Project maps to:

  • A Supervisor Namespace

  • One or more Kubernetes clusters

  • A set of Kubernetes namespaces inside those clusters

But the Project itself is not Kubernetes-native.
It’s a management abstraction above Kubernetes.

Think of it as:

“A governance wrapper around Kubernetes and VM resources.”

What a Project is NOT

Not a Project Why
vSphere Cluster That’s infrastructure
Supervisor Cluster That’s a control-plane runtime
Supervisor Namespace Namespace is a technical implementation detail
Organization Higher-level container of projects
Folder Too simplistic; no policy semantics

Project Usage Patterns

VCF Automation supports both one-to-one and one-to-many mappings between Projects and Supervisor Namespaces; the choice is an architectural pattern, not a platform constraint.

Pattern A — 1 Project → 1 Supervisor Namespace

Very common, very clean:

Organization: Coca-Cola
├── Project: Coke-Prod  → ns-coke-prod
├── Project: Coke-Stage → ns-coke-stage
└── Project: Coke-Dev   → ns-coke-dev

This provides:

  • Clean blast-radius isolation
  • Clear RBAC and quotas per environment
  • Easy mental model
  • Common in regulated / prod-heavy setups

This is a design choice, not a platform requirement.

Pattern B — 1 Project → Multiple Supervisor Namespaces

(Equally valid, also common in larger deployments)

Organization: Coca-Cola
└── Project: Coke
    ├── ns-coke-dev
    ├── ns-coke-stage
    └── ns-coke-prod

This provides:

  • Shared governance (same people, same blueprints)
  • Environment separation handled at namespace level
  • Easier cross-env promotion workflows
  • Fewer projects to manage

In VMware Cloud Foundation:

  • A Project is a logical tenant boundary
  • It governs who can deploy what, where, and how
  • It sits above infrastructure and below the organization
  • It is the unit of consumption for developers and teams
  • It enables Multi-Tenancy, Governance, and self-service

In VCF Automation, an Organization is not a technical runtime boundary—It is a business-facing logical entity that defines who is allowed to consume cloud capabilities, under which rules, and with which visibility.

It is the outermost “contract boundary” for consumption, not execution.

An Organization is a logical tenant boundary that groups identity, policy, quotas, visibility, and approval flows for consuming infrastructure and platform services.

It answers:

  • Who is consuming?
  • What are they allowed to consume?
  • Under which business rules?
    —not how workloads actually run.

Organization is “Logical”, Namespace is the Real Boundary

Because Supervisor Namespaces are the actual enforcement boundary, a single Organization can define multiple tenants.

But those “tenants” are operational tenants, not consumption tenants.

There are two different kinds of tenancy in VCF:

Layer Kind of tenant Enforced by
Consumption Layer Business / Contract Tenant Organization
Runtime Layer Technical / Isolation Tenant Namespace

They are deliberately decoupled.

Namespace is the “real” Tenant Boundary

A Supervisor Namespace enforces:

  • RBAC
  • Resource quotas
  • Network isolation
  • Storage policies
  • Pod security boundaries
  • Runtime blast radius
    This is where:
  • Kubernetes says “no”
  • vSphere enforces limits
  • Workloads are actually prevented from crossing boundaries

Namespace = hard enforcement boundary

Organizations define intent and Governance.
An Organization:

  • Defines who is allowed to create tenants
  • Defines how many tenants
  • Defines under which rules
  • Defines approval and lifecycle policies

It’s closer to:

  • “Who is allowed to mint isolation boundaries?”
    not:
  • “Where isolation happens.”

An Organization Can Have Multiple Tenants

A single Organization can own:

  • Multiple Projects
  • Each Project can map to:
    • One Supervisor Namespace
    • Or several namespaces (depending on policy / model)
Organization: Acme Corp
├── Project: Acme-Prod     → ns-acme-prod
├── Project: Acme-Stage    → ns-acme-stage
├── Project: Acme-Dev      → ns-acme-dev
└── Project: Acme-Analytics → ns-acme-analytics

Each namespace is a real tenant boundary.

But from a business standpoint:

  • Acme Corp is still one customer:
    • One billing entity
    • One governance envelope
    • One policy domain

This construct enables the following scenarios (which would have been impossible otherwise).

1. One Customer, Many Isolated Environments

Classic need:

  • Dev / Stage / Prod
  • Regulated vs non-regulated
  • Different trust zones

All under:

  • One contract
  • One identity plane
  • One catalog

2. Platform Teams Acting as Internal Providers

Example:

  • Organization = “Platform Team”
  • Projects = internal product teams
  • Namespaces = per-team isolation

This would break if Organization == Namespace.

3. Managed Services / MSP model

Example:

  • Organization = External customer
  • Projects = customer environments
  • Namespaces = actual enforcement boundaries

Same model works for:

  • Internal IT
  • SaaS providers
  • Telco multi-tenant platforms

Namespace is the True Invariant

If something must be isolated, it must be isolated at the namespace level.

Anything above that:

  • Is governance
  • Is visibility
  • Is policy
  • Is economy

Organizations define who may create tenants; namespaces are the tenants.

Organization ≈ Cloud Account with Governance

Similar to:

  • AWS Account
  • Azure Subscription (with Management Groups layered above)
  • GCP Project (with Org/Folder hierarchy)

But in VCF, it’s explicitly framed as consumption-first, not infrastructure-first.

What Is an Organization

1. A Tenant Boundary

  • Separates consumers (e.g., companies, business units, external customers)
  • Enforces isolation in:
    • Identity and Access
    • Visibility
    • Quotas and Limits
    • Chargeback / showback

2. A Policy Anchor

Organizations are where policies attach, such as:

  • Who can request what
  • Approval workflows
  • Budget limits
  • Lease durations
  • Allowed regions / environments
  • Allowed workload types (VM, Kubernetes, services)

These policies then flow downward.

3. A Consumption Namespace

Everything inside an Organization is:

  • Visible only to that Organization (unless shared)
  • Governed consistently

This includes:

  • Projects
  • Blueprints / templates
  • Catalog items
  • Requests
  • Cost views
Organization
├── Projects
│   ├── Policies
│   ├── Quotas
│   ├── Users / Groups
│   └── Workload requests
│
└── (maps indirectly to)
    └── Cloud Zones
        └── Infrastructure (vSphere, Supervisor, VKS, etc.)
  • Infrastructure never belongs to an Organization
  • Consumption always does

Consumption Abstraction

Organization deliberately hides infrastructure topology.

From the Organization’s point of view:

  • There is no ESX
  • No cluster
  • No datastore
  • No vSphere Supervisor

There is only:

  • “What can I request?”
  • “What am I allowed to use?”
  • “Who approves this?”
  • “How much does it cost?”

This abstraction enables:

  • Multi-tenant platforms
  • Internal private clouds
  • Managed services
  • SaaS-like experience on top of VCF

Organization and Projects

If Organization is “who is the customer”,
then Project is “what they are working on”.

Typical mapping:

Organization: Acme Corp
├── Project: Acme-Prod
├── Project: Acme-Stage
└── Project: Acme-Dev

Projects:

  • Scope quotas
  • Bind to cloud zones
  • Eventually map to Supervisor Namespaces or runtime constructs

Organizations never do.

Summary

  • Organization = Governance + visibility
  • It exists to model consumption, not execution
  • It owns policy, identity, and economics
  • It deliberately abstracts away infrastructure
  • Everything technical happens below it, via Projects and Cloud Zones

  • One tenant could impact others

Management Cluster

What is the Management Cluster in VMware Cloud Foundation (VCF)?

In VMware Cloud Foundation, the Management Cluster is the foundational cluster that runs the platform itself.
It hosts the management plane—the services that build, operate, upgrade, and secure the rest of the cloud.

Put simply:

If the Management Cluster is down, you can’t manage the cloud—only the workloads keep running.

Where It Lives (Important Precision)

The Management Cluster is part of the Management Domain (a special Workload Domain).
It is created first during Day-0 bring-up and is non-optional.

What Runs on the Management Cluster

Typical components include:

  • SDDC Manager

  • vCenter Server (management vCenter)

  • NSX Managers

  • vSAN management services

  • Optional: Operations / Automation services (depending on deployment)

These components do not belong to tenants. They belong to the platform.

What the Management Cluster Is Responsible For

1️⃣ Platform Control Plane

It runs the services that:

  • Define topology

  • Enforce lifecycle sequencing

  • Validate upgrades

  • Maintain inventory and state

2️⃣ Lifecycle Authority

All VCF lifecycle operations (create domain, expand cluster, upgrade stack) are coordinated from here.

3️⃣ Isolation from Tenant Workloads

Tenant workloads do not run here (by design):

  • No tenant VMs

  • No tenant Kubernetes workloads

  • No customer apps

This protects:

  • Stability

  • Security

  • Upgrade safety

What the Management Cluster Is Not

It is not:

  • A tenant cluster

  • A workload execution environment

  • A shared resource pool for apps

  • Optional or replaceable

Treating it like a general-purpose cluster is a serious anti-pattern.

Management Cluster vs Workload (VI) Clusters

Aspect Management Cluster Workload (VI) Cluster
Purpose Run the platform Run tenant workloads
Owner Platform Admin Tenant / Project
Lifecycle Highly controlled More flexible
Blast Radius Very high Scoped
Tenant Access None Yes

Why VCF Separates It

This separation ensures:

  • Predictable upgrades (no tenant interference)

  • Clear blast-radius control

  • Strong security boundaries

  • Operational sanity at scale

It’s the same pattern used by:

  • Kubernetes (control plane vs worker nodes)

  • Public clouds (provider plane vs customer plane)

Mental Model (Kubernetes Analogy)

  • Management Cluster → Kubernetes control plane nodes

  • Workload Clusters → Kubernetes worker nodes

  • SDDC Manager → Cluster lifecycle operator

You wouldn’t run apps on the Kubernetes control plane; same rule applies here.

Failure & Availability Considerations

  • Management Cluster is typically:

    • Small (minimum viable size)

    • Highly available

    • Strictly protected

  • If it fails:

    • Existing workloads keep running

    • You lose management, scaling, and upgrades until restored

Key Takeaways

  • The Management Cluster runs the cloud, not the workloads

  • It is part of the Management Domain

  • It hosts SDDC Manager, vCenter, NSX Managers

  • It is isolated, mandatory, and high-blast-radius

  • Treat it as sacred infrastructure

What is a Workload Cluster in VMware Cloud Foundation (VCF)?

In VMware Cloud Foundation, a Workload Cluster is a cluster dedicated to running tenant workloads—VMs or Kubernetes-backed workloads—as opposed to running the platform itself.

It lives inside a Workload Domain (VI Domain) and is where actual business applications execute.

In plain terms:

Workload Clusters run applications; Management Clusters run the cloud.

Where a Workload Cluster Fits

VCF Platform
│
├── Management Domain
│   └── Management Cluster  (platform services only)
│
└── Workload Domain (VI Domain)
    ├── Workload Cluster A  ← tenant workloads
    ├── Workload Cluster B  ← tenant workloads
    └── Workload Cluster C  ← tenant workloads

  • A Workload Domain can contain multiple Workload Clusters

  • Each cluster is an operational and failure boundary

  • Tenants never see the Management Cluster, but they consume resources from Workload Clusters

What Runs on a Workload Cluster

Depending on how VCF is used, a Workload Cluster may host:

VM Workloads

  • Traditional virtual machines

  • App servers, databases, legacy systems

  • Managed via vCenter

  • Governed via VCF Automation (orgs, projects, policies)

Kubernetes / Pod Workloads

  • Supervisor-backed workloads

  • Guest clusters (e.g., TKG)

  • PodVM-based workloads

  • Namespaces mapped to projects

The cluster itself is workload-agnostic; policy determines what lands there.

Who Owns a Workload Cluster

Role Responsibility
Platform Admin Creates clusters, upgrades ESXi/NSX/vSAN
Tenant / Project Admin Consumes capacity, deploys workloads

This split is critical:

  • Tenants use clusters

  • Platform Admins operate clusters

Key Responsibilities of a Workload Cluster

1️⃣ Workload Execution

This is where:

  • CPU, memory, storage are consumed

  • Network traffic flows

  • Availability and performance matter most

2️⃣ Isolation & Blast Radius

Each Workload Cluster provides:

  • Fault isolation

  • Upgrade isolation

  • Security boundaries

A problem in one cluster should not affect others.

3️⃣ Lifecycle Boundaries

Clusters are upgraded:

  • Under SDDC Manager control

  • In a controlled, validated sequence

  • Independently of other clusters (within constraints)

What a Workload Cluster Is Not

It is not:

  • A management plane

  • A place for SDDC Manager or NSX Managers

  • A tenant abstraction by itself

  • A free-for-all environment

A Workload Cluster is shared infrastructure, not a tenant.

Workload Cluster vs Management Cluster (Quick Contrast)

Dimension Management Cluster Workload Cluster
Purpose Run the platform Run applications
Tenant workloads ❌ No ✅ Yes
Owner Platform Admin Shared
Blast radius Global Scoped
Upgrade sensitivity Extreme High but contained

Relationship to Multi-Tenancy

Important nuance:

  • Tenancy does not equal cluster

  • Multiple tenants can share:

    • The same Workload Cluster

    • While remaining isolated via:

      • Organizations

      • Projects

      • Namespaces

      • NSX micro-segmentation

Clusters are capacity and fault boundaries, not business boundaries.

Mental Model (Kubernetes Analogy)

  • Workload Cluster → Kubernetes worker node pool

  • Management Cluster → Kubernetes control plane

  • Namespaces / Projects → Tenant isolation

  • VCF Automation → Policy and governance layer

You scale, isolate, and govern above the cluster.

Why VCF Uses Workload Clusters

Pros

  • Strong fault isolation

  • Predictable lifecycle

  • Flexible placement strategies

  • Supports mixed workload types

Cons

  • More moving parts

  • Requires capacity planning

  • Requires clear cluster placement policy

VCF optimizes for enterprise-scale control, not minimal topology.

Key Takeaways

  • Workload Clusters are where applications run

  • They live inside Workload (VI) Domains

  • They are operated by Platform Admins

  • They can host VM and Kubernetes workloads

  • They are shared but strongly governed

VCF Domain

A VCF Domain is a core VMware Cloud Foundation platform construct.

  • It is defined and lifecycle-managed by VCF itself (via SDDC Manager).
  • VCF Automation consumes it for tenant self-service and provisioning.
  • VCF Operations observes it for monitoring, health, and capacity.

A VCF Domain is an isolated SDDC unit inside VCF with its own full stack:

Layer Included
Compute One or more ESXi clusters
Storage vSAN (or supported external storage)
Networking NSX (T0/T1, segments, firewall)
Management vCenter Server
Lifecycle Patching, upgrades, drift control

Each domain is independently lifecycle-managed (upgrade cadence, patch windows, scaling).

A VCF Domain in a VCF Instance consists of the following components:

  • One vCenter instance.
  • One or more vSphere Clusters with vSphere HA and vSphere DRS enabled.
  • At least one vSphere Distributed Switches per cluster for system traffic and NSX Segments for workloads.
  • Dedicated NSX Manager instance, either specifically for the VCF Domain or shared between VCF Domains, for configuring and implementing software-defined networking.
  • Optional: One or more NSX Edge Clusters, added after you create the Workload Domain, that connects the workloads in the Workload Domain for logical switching, logical dynamic routing, and load balancing.
  • One or more shared storage allocations as principal or supplemental storage.

Types of VCF Domains

There are two types of VCF Domains:

  • Management Domain is created during the deployment or convergence process.
  • Workload Domain runs consumer workloads on Day 2.

Administrative Roles in VCF

#tenancy

In VMware Cloud Foundation, Platform Admin and Tenant Admin are useful conceptual terms, but not official role names in VCF.

VCF defines explicit, documented roles such as:

  • Organization Administrator
  • Project Administrator
  • (and related Auditor / User roles)

This page uses Platform Admin as a conceptual responsibility boundary, and Organization / Project Administrator as the official tenant-facing roles defined by VCF.

The distinction is intentional and fundamental.

Who Owns the Metal

VCF separates infrastructure ownership from consumption governance: platform infrastructure is operated outside VCF Automation, while Organization Administrators and Project Administrators govern what runs within tenant boundaries.

What Organization and Project Administrators Own

Organization Administrator

Organization Administrator has Tenant-level authority.

An Organization Administrator represents the tenant boundary in VCF Automation.

They manage:

  • Organization-level configuration
  • Tenant users and role assignments
  • Organization-wide quotas and policies
  • Governance within the organization

They cannot:

  • Modify platform topology
  • Upgrade core infrastructure
  • Affect other organizations
  • Bypass platform guardrails

Project Administrator

The Project Administrator is a Project-scoped operational authority.

A Project Administrator manages consumption and operations within a single Project, which maps to one (typical, but can be more than one) Supervisor Namespace.

The Project Administrator can create Projects, assign users and groups, and allocate quotas to restrict the configurations available to the tenants.
|
By creating Projects, you can isolate security and networking objects across tenants in a single NSX deployment.

Project Administrators manage:

  • Project users and access
  • Workload provisioning (VMs, Kubernetes clusters, services)
  • Project-scoped quotas and approvals
  • Day-to-day operational policies

Their authority is strictly bounded by:

  • The Organization
  • Platform-defined constraints

Why VCF Enforces This Split

Without this separation:

  • Tenants could break upgrade paths
  • Platform consistency would erode
  • Supportability would collapse
  • One tenant could impact others

VCF deliberately trades tenant flexibility for platform predictability, safety, and lifecycle integrity.

Summary

  • Organization Administrator is the true tenant administrator in VCF 9
  • Project Administrator governs consumption within a tenant
  • The separation protects multi-tenancy, lifecycle management, and supportability
  • This boundary is foundational to VCF’s architecture

A vSphere Cluster (or Supervisor Cluster) is a logical grouping of ESX hosts that vCenter treats as a single resource pool and failure domain.

  • It exists only in vCenter
  • ESX hosts do not know they are “in a cluster”.

When you create a cluster in vCenter Server, you are telling:
“*These ESX hosts agree to be scheduled, balanced, and failed over together.”

This enables cluster-level logic, such as

  1. DRS (Distributed Resource Scheduler):
    • vCenter decides
      • which host should run which VM.
      • When to rebalance the load.
    • Uses vMotion to move VMs between hosts.
  2. HA (High Availability)
    • Hosts run a lightweight HA agent.
    • If a host dies:
      • Other hosts detect it
      • vCenter (or preconfigured HA logic) restarts VMs elsewhere.
  3. Shared resource abstraction
    • Instead of thinking:
      • Host A: 64 cores Host B: 64 cores
    • vCenter presents
      • Cluster: 128 cores

VM placement and limits are logical, not physical.

Consensus system No quorum, no RAFT

Resource Governance

in Supervisor namespaces, resource governance is outside Kubernetes.

In Kubernetes:

  • CPU/memory limits = ResourceQuota
  • Storage = StorageClass + PVC
  • Network = CNI rules

In Supervisor:

  • CPU/memory = vSphere resource pools
  • Storage = datastore policies
  • Network = NSX segments
  • Limits are hard boundaries, not advisory

So when you say:

"This namespace gets 8 vCPU and 64 GB RAM"

That's not a Kubernetes promise---that's vSphere enforcing physics.

HA

HA (High Availability) in VMware Cloud Foundation (VCF)

Short definition

HA (High Availability) in VCF is the vSphere-native mechanism that automatically restarts workloads when a physical host fails.

It is not a single VCF feature; rather, VCF relies on and standardizes HA across its managed infrastructure.

Where HA lives in the VCF stack

HA is not invented by VCF. It comes from vSphere and is enforced consistently by VCF.

VMware Cloud Foundation
└── vSphere
    └── Cluster
        └── HA (enabled at cluster level)
            └── VM restart on host failure

  • Implemented by: VMware vSphere

  • Configured in: VMware vCenter Server

  • Consumed by: All VMs running in VCF-managed clusters

VCF ensures HA is correctly configured, validated, and lifecycle-managed as part of its opinionated design.

What HA actually does

Core responsibility

Detect host failure → restart affected VMs on surviving hosts

How it works (simplified)

  1. ESXi hosts exchange heartbeats

  2. A host stops responding

  3. Remaining hosts declare it failed

  4. vCenter selects new hosts

  5. VMs are restarted from disk

⚠️ Important: VMs are restarted, not live-migrated

What HA protects against (and what it does not)

✅ Protects against

  • Physical ESXi host failure

  • Power loss to a host

  • PSOD / host crash

  • Host isolation scenarios

❌ Does NOT protect against

  • Guest OS crashes

  • Application-level failures

  • Storage corruption

  • Network partition inside the VM

  • Data loss between last write and crash

HA answers “Is the VM running?”, not “Is the app healthy?”

HA vs related VCF concepts (very common confusion)

Feature What it does Relationship to HA
HA Restarts VMs after host failure Baseline resilience
vMotion Live-migrates running VMs Avoids downtime
DRS Load-balances VMs Performance optimization
Fault Tolerance Zero-downtime failover Much stricter, limited
VMware Live Recovery Site-level disaster recovery Cross-site

HA inside a VCF Workload Domain

In VCF, every workload domain cluster is typically:

  • HA-enabled

  • DRS-enabled

  • Validated against design rules

Example:

Workload Domain
└── Cluster
    ├── HA: Enabled
    ├── DRS: Fully Automated
    └── Admission Control: Enabled

VCF expects HA to be on; disabling it usually breaks design assumptions.

Admission Control (critical but often misunderstood)

HA includes Admission Control, which prevents you from powering on too many VMs.

Purpose:

Ensure enough spare capacity exists to restart VMs after a failure.

If Admission Control says “no”:

  • Nothing is broken

  • You are overcommitted beyond your failure tolerance

This is HA doing its job, not being restrictive.

HA vs Kubernetes-style HA (important mental shift)

VMware HA Kubernetes HA
Restarts VMs Restarts pods
Minutes-scale recovery Seconds-scale recovery
Infrastructure-level Application-level
Blind to app health Health-probe driven

VCF HA is infrastructure HA, not cloud-native app HA.

Anti-patterns

❌ Turning off HA to “fit more VMs”
→ Silent loss of failure guarantees

❌ Expecting HA to replace app clustering
→ Wrong layer

❌ Confusing HA with zero downtime
→ There is always a restart

Key takeaways

  • HA in VCF = vSphere HA, standardized and enforced

  • It restarts VMs after host failure

  • It does not provide zero-downtime or app-level resilience

  • It is a non-optional baseline for VCF design assumptions

Fault Tolerance

Fault Tolerance (FT) in VMware Cloud Foundation (VCF)

Short definition

Fault Tolerance (FT) is a vSphere feature that provides zero-downtime protection for a single VM by running a lockstep secondary copy of that VM on another host.
If the primary host fails, the secondary immediately takes over with no reboot and no data loss.

VCF inherits FT from vSphere; it does not redefine it.

Where FT lives in the VCF stack

VMware Cloud Foundation
└── vSphere
    └── Cluster
        └── Fault Tolerance (per-VM feature)
            ├── Primary VM
            └── Secondary VM (lockstep)

  • Implemented by: VMware vSphere

  • Configured in: VMware vCenter Server

  • Applies to: Individual VMs (opt-in)

What FT actually does

Core behavior

  • Executes the same CPU instructions on two VMs simultaneously

  • Streams execution state over a dedicated FT logging network

  • On host failure → instant switchover, no restart

HA restarts VMs.
FT never restarts them.

FT vs HA vs DRS (the clean separation)

Feature Failure tolerance Downtime Scope
FT Host failure Zero Single VM
HA Host failure Minutes All VMs
DRS Resource imbalance None Cluster optimization

FT is about continuity, not balancing or recovery speed.

Why FT is rarely used in VCF

FT is powerful but expensive.

Constraints (by design)

  • Limited VM sizes (CPU / memory caps)

  • No snapshots

  • No suspend

  • No Storage vMotion

  • Requires identical CPU features across hosts

  • Heavy network & CPU overhead

Because of this, FT is typically reserved for:

  • Legacy apps that cannot tolerate any downtime

  • Single-instance workloads with no clustering option

FT in a VCF world (practical reality)

VCF environments usually prefer:

  • HA + DRS for infrastructure resilience

  • Application-level HA (clusters, replicas, quorum)

  • Kubernetes controllers for pod-level restart

  • Site-level DR for disasters

FT remains available, but it is not a default design assumption in VCF.

Mental model (simple analogy)

  • HA: “If the server crashes, reboot it elsewhere”

  • DRS: “Move servers around to keep things fair”

  • FT: “Run two brains in sync at all times”

FT is like live brain mirroring — effective, but costly.

Anti-patterns

❌ Using FT as a substitute for application HA
❌ Enabling FT broadly “just in case”
❌ Forgetting FT network requirements
❌ Expecting FT to scale like Kubernetes replicas

Key takeaways

  • FT provides true zero-downtime VM failover

  • It is a vSphere feature, inherited by VCF

  • High overhead and strict limits make it niche

  • In modern VCF designs, FT is the exception, not the rule

DRS

DRS (Distributed Resource Scheduler) in VMware Cloud Foundation (VCF)

Short definition

DRS (Distributed Resource Scheduler) is the vSphere cluster service that continuously balances CPU and memory load by automatically placing and live-migrating VMs (via vMotion) across hosts.

In VCF, DRS is foundational, not optional.

Where DRS lives in the VCF stack

VMware Cloud Foundation
└── vSphere
    └── Cluster
        ├── HA  → “Can the VM run?”
        └── DRS → “Where should it run?”

  • Implemented by: VMware vSphere

  • Executed via: VMware vCenter Server

  • Acts on: VMs inside a cluster

VCF assumes DRS is enabled and automated for predictable behavior.

What DRS actually does

Core responsibilities

  1. Initial Placement – chooses the best host when a VM powers on

  2. Continuous Balancing – monitors CPU & memory pressure

  3. Live Migration – moves VMs using vMotion

  4. Rule Enforcement – respects affinity / anti-affinity constraints

Mental model

If HA answers “Can the VM run?”,
DRS answers “Where should the VM run right now?”

DRS modes (VCF defaults matter)

Mode Behavior VCF expectation
Manual Recommends only ❌ Rare in VCF
Partially Automated Auto-place, manual move ⚠️ Transitional
Fully Automated Auto-place & auto-move ✅ Standard

VCF designs assume Fully Automated DRS.

How DRS makes decisions (simplified)

DRS continuously evaluates:

  • Host CPU utilization

  • Host memory pressure

  • VM demand & entitlements

  • Resource Pool shares & limits

  • Affinity / anti-affinity rules

When imbalance exceeds a threshold:
→ vMotion is triggered
→ No VM restart

DRS vs HA vs “DR” (constant confusion)

Feature Trigger Action
HA Host failure Restart VM
DRS Resource imbalance Live migrate VM
DR Site failure Recover elsewhere

DRS is performance & fairness, not failure recovery.

Affinity & Anti-Affinity rules (important)

DRS can enforce rules such as:

  • “These VMs must stay together”

  • “These VMs must never share a host”

Typical use cases:

  • App + cache locality

  • HA for clustered apps

  • Licensing constraints

⚠️ Over-constraining rules reduces DRS effectiveness.

DRS and Resource Pools (connection)

DRS respects:

  • Resource Pool shares

  • Resource Pool reservations

  • Resource Pool limits

Meaning:

DRS balances within the rules you define — it does not override governance.

DRS inside VCF Workload Domains

In VCF, clusters are typically:

Cluster
├── HA: Enabled
├── DRS: Fully Automated
├── Admission Control: On
└── vMotion: Required

If DRS is disabled:

  • Initial placement degrades

  • Load hotspots form

  • HA restarts may fail due to poor placement

What DRS is not

  • ❌ Not a tenant boundary

  • ❌ Not a scaling mechanism

  • ❌ Not application-aware

  • ❌ Not Kubernetes scheduling

DRS operates strictly at the VM + host layer.

Anti-patterns

❌ Disabling DRS for “predictability”
→ Creates hotspots and manual ops

❌ Overusing VM-Host affinity rules
→ Breaks automatic balancing

❌ Expecting DRS to fix under-provisioning
→ It redistributes; it does not create capacity

Key takeaways

  • DRS is the brain of a vSphere cluster

  • It continuously optimizes where VMs run

  • VCF assumes DRS is enabled and automated

  • HA + DRS together provide resilience + efficiency

IaaS

IaaS (infrastructure as a service) is a cloud computing model where you rent fundamental IT infrastructure—instead of owning and managing physical hardware.

IaaS provides fundamental computing resources where the consumer can deploy and run arbitrary software, including operating systems and applications.

In plain terms:

IaaS = someone else runs the data center; you control the machines.

What you get with IaaS

The cloud provider gives you raw building blocks:

  • Compute → virtual machines (VMs)
  • Storage → block, object, file storage
  • Networking → virtual networks, IPs, load balancers, firewalls

You decide:

  • Which OS to run
  • How to configure the machine
  • What software to install
  • How to secure and scale it

Responsibility Split

Layer Who manages it?
Physical data center Cloud provider
Servers & disks Cloud provider
Virtualization Cloud provider
Operating system You
Patching & hardening You
Applications You
Data You

Analogy:

  • Owning hardware → buying a house
  • IaaS → renting an empty apartment (walls and plumbing are there; furniture and rules are up to you)

Common IaaS examples

  • Amazon Web Services → EC2
  • Microsoft Azure → Virtual Machines
  • Google Cloud → Compute Engine

All of these give you VMs, networks, and storage—but no application opinion.

How IaaS compares to PaaS and SaaS

Model What you control Typical use
IaaS OS → app → data Max flexibility, infra-heavy systems
PaaS App & data only Developers want speed
SaaS Almost nothing End-user software

Example:

  • IaaS → “Give me a Linux box”
  • PaaS → “Run my app”
  • SaaS → “Just let me use the app”

Why people choose IaaS

Pros

  • Maximum control and flexibility
  • Works for legacy apps
  • Fine-grained networking & security
  • Familiar mental model for infra engineers

Cons

  • You manage OS patching and security
  • Higher operational burden
  • Easier to misconfigure
  • Less opinionated guardrails

In VMware / VCF Terms

  • vSphere-style VM workloads map very naturally to IaaS
  • VCF abstracts hardware, but still exposes infra
  • Kubernetes-on-VCF starts moving you away from pure IaaS toward PaaS-like models

That’s why a workload is a workload is a workload: From the platform’s perspective, VMs and pods are both just scheduled consumers of infrastructure.

Summary

  • IaaS = rented infrastructure
  • You control the OS and above
  • Most flexible, most responsibility
  • Foundation layer beneath PaaS and SaaS
  • Conceptually closest to traditional virtualization

SDKs

These are language bindings.

Where available, VMware/Broadcom provides SDKs that wrap REST APIs in native language constructs:

  • Object models
  • Typed requests/responses
  • Auth helpers
  • Pagination, retries, error handling

These SDKs are ideal when:

  • You are building a platform or product
  • You need maintainable, testable code
  • Shell scripts are no longer enough

What VCF APIs Intentionally Do Not Do

They do not:

  • Replace vCenter or NSX APIs
  • Provide low-level tuning knobs
  • Allow unsupported lifecycle paths
  • Let you bypass validation logic

This is by design. VCF optimizes for predictability and supportability over raw flexibility.

Common Usage Patterns

GitOps-style infrastructure

  • Git declares desired domains / versions
  • Pipeline calls SDDC Manager APIs
  • Platform converges safely

Enterprise portals

  • Internal UI talks to VCF Automation APIs
  • End users never see infrastructure details

Day-2 ops automation

  • Health checks
  • Controlled upgrades
  • Capacity expansion

All API-driven, no UI clicks.

Summary

  • VCF SDK APIs are the supported automation surface
  • SDDC Manager APIs control lifecycle and topology
  • VCF Automation APIs control consumption and governance
  • CLIs are just API clients
  • Using these APIs preserves VCF invariants and supportability

VCF Installer

VCF Installer is the one-time (or rare-use) bootstrap and bring-up tool for VMware Cloud Foundation.

VCF Installer’s sole purpose is to stand up the very first VCF environment and hand control over to SDDC Manager.

If SDDC Manager is the factory control system, VCF Installer is the factory construction crew that builds the factory in the first place.

Core Purpose

VCF Installer exists to answer a very specific question:

“Given raw hardware, how do I get to a fully functional VCF Management Domain with SDDC Manager running?”

Once that answer is executed successfully, VCF Installer is done.

Primary Responsibilities

1. Day-0 Bring-Up (Bootstrap)

VCF Installer is responsible for:

  • Validating hardware readiness
  • Validating network configuration
  • Validating DNS / NTP / IPAM
  • Validating credentials and certificates
  • Orchestrating the initial deployment of:
    • ESX on hosts
    • vCenter Server
    • NSX
    • SDDC Manager itself
    • Management Domain components

This is Day-0 only work.

2. Environment Validation & Guardrails

Before anything is deployed, VCF Installer runs extensive prechecks, including:

  • Hardware compatibility (HCL alignment)
  • Network reachability
  • MTU consistency
  • IP pool correctness
  • Time synchronization
  • DNS forward/reverse lookup
  • Credential access

Think of it as a fail-fast safety gate: Better to stop early than build a broken cloud.

3. Declarative Input, Imperative Execution

VCF Installer typically operates with:

  • Structured input (JSON / guided wizard)
  • Explicit topology definitions
  • Explicit IP assignments

You declare what you want, and the installer executes a deterministic build.

There is no concept of “incremental evolution” here: That belongs to SDDC Manager.

What VCF Installer Is Not

It is not:

  • A lifecycle manager
  • A long-running control plane
  • A tenant or consumption interface
  • A policy engine
  • An operations tool

After bring-up:

  • It is not used daily
  • It does not manage upgrades
  • It does not manage workload domains

Those responsibilities move entirely to the SDDC Manager.

Relationship to SDDC Manager

Phase Tool
Day-0 (Initial Build) VCF Installer
Day-1+ (Operate & Expand) SDDC Manager

Once installation completes:

  • SDDC Manager becomes the authoritative owner
  • VCF Installer steps out of the picture

You can think of it as:

VCF Installer creates SDDC Manager; SDDC Manager creates and manages everything else.

Why VCF Installer Exists Separately

A natural question is: “Why not just use SDDC Manager from the start?”

Because:

  • SDDC Manager itself must be deployed
  • That deployment requires validated sequencing
  • That sequencing must run before SDDC Manager exists

VCF Installer solves this bootstrapping paradox.

Kubernetes Analogy

  • VCF Installer → kubeadm init
  • SDDC Manager → Kubernetes control plane (after init)
  • VCF Automation → Higher-level UX / APIs for consumers

You don’t run kubeadm init every day, and you don’t run VCF Installer every day either.

Summary

  • VCF Installer is a Day-0 tool
  • It bootstraps the Management Domain
  • It deploys the SDDC Manager itself
  • It performs strict prechecks and validation
  • After success, it is effectively retired

VCF Operations Orchestrator

VCF Operations Orchestrator provides a library of workflows that perform common management tasks. VCF Operations Orchestrator also provides libraries of the individual actions that the workflows perform.

Workflows combine actions, decisions, and results that, when performed in a particular order, finish a specific task or a specific process in a virtual environment.

Workflows perform tasks such as provisioning virtual machines, backing up, performing regular maintenance, sending emails, performing SSH operations, managing the physical infrastructure, and other general utility operations. Workflows accept inputs according to their function.

You can create workflows that run according to defined schedules, or that run if certain anticipated events occur.

Information can be provided by you, by other users, by another workflow or action, or by an external process such as a Web service call from an application.

Workflows perform some validation and filtering of information before they run.

Workflows can be chained. That is, they can call upon other workflows. For example, you can have workflow that calls up another workflow to create a new virtual machine.

You create workflows by using the VCF Operations Orchestrator’s integrated development environment (IDE), that provides access to the workflow library and the ability to run workflows on the workflow engine.

The workflow engine can also take objects from external libraries that you plug in to the VCF Operations Orchestrator. This allows you to customize processes or implement functions that third-party applications provide.

Is It the Same Thing as **vSphere Cluster

Yes’nt.

A vSphere Cluster and a Supervisor Cluster are not the same thing. However, they are so tightly-related in a Supervisor-enabled Namespace, you can “colloquially” say that they are the same.

Yet, they sit at different abstraction layers.

vSphere Cluster

This is the base construct.

A vSphere Cluster is a classic virtualization construct:

  • A group of [[ESX] hosts
  • Managed by vCenter
  • Provides:
    • [DRS] (scheduling)
    • [HA] (failover)
    • Resource Pooling (CPU / memory)
    • Shared storage and networking

However, vSphere Cluster alone does not imply

  • Any Kubernetes
  • Any namespaces
  • Any tenants
  • Any container workloads

You can run only VMs, forever, and never enable anything Kubernetes-related.

Supervisor Cluster

This is the layer on top.

A Supervisor Cluster is not a separate cluster of hosts.

It is:

A Kubernetes control plane embedded into a vSphere Cluster

When you enable the Supervisor feature on vSphere:

  • vSphere deploys Kubernetes control plane VMs
  • ESX hosts become Kubernetes worker nodes
  • vSphere exposes:
    • Supervisor Namespaces
    • Pod Workloads
    • Guest Clusters (e.g., VKS)
  • vSphere becomes Kubernetes-aware

The key point is: The same ESX hosts of the same vSphere Cluster are used, but now they serve two schedulers**:

  • vSphere (VMs)
  • Kubernetes (pods / guest clusters)

So the formal relationship is:

A Supervisor Cluster is a capability of a vSphere Cluster, not a different cluster type.

But, it walks like a duck, talks like a duck, and it surely is a very-sophisticated duck: Operationally, and practically, it is the same thing.

A Supervisor Cluster is a Kubernetes control plane that runs on and transforms a vSphere Cluster into a Kubernetes-aware platform.

For the ontology-enthusiasts out there:

Aspect vSphere Cluster Supervisor Cluster
Nature Virtualization construct Kubernetes construct
Defined by vCenter Kubernetes control plane
Exists without K8s? Yes No
Exists without vSphere? No No

Failure domains

This matters a lot.

  • vSphere cluster failure → everything dies
  • Supervisor control plane failure → VMs keep running

Although subtle, this distinction becomes very important when discussing:

  • Security boundaries
  • Upgrade paths
  • Failure isolation
  • Scheduling semantics
  • VCF Automation mappings

In a Supervisor-enabled vSphere, the Supervisor Cluster and the vSphere Cluster are co-resident and inseparable operationally, but they remain distinct architectural constructs with different control planes and failure domains.
