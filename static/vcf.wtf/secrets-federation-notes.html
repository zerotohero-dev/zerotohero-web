  • Sync into an internal OpenBao (or internal Vault) via ESO

  • You want no path collisions and no possibility of cross-tenant access

There are two separate problems:

  1. Storage namespace collision in internal OpenBao
  2. Runtime authorization when workloads fetch secrets (injection)

Mapping external Vault paths into internal OpenBao without conflicts

this is the first half of the problem.

Injecting secrets without cross-tenant association

Based on the official VCF Secret Service docs, a lot of the “last-mile consumption” is already solved (create secret → store it → inject it into pods/VMs as files), but tenancy isolation is explicitly coarse and largely stops at the Supervisor Namespace boundary. It’s there, but it’s not the fine-grained. That is any OpenBao secret bound to a namespace is visible to any workload in that namespace.

That should be fine since tenants are effectively namespace-bound.

Delivery Format

The “Managing Secrets…” doc spells out delivery formats:

  • vSphere Pods → file mount under /vault/secrets/<user-provided-name>
  • VM Service VMs → disk-based transfer + read-only mount
  • VKS (guest) cluster pods → file mount similar to vSphere Pods

This solves consistent delivery semantics per workload type.

All service accounts have read access to all secrets of that namespace.

^ that is the granularity we get. Which may or may not be fine based on the customers’ requirements.

Also it’s worth noting that any notion of Organization/Project tenancy in the Secret Store model does not exist by architecture (which is also fine because secrets are namespace-bound, and tenants are also namespace-bound).

None of the flows refer to Organization/Project at all. They are consistently “pick a Supervisor namespace; secrets live there”.

That’s consistent with what we discussed earlier: governance constructs don’t appear at runtime.

The docs show a pattern like secret/data/<namespace>/<secretName> for retrieval.

  1. Secrets are stored in a “corresponding namespace” (meaning: per Supervisor namespace).
  2. VKS isolation is only at Supervisor namespace level (explicit).
  3. Within a namespace, all service accounts can read all secrets (explicit).

So the tenancy story is not missing — it’s just coarse:

Tenant isolation = Supervisor Namespace isolation (and nothing finer is guaranteed).

Secret Mapping

A safe mapping consistent with the doc’s retrieval convention would look like:

  • External: credentials/db/finance
  • Internal: secret/data/<supervisor-namespace>/credentials/db/finance
    (or secret/data/<supervisor-namespace>/<some-secret-name> that contains those K/Vs)

That avoids cross-tenant collisions because the Supervisor namespace becomes the first path segment.

There is a Single OpenBao Instance

According to the docs, all tenants share the same unified internal secrets store (OpenBao).

How Does Injection Refer to This

In the doc’s annotation model, you specify a secret “path” in the injector annotation. They show:

  • vault.hashicorp.com/agent-inject-secret-...: "secret/data/demo/db-cred"
    Inject Secrets in to a vSphere …

So for Pepsi, if you used the flattened name approach, you’d inject like:

  • secret/data/pepsi/auth__finance__db__creds
    …and because the auth role is tied to the namespace (example role demo), Pepsi workloads would authenticate with role pepsi and read from the pepsi/* prefix.

Takeaways

  • ✅ In this doc model, within a tenant’s Supervisor namespace, any Pepsi workload can access any Pepsi secret.
  • ❌ Restricting secrets to only pods in a specific Kubernetes namespace inside the tenant is not supported here (especially for VKS guest clusters).
  • ✅ OpenBao paths look like secret/data/<supervisor-namespace>/<secretName> (KV v2 style).
  • ✅ In shared OpenBao, prevent collisions by ensuring the first path segment is the tenant namespace (pepsi, cocacola, …).

Cross-Tenant Access to Secrets

VCF Secret Store Service does have guardrails that prevent a Coca-Cola workload from mounting Pepsi secrets, as long as you stay within the supported model.

But those guardrails are namespace-scoped, not “tenant-aware in the abstract”, and they rely on how identities and roles are provisioned, not on OpenBao magically inferring tenancy.

The guardrails are not:

  • “OpenBao automatically understands tenants”
  • “Vault paths are globally protected by magic”
    They are:

Guardrail #1 — Namespace-scoped identity issuance

The Vault/OpenBao token that a workload receives is issued in the context of a specific Supervisor Namespace.
That token is not shared across namespaces.
So:

  • A Coca-Cola workload never gets the same token as a Pepsi workload
  • Tokens are not reusable across namespaces

This is enforced by Supervisor, not by Kubernetes RBAC alone.

Guardrail #2 — Namespace-scoped auth roles

The injector examples in the docs consistently show:

  • a role name tied to the namespace (e.g. demo)
  • secrets read from secret/data/<namespace>/...
    That implies (and matches how VMware typically wires these systems):

    Each Supervisor Namespace has a corresponding OpenBao role and policy.
    You don’t see the policy YAML in the docs, but the model only works if:

  • role pepsi → policy allows secret/data/pepsi/*
  • role cocacola → policy allows secret/data/cocacola/*
    Otherwise the examples would be unsafe.

Guardrail #3 — Admission + injector coupling

A pod can’t just arbitrarily say:
vault.hashicorp.com/agent-inject-secret-foo: secret/data/pepsi/db
from another namespace, because:

  • the injector runs with Supervisor context
  • the auth role used is bound to the namespace
  • the token issued is constrained by that role
    So even if a Coca-Cola pod requests a Pepsi path, the read fails.
    That’s the critical point: request ≠ authorization.

ref:

“All service accounts have read access to all secrets of that namespace.”

That sentence is not saying:

“All namespaces can read all secrets.”
It’s saying:
“Within one Supervisor Namespace, we don’t give you finer-grained scoping.”

Adding Federation In

If we ensure that:

  • Pepsi secrets are materialized under secret/data/pepsi/...
  • Coca-Cola secrets under secret/data/cocacola/...
    Then VCF Secret Store Service guardrails already align with that:
  • Pepsi workloads authenticate as “Pepsi namespace”
  • Pepsi role can only read secret/data/pepsi/*
  • Coca-Cola role cannot read Pepsi paths
    So yes:

If we respect the namespace-prefix contract, the rest of the consumption plumbing is already handled safely by VCF Secret Store Service.

We are not punching a hole in the security model by doing the sync.

When Can This Model Break

The model breaks only if you bypass the intended trust boundaries, for example:

  • A single ESO instance with a token that can write/read all tenant prefixes
  • Manually creating secrets in the wrong namespace
  • Reusing roles or tokens across namespaces
  • Exposing OpenBao externally and letting workloads authenticate out-of-band

Those are operator errors, not product gaps — but they’re exactly what auditors look for.

One Line for the Design Doc

VCF Secret Store Service enforces tenant isolation at the Supervisor Namespace boundary by issuing namespace-scoped identities and policies for secret access. As long as tenant secrets are materialized under their corresponding namespace paths, cross-tenant secret access is prevented by design. Fine-grained intra-namespace isolation is intentionally not provided.
