vSphere Pod

A vSphere Pod is a Kubernetes Pod whose sandbox runs inside a lightweight VM (via ESX]), not inside a container runtime on a shared Linux node.

So:

  • API semantics → Kubernetes
  • Runtime semantics → vSphere / VM

Why vSphere Pods Exist

Traditional Kubernetes Pods:

  • Share a Linux kernel
  • Rely on namespaces + cgroups
  • Are fast and efficient
  • But have weaker isolation (from a hypervisor POV)
    vSphere Pods were created to:
  • Give VM-grade isolation
  • While keeping Pod-level developer UX
  • For multi-tenant / regulated environments
    This matters a lot for:
  • Service providers
  • Telco
  • Regulated enterprise workloads
  • “Pods from different customers must never share a kernel”

Comparison

Aspect Regular K8s Pod vSphere Pod
Where it runs On a Linux node Inside a micro-VM
Kernel Shared host kernel Dedicated kernel
Isolation Namespace / cgroup Hypervisor-level
Failure blast radius Node-wide Pod-local

Regular Kubernetes:

Node (Linux)
└── kubelet
    ├── Pod A
    ├── Pod B
    └── Pod C

vSphere Pods:

ESXi
├── Pod-VM A
├── Pod-VM B
└── Pod-VM C
  • No shared container runtime
  • No shared kubelet per “node”
  • Each pod is scheduled directly by vSphere

Kubernetes still thinks it’s scheduling Pods — but vSphere translates that into VM placement.

So in vSphereland, “One Pod ~= Tiny VM”



Scheduling is Co-Owned By K8s and vSphere

  • Kubernetes:
    • Decides which pod should run
  • vSphere:
    • Decides where the pod-VM runs
    • Enforces CPU, memory, NUMA, HA, DRS
      This is what “Kubernetes-native mode” really looks like under the hood.

Networking and storage are vSphere-native

From the Pod’s perspective:

  • It has an IP
  • It mounts volumes
  • It exposes ports
    Under the hood:
  • Networking is NSX-backed
  • Storage is datastore-policy-backe
  • Attach/detach happens at VM granularity
    Again: Pod UX, VM machinery.

Developers (mostly) Won’t Care

Despite the VM runtime, vSphere Pods still use:

  • Pod specs
  • Containers
  • Init containers
  • ConfigMaps
  • Volume mounts
  • Pod lifecycle (create/delete/restart)
  • Kubernetes Services (with some constraints)
    So developers mostly don’t care—and that is by design.

Conceptual Model

  • Regular Pod → process in a shared OS
  • vSphere Pod → process in its own micro-VM
    Or:
  • Regular K8s → Linux container model
  • vSphere Pods → “micro-VM container” model
    Essentially:
  • vSphere Pods are not regular Kubernetes Pods
  • They are Pod-shaped workloads backed by per-Pod VMs
  • Kubernetes API semantics remain intact
  • Runtime isolation is hypervisor-level
  • This is intentional and central to Supervisor’s design
