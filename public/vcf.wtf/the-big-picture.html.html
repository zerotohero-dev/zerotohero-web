At a High Level

VMware Cloud Foundation (VCF) is a unified platform to design, plan, deploy, and manage your SDDC by using automation capabilities.

A Workload is a Workload is a Workload

That is one of the core tenents of VCF, and thus, it requires clarification before anything else.

In VCF-speak, the phrase “a workload is a workload” is not a tautology—it’s a deliberate abstraction boundary.

It means that VCF intentionally collapses many very different execution models into a single management concept, and refuses to care about their internal form unless it must.

Below is the precise meaning, layered from philosophy, to architecture, to consequences.

From VCF’s point of view, a workload is simply “something that consumes resources and is governed by policy.”

It is not:

  • a VM
  • a Container
  • a Pod VM
  • a Kubernetes pod
  • a Guest Cluster
  • a Helm release

Those are implementation details.

VCF asks only:

  • Where does it run?
  • Who owns it?
  • What resources does it consume?
  • What policies apply?
  • What lifecycle rules govern it?

If those questions can be answered, VCF is satisfied.

But, Why Does VCF Need This Abstraction

VCF exists above:

  • ESX
  • vSphere Clusters
  • vSphere Supervisor
  • Kubernetes
  • VKS
  • VMs
  • Containers

If VCF tried to manage each kind of workload differently, it would:

  • explode in complexity
  • duplicate logic
  • couple business policy to runtime mechanics

So instead, VCF defines:

Workload = Policy + Placement + Lifecycle + Resources

Everything else is delegated downward.

Examples of Workloads

All of the following are equally valid workloads in VCF:

Example Internally From VCF’s view
Traditional VM vSphere VM Workload
Pod Workload vSphere Pod on Supervisor Workload
VKS Guest Cluster Kubernetes cluster of nodes Workload
Stateful app Pods + PVCs Workload
Stateless service ReplicaSets Workload
Helm-installed app CRDs + Pods Workload

VCF deliberately does not rank or privilege any of these.

In VCF, a workload is anything that consumes platform resources and is subject to platform policy—its internal shape is intentionally irrelevant.

  • VCF intentionally erases runtime distinctions
  • Governance > implementation
  • Supervisor is the abstraction bridge
  • If two things consume resources and obey policy, VCF treats them identically

But, How is that Possible

vSphere Supervisor makes it possible.

vSphere Supervisor converts:

  • Pods to lightweight VMs
  • Kubernetes objects to vSphere-managed entities
  • Namespaces to policy and tenancy units
  • From below, everything is Kubernetes.
  • From above, everything is vSphere resources.

VCF sits above Supervisor, so it sees:

  • Resource Pools
  • Namespaces
  • Quotas
  • Policies

The kind or form of the workload becomes irrelevant.

VCF does NOT care:

  • whether the workload uses containerd or not
  • whether it’s a PodVM or classic VM
  • whether it runs Go, Java, or Bash
  • whether it was deployed via Helm, kubectl, or Terraform
  • whether it is “cloud-native” or “legacy”

Those are runtime-layer concerns, not platform-layer concerns.

A power grid doesn’t care if you plug in a toaster or a server rack—only how much power you draw and what safety rules apply.

This is why statements like these are true:

  • “Pod workloads are not special.”
  • “VM workloads are not special.”
  • “Guest clusters are just workloads.”
  • “Infrastructure doesn’t care about your app model.”

So, you don’t fight against the platform to fit the workload in; you design the workload to fit the platform’s (opinionated) invariants.

To internalize VCF, don’t think in:

  • Pods vs VMs
  • Containers vs Processes
  • Control Planes vs Data Planes

Instead, think in Governance:

  • Who owns this?
  • What are its limits?
  • How is it created and destroyed?
  • How does it fail?
  • How is it audited?

Don’t anchor your designs to what runs. Anchor them to how they are governed.

vSphere

#architecture

At its core, vSphere is a compute, storage, and network abstraction layer: A distributed operating system for data centers.

It manages large collections of infrastructure, such as CPUs, storage, networking, GPUs, and DPUs as a seamless and dynamic operating environment, and also manages the complexity of a data center.

 
It virtualizes physical infrastructure into managed CPU, storage, and networking resources.

vSphere
├── ESX (data plane)
├── vCenter Server (control plane)
└── Integrated services (HA, DRS, vMotion, etc.)

vSphere is a distributed system that provides the following:

Primitive What it abstracts
Compute CPU scheduling, memory isolation
Storage Policy-driven persistence
Networking Virtual L2–L7 constructs
Identity RBAC, tenancy
Lifecycle Placement, HA, DRS

ESX and vCenter

The two core components of VMware vSphere are ESX and vCenter.
Whereas vSphere Client is the Web UI that interacts with the APIs of vSphere components.

vSphere is a product contract, not a system.
ESX and vCenter are the systems that implement that contract.

Lifecycle Management

Here is how vSphere facilitates LCM:

  • Lifecycle management via pre-staging ESX images.
  • Near-zero-downtime live patching for ESX.

Workload Performance

vSphere as a platform, manages workload performance in several dimensions:

  • With its GPU support, vSphere improves workload performance for modern AI workloads
  • DRS enables automatic load balancing of resources allocated to workloads in a vSphere Cluster.
  • Storage DRS optimizes VM data placement as the VM is created and used over time.

Business Continuity

Moving workloads seamlessly across the fabric, and ensuring things are always ship-shape, is what vSphere excels at:

  • High Availability with vSphere automatically restarts your VMs following physical machine failure.
  • Fault Tolerance provides continuous availability of any application in the event of a hardware failure with no data loss or downtime.
  • vMotion enables live migration of virtual machines with no disruption to users or loss of service, eliminating the need to schedule application downtime for planned server maintenance.
    • Storage vMotion avoids downtime for planned storage maintenance.

Security

At vSphere security is by default, not an afterthought:

  • Identity Federation with Microsoft Entra ID, ADFS and Okta.
  • Virtual Machine Encryption: Data-at-rest encryption for virtual machine data and disks.
  • vSphere Trust Authority: Remote attestation for sensitive workloads.

Is vSphere “ESX and vCenter”

Yesn’t.

Saying “vSphere is ‘ESX and vCenter’.” is colloquially common, but technically incomplete.

See vSphere, ESX, and vCenter for a more nuanced analysis.

You might also want to check vSphere Constructs for an idea of what vSphere directly controls, and what capabilities it has.

NSX

#tenancy #architecture

NSX enables network connectivity, operations, and scale. It uses a software-defined approach to networking that spans data centers, clouds and application frameworks.

NSX abstracts network services from the underlying hardware, enabling programmable infrastructure through APIs. You can implement policies and workflows to automate tasks like network provisioning, security enforcement, load balancing, streamlining operations, and improving efficiency.

Layer 2–Layer 7 Network Stack and Overlay Services

NSX provides a complete set of logical networking capabilities and services, including logical switching, routing, distributed load balancing, virtual private network (VPN), quality of service (QoS), and monitoring.

Switching

Enable logical Layer 2 overlay extensions across a routed (Layer 3) fabric within and across data center boundaries.

Routing

Dynamic routing between virtual networks that is performed in a distributed manner in the hypervisor kernel, and scale-out routing with active-active failover with physical routers. Static routing and dynamic routing protocols are supported, including support for IPv6.

Virtual Routing and forwarding (VRF)

Complete data plane isolation among tenants with a separate routing table, network address translation (NAT), and edge firewall support in each VRF on the 

NSX Tier-0 gateway

Support for bridging between VLANs configured on the physical network and NSX overlay networks, for seamless connectivity between virtual and physical workloads.

VPN

Site-to-site and unmanaged VPN for cloud gateway services.

Multi-Site Networking and Disaster Recovery

By using NSX, you can move workloads between racks and data centers by decoupling them from the underlying networks.

Federation

Centralized policy configuration and enforcement across multiple locations from a single pane of glass, enabling network-wide consistent policy, operational simplicity, and simplified disaster recovery architecture.

Multi-Cloud Networking

Enable consistent networking and security across data center sites, and across private cloud boundaries, irrespective of underlying physical topology or cloud platform.

Network Automation with Declarative APIs

NSX can manage the provisioning, deployment, operations and retirement of networking infrastructure and applications from a central control pane.

You can automate infrastructure operations and offload manual tasks with workflows and agile templating using tools such as VCF Automation, Terraform, and Ansible. 

Self-Service Networking with VPCs and Multi-Tenancy

VPCs provide isolated network environments within a shared infrastructure. You can use Virtual Private Cloud (VPC) constructs to provision and manage network resources to business units or project teams independently, with granular control over networking policies and resource allocation.

vSAN

vSAN aggregates local or direct-attached capacity devices of all ESX hosts in a cluster and creates a single storage pool shared across all ESX hosts in the 
vSAN cluster.

Use vSAN as a VI administrator or storage administrator to enhance storage performance by aggregating local storage into a single shared pool across all hosts in a vSAN cluster.

You can use vSphere features that require shared storage, such as vSphere HA, vSphere vMotion, and vSphere DRS, without the need for external shared storage.

Unified Operations and Lifecycle Management

vSAN provides operational consistency from edge to cloud, with complete visibility and lifecycle management of compute and storage from a single UI.

VM-Centric Policy-Based Management

You can perform consistent, VM-centric operations by using policy-based management. Within the vSAN ESA, per-VM policies increase flexibility of management, enabling compression policies, or customize data services on a per-VM basis.

About VCF

#architecture

See also: VMware Cloud Foundation

  • VCF is a full-stack Infrastructure as a Service (IaaS) platform that delivers software-defined compute, storage, networking, security, and management.
  • Integrated automation enables a self-service platform to deploy VMs and containers for developer agility.
  • Hardened platform offering built-in resilience, scaling, and clustering for non-stop operations.
  • It provides automation and orchestration to simplify Day 0, Day 1, and Day 2 tasks.
  • It simplifies full stack deployment.

The purpose of VCF is to design, plan, deploy, and manage your SDDC by using automation capabilities.

VMware Cloud Foundation is a unified private-cloud platform that combines the scale and agility of public cloud with security and performance of private cloud. delivering increased productivity and lower Total Cost of Ownership (TCO).

VCF Day 0 Tasks

#lifecycle

VMware Cloud Foundation Day 0 tasks are everything required to prepare and bootstrap the platform before any tenant, workload, or higher-level automation exists.

Think of Day 0 as laying the concrete and pouring the foundation—after this, you can build floors (VCF Day 1 Tasks) and move people in (VCF Day 2 Tasks).

  • Day 0: Make the platform exist
  • Day 1: Configure and extend the platform
  • Day 2: Operate workloads on the platform

VCF is very opinionated: nothing meaningful can happen until Day 0 is clean and correct.

Core Categories of VCF Day 0 Tasks

1. Physical and Infrastructure Prerequisites

These are outside VCF but mandatory.

  • Rack & cable hosts
  • Power, cooling, BIOS/firmware
  • NTP, DNS, IPAM
  • VLANs / MTU (often jumbo frames)
  • Out-of-band management (iDRAC / iLO)

If this is wrong, everything later becomes “mysteriously flaky.”

2. ESX Installation and Host Preparation

Before VCF can orchestrate anything, ESX must exist.

  • Install ESX on all hosts
  • Apply compatible firmware and drivers
  • Validate hardware against VCF BOM
  • Configure basic networking (vmnics, MTU)

These hosts are raw compute, not yet part of any SDDC.

3. SDDC Manager Deployment

This is the true start of VCF.

  • Deploy SDDC Manager Appliance
  • Connect it to:
    • DNS
    • NTP
    • Identity source
  • Verify reachability to all ESX hosts

From this point on, VCF owns lifecycle.

4. Initial Management Domain Bring-Up

This creates the core control plane SDDC.

Includes automated deployment of:

  • vCenter Server
  • NSX Manager
  • vSAN
  • ESX Management Cluster

This is where VCF becomes a self-hosting platform.

After this, you no longer treat components individually.

5. Network and Security Baseline

Baseline platform networking is established.

  • NSX fabric initialization
  • Transport zones
  • Edge nodes (management edges)
  • Tier-0 / Tier-1 gateways
  • Initial firewall policies

This defines how everything can talk later.

6. Identity, Access, and Trust

Foundational access control.

  • Integrate vCenter / SDDC Manager with:
    • LDAP / AD / Identity Provider
  • Define platform admin roles
  • Secure service accounts

Without this, automation has no authority.

7. Certificates and Cryptographic Baseline

Often underestimated, always painful if skipped.

  • CA strategy (internal vs external)
  • TLS certificates for:
    • vCenter
    • NSX
    • SDDC Manager
  • Trust alignment across components

This is where future mTLS, OIDC, and zero-trust efforts live or die.

8. Lifecycle and Compliance Baseline

This is about future safety.

  • Validate VCF BOM
  • Enable lifecycle management workflows
  • Confirm patching & upgrade paths
  • Confirm supportability state

If skipped, VCF Day 2 Tasks turns into firefighting.

What Day 0 Explicitly Does NOT Include

  • No tenants
  • No projects
  • No Supervisor Namespaces
  • No Kubernetes workloads
  • No automation pipelines
  • No business logic

Day 0 is pure platform plumbing.

Common Day 0 Failure Modes

Mistake Consequence
Skipping certificate planning Painful re-issuance later
Weak IPAM discipline Broken automation
Ad-hoc networking NSX refactors
Treating vCenter as “just UI” Lifecycle conflicts
Rushing bring-up Fragile management domain

Summary

  • VCF Day 0 = platform existence
  • It is non-negotiable and opinionated
  • Focuses on infrastructure, identity, network, and lifecycle
  • No tenants, no workloads, no business logic
  • Done right → VCF Day 1 Tasks and VCF Day 2 Tasks feel boring (the best outcome)

VCF Day 1 Tasks

#lifecycle

VMware Cloud Foundation Day 1 tasks start after the platform exists and focus on making it usable, consumable, and policy-driven—without yet running real business workloads.

If Day 0 pours concrete, Day 1 frames the building and installs utilities.

  • Day 0 → Platform exists
  • Day 1 → Platform is configurable & consumable
  • Day 2 → Platform is used & operated

Day 1 is where intent, policy, and abstraction appear.

Core Categories of VCF Day 1 Tasks

1. Organizational & Tenancy Modeling

This is where business structure enters the system.

  • Define Organizations
  • Define Projects
  • Map projects to:
    • Supervisor Namespaces
    • Resource boundaries
  • Define quotas and limits

This is not Kubernetes yet—it’s VCF’s business abstraction layer.

2. Identity, RBAC, and Delegation

From “admins only” to multi-role access.

  • Define administrative roles
  • Map identity groups to roles
  • Separate:
    • Organization Administrators (colloquially known as Platform Admins)
    • Project Administrators (colloquially known as Tenant Admins)
    • Developers
  • Enable delegated self-service

This is where “secure Multi-Tenancy” becomes real.

3. Network Consumption Models

Networking shifts from fabric to consumption.

  • Project-level networking
  • Tenant Tier-1 gateways
  • NAT, routing, firewall policies
  • Load balancer enablement
  • Network isolation per project

Infrastructure networking (Day 0) becomes tenant networking.

4. Storage Consumption and Policies

Turn raw storage into governed capacity.

  • Define storage policies (performance, availability)
  • Assign policies to Projects
  • Enable vSAN / NFS / VMFS consumption
  • Set quotas and constraints

This prevents noisy neighbors and capacity abuse.

5. Compute Abstractions

Expose compute without exposing chaos.

  • Define resource pools
  • Assign clusters to projects
  • Enable placement policies
  • Configure HA / DRS behavior

Compute becomes schedulable capacity, not hardware.

6. Kubernetes Enablement

This is where container platforms appear.

  • Enable vSphere Supervisor
  • Configure:
    • Supervisor Namespaces
    • Kubernetes versions
    • CNI / CSI integration
    • VKS clusters

Kubernetes exists because Day 1 enabled it, not because ESX magically does containers.

7. Automation and Self-Service Setup

Where humans stop clicking.

  • Enable VCF Automation
  • Define Blueprints
  • Approval flows
  • Day-1 actions exposed as APIs
  • Service Catalog creation

This is the front door for tenants.

8. Policy, Guardrails, and Governance

This is preventative control, not enforcement after damage.

  • Quotas
  • Naming standards
  • Placement constraints
  • Lifecycle constraints
  • Compliance policies

Day 1 policies determine how painful Day 2 is.

9. Platform Observability

Before workloads, you need visibility.

  • Enable logging pipelines
  • Metrics collection
  • Alert baselines
  • Capacity visibility

You can’t operate what you can’t see.

What Day 1 Explicitly Avoids

  • No real business workloads
  • No app-specific tuning
  • No scaling incidents
  • No break/fix operations

Day 1 is still design-heavy, not ops-heavy.

Common Day 1 Failure Modes

Mistake Impact
Over-modeling orgs/projects Administrative drag
Under-modeling quotas Resource starvation
Skipping automation Ticket-driven misery
Treating Kubernetes as special Broken abstraction
Weak RBAC Security or usability issues

Summary

  • VCF Day 1 = platform enablement
  • Introduces tenancy, policy, automation
  • No real workloads yet
  • Sets the rules for everything that follows
  • Good Day 1 → boring, predictable Day 2

VCF Day 2 Tasks

#lifecycle #operations

VMware Cloud Foundation Day 2 tasks begin once real workloads exist.
This is where theory meets entropy: failures happen, usage grows, humans make mistakes, and the platform must keep working anyway.

If:

  • Day 0 pours the concrete
  • Day 1 frames the building
  • Day 2 is living in it for years

Or:

  • Day 0 → Existence
  • Day 1 → Enablement
  • Day 2 → Sustainment

Day 2 is continuous, not a phase you “finish”.

Core Categories of VCF Day 2 Tasks

1. Workload Lifecycle Operations

This is the obvious part—but it’s only the surface.

  • Deploy workloads (VMs, Pods, clusters)
  • Scale up/down
  • Start / stop / suspend
  • Migrate workloads
  • Retire workloads

2. Monitoring, Alerting, and Observability

Day 2 starts the feedback loop.

  • Platform health monitoring
  • Workload health monitoring
  • Capacity tracking
  • Performance baselines
  • Alert tuning (noise reduction)

Bad alerts = ignored alerts = outages.

3. Incident, Break/Fix and Recovery

The real test of architecture.

  • Host failures
  • Network partitions
  • Storage degradation
  • Control plane issues
  • Application outages

Includes:

  • HA / DRS behavior
  • Failover validation
  • Backup and restore
  • DR testing

If this fails, Day 0 and Day 1 didn’t matter.

4. Security Operations

Security shifts from design to enforcement.

  • Certificate rotation
  • Secret rotation
  • Credential revocation
  • Firewall rule changes
  • Vulnerability remediation
  • Audit and compliance reporting

5. Lifecycle Management and Upgrades

One of VCF’s core promises.

  • [] patching
  • vCenter upgrades
  • NSX upgrades
  • Kubernetes version upgrades
  • Firmware alignment (BOM compliance)

All while:

  • Workloads are running
  • SLAs are enforced

This is why VCF is opinionated.

6. Capacity and Cost Management

Reality: resources are finite.

  • Capacity planning
  • Quota adjustments
  • Overcommit tuning
  • Chargeback / showback
  • Cluster expansion

This is where finance meets engineering.

7. Policy Drift and Governance Enforcement

Day 1 defines policies.
Day 2 fights drift.

  • Detect misconfigurations
  • Enforce standards
  • Handle exceptions
  • Reconcile “special cases”

Without this, entropy wins.

8. Tenant and Developer Experience

Day 2 success is measured socially, not technically.

  • Self-service reliability
  • Automation success rates
  • Time-to-resource
  • Support ticket volume
  • Developer trust

A stable platform feels boring. That’s success.

9. Platform Evolution

Day 2 never freezes architecture.

  • Introduce new workload types
  • Add regions / clusters
  • Enable new services
  • Deprecate old patterns

VCF must evolve without reset.

Common Day 2 Failure Modes

Failure Root Cause
Upgrade outages Weak Day 0 lifecycle discipline
Security incidents Skipped Day 1 guardrails
Capacity crises No Day 2 forecasting
Alert fatigue Poor observability design
Manual firefighting Automation gaps

Day 0 / Day 1 / Day 2 in One View

Phase Question Answered
Day 0 Can the platform exist safely?
Day 1 Can tenants consume it correctly?
Day 2 Can it survive reality?

Summary

  • VCF Day 2 = continuous operations
  • Includes incidents, upgrades, security, scale
  • Validates Day 0 and Day 1 decisions
  • Workload type becomes irrelevant
  • Boring Day 2 is the highest compliment

Multi-Tenancy

#tenancy

See also: Tenants in Supervisor

In a VCF-based platform, tenant isolation is realized through vSphere Supervisor Namespaces, which serve as the enforced execution boundary for workloads.

A single tenant may be associated with one or more Supervisor Namespaces, typically aligned to Projects within an Organization.

Supervisor Namespaces provide hard isolation for compute, networking, storage, and identity, while VCF Automation operates above this layer to deliver governance, policy enforcement, and self-service without weakening those isolation guarantees.

In VMware Cloud Foundation, multi-tenancy is the ability to securely host multiple independent consumers (tenants) on the same physical cloud infrastructure, while preserving isolation, governance, and lifecycle safety.

VCF does not implement multi-tenancy at a single layer.
It is layered, intentional, and asymmetric:

The Three Layers of Tenancy

1. Infrastructure Tenancy (Hard / Physical-ish)

Where it lives

  • Workload Domains
  • Clusters
  • NSX Segments
  • Resource Pools

Who owns it

  • Platform / Cloud Admins

What it provides

  • Strong isolation
  • Failure-domain separation
  • Compliance boundaries

Important

  • Infrastructure tenancy is coarse-grained
  • Tenants do not see this layer

“Which hardware and fabric does this workload ultimately land on?”

2️. Consumption Tenancy (Primary, First-Class)

Where it lives

  • Organizations
  • Projects
  • Policies, quotas, approvals

This layer is primarily owned by VCF Automation.

Organizations

  • Top-level tenant boundary
  • Represents a company, business unit, or customer
  • Defines identity, access, quotas, and governance

Projects

  • Sub-tenants within an organization
  • Environment—or team—scoped (Dev / Test / Prod, Team A / Team B)

This is where true SaaS-like multi-tenancy exists.

3. Execution Tenancy (Where Workloads Actually Run)

Depending on workload type:

VM workloads

  • vSphere constructs
  • Backed by resource pools, folders, NSX isolation

Kubernetes / pod workloads

  • Supervisor Namespaces
  • Mapped (typically) 1-to-1 with Projects
  • Enforced by vSphere + Kubernetes RBAC

Supervisor Namespaces are execution boundaries, not business tenants.

They implement tenancy; they are not the tenant itself.

Organizations, Projects, Namespaces

VCF Infrastructure (shared)
│
├── Organization: Acme
│   ├── Project: Acme-Prod   → Supervisor Namespace ns-acme-prod
│   ├── Project: Acme-Dev    → Supervisor Namespace ns-acme-dev
│   └── Project: Acme-Test   → Supervisor Namespace ns-acme-test
│
├── Organization: Globex
│   ├── Project: Globex-Prod → Supervisor Namespace ns-globex-prod
│   └── Project: Globex-Dev  → Supervisor Namespace ns-globex-dev
  • Organizations = business tenants
  • Projects = operational sub-tenants
  • Namespaces = runtime enforcement

Isolation Guarantees (by Layer)

Layer Isolation Type Enforced By
Organization Identity, policy, quota VCF Automation
Project Resource and approval scope VCF Automation
Supervisor Namespace Runtime and API isolation vSphere Supervisor + K8s
NSX Network micro-segmentation NSX
Workload Domain Failure and lifecycle isolation SDDC Manager

Isolation is compositional, not singular.

Pros

  • Strong isolation without hardware duplication
  • Clear separation of duties
  • Supports enterprise and service-provider use cases
  • Preserves lifecycle determinism

Cons

  • Conceptually complex
  • Multiple abstractions to understand
  • Requires discipline in mapping orgs/projects/namespaces

VCF optimizes for large-scale, regulated environments, not minimalism.

What Multi-Tenancy Means in VCF

In VMware Cloud Foundation, multi-tenancy does not mean soft separation within a shared runtime. Instead, it is defined by:

  • Explicit tenant boundaries
  • Platform-enforced isolation
  • No cross-tenant visibility or control
  • Lifecycle independence between tenants

This design prioritizes predictability, security, and supportability over maximal flexibility.

Administrative Boundaries vs Isolation Boundaries

VCF intentionally separates administrative constructs from enforcement points.

  • Organizations and Projects define who can do what
  • Supervisor Namespaces define what can actually run and where

While Organizations and Projects are essential for governance and delegation, isolation becomes real only at the Supervisor Namespace layer.

At this boundary, VCF enforces:

  • Resource allocation and limits
  • Network segmentation
  • Storage policy enforcement
  • Identity and access scoping

Tenant-to-Namespace Mapping

A common tenant layout in VCF looks like this:

  • One Organization represents a tenant (i.e. Coca Cola)
  • One or more Project exists within that Organization (i.e. Dev / Staging / Prod)
  • Each Project (typically) maps to one Supervisor Namespace (but can map to more than one)
  • A tenant may therefore own multiple namespaces, often separated by:
    • Environment (dev / stage / prod)
    • Application domain
    • Regulatory or compliance boundary

This model allows tenants to scale horizontally without eroding isolation.

The Role of VCF Automation

VCF Automation provides the control plane for consumption, not the enforcement layer.

It is responsible for:

  • Self-service access to platform capabilities
  • Policy-driven approvals and quotas
  • Governance and auditability
  • Abstraction of underlying platform complexity

VCF Automation:

  • Does not bypass Supervisor Namespace isolation
  • Does not weaken platform guardrails
  • Does not act as the isolation mechanism itself

It governs access to platform-enforced boundaries—it does not replace them.

It Is a Layered Cake

If we think in layers:

  • Infrastructure layer: Hosts, clusters, networking, storage (operated outside VCF Automation)
  • Isolation layer: Supervisor Namespaces (hard tenant boundary)
  • Governance layer: Organizations and Projects (who can consume what)
  • Workload layer: VMs, Kubernetes clusters, and services

Multi-tenancy in VCF emerges from the composition of these layers, not from any single construct.

Summary

  • Multi-tenancy in VCF is layered
    • This layered design is foundational to VCF’s security and supportability
      • This design balances power, safety, and scale
  • Organizations and Projects define business tenancy
    • Organizations and Projects provide governance, not isolation
  • Supervisor Namespaces enforce runtime isolation
    • Supervisor Namespaces are the hard isolation boundary in VCF
    • Tenants may span multiple namespaces
  • Workload Domains protect lifecycle and failure domains
  • VCF Automation governs consumption without compromising isolation

In VMware Cloud Foundation, the Management Cluster is the foundational cluster that runs the platform itself.
It hosts the management plane—the services that build, operate, upgrade, and secure the rest of the cloud.

Put simply:

If the Management Cluster is down, you can’t manage the cloud—only the workloads keep running.

The Management Cluster is part of the Management Domain (a special Workload Domain). It is created first during Day-0 bring-up and is non-optional.

What Runs on the Management Cluster

Typical components include:

  • SDDC Manager
  • vCenter Server (management vCenter)
  • NSX Managers
  • vSAN management services

These components do not belong to the tenants. They belong to the platform.

Responsibilities

Platform Control Plane

It runs the services that:

  • Define topology
  • Enforce lifecycle sequencing
  • Validate upgrades
  • Maintain inventory and state

Lifecycle Authority

All VCF lifecycle operations (create domain, expand cluster, upgrade stack) are coordinated from here.

Isolation from Tenant Workloads

Tenant workloads do not run here (by design):

  • No tenant VMs
  • No tenant Kubernetes workloads
  • No customer apps

This protects:

  • Stability
  • Security
  • Upgrade safety

What the Management Cluster Is Not

It is not:

  • A tenant cluster
  • A workload execution environment
  • A shared resource pool for apps
  • Optional or replaceable

Treating it like a general-purpose cluster is a serious anti-pattern.

Management Cluster vs Guest (VI Workload) Clusters

Aspect Management Cluster Guest Cluster
Purpose Run the platform Run tenant workloads
Owner Platform Admin Tenant / Project
Lifecycle Highly controlled More flexible
Blast Radius Very high Scoped
Tenant Access None Yes

Why Does VCF Separate Cluster Kinds

This separation ensures:

  • Predictable upgrades (no tenant interference)
  • Clear blast-radius control
  • Strong security boundaries
  • Operational sanity at scale

It’s the same pattern used by:

  • Kubernetes (control plane vs worker nodes)
  • Public clouds (provider plane vs customer plane)

Kubernetes Analogy

  • Management Cluster → Kubernetes control plane nodes
  • Guest Clusters → Kubernetes worker nodes
  • SDDC Manager → Cluster lifecycle operator

You wouldn’t run apps on the Kubernetes control plane; same rule applies here.

Failure and Availability Considerations

  • Management Cluster is typically:
    • Small (minimum viable size)
    • Highly available
    • Strictly protected
  • If it fails:
    • Existing workloads keep running
    • You lose management, scaling, and upgrades until restored

Summary

  • The Management Cluster runs the cloud, not the workloads
  • It is part of the Management Domain
  • It hosts SDDC Manager, vCenter Server, NSX Managers
  • It is isolated, mandatory, and high-blast-radius
  • Treat it as sacred infrastructure

In VMware Cloud Foundation, a Guest Cluster (workload cluster) is a cluster dedicated to running tenant workloads—VMs or Kubernetes-backed workloads—as opposed to running the platform itself.

It lives inside a Workload Domain (VI Domain) and is where actual business applications execute.

Guest Clusters run applications; Management Clusters run the cloud.

Once a VKS Guest Cluster (also known as “Workload Cluster”) is bound to a Supervisor Namespace:

  • ✅ The cluster knows which Supervisor Namespace it lives in
  • ❌ It has no intrinsic knowledge of:
    • Organization
    • Project
    • Automation constructs
    • Approval flows
    • Business tenancy
    • ^ all of these concepts stop at the **VCF Automation boundary.

When a guest cluster is bound to a Supervisor Namespace, we are talking about a VKS workload cluster created by the vSphere Supervisor.

That implies:

  • The cluster is created via Supervisor APIs
  • It is vSphere Namespaces-scoped
  • It is lifecycle-managed by vSphere
  • It runs on ESX using VM-based worker nodes
VCF Platform
│
├── Management Domain
│   └── Management Cluster  (platform services only)
│
└── Workload Domain (VI Domain)
    ├── Guest Cluster A  ← tenant workloads
    ├── Guest Cluster B  ← tenant workloads
    └── Guest Cluster C  ← tenant workloads
  • A Workload Domain can contain multiple Guest Clusters
  • Each cluster is an operational and failure boundary
  • Tenants never see the Management Cluster, but they consume resources from Guest Clusters

What Runs on Guest Cluster

VM Workloads

  • Traditional virtual machines
  • App servers, databases, legacy systems
  • Managed via vCenter
  • Governed via VCF Automation (orgs, projects, policies)

Pod Workloads

  • Supervisor-backed workloads
  • PodVM-based workloads
  • Supervisor Namespaces mapped to Projects

The cluster itself is workload-agnostic; policy determines what lands there.

Who Owns a Guest Cluster

Role Responsibility
Organization Administrator Creates clusters, upgrades ESX/NSX/vSAN
Project Administrator Consumes capacity, deploys workloads

This split is critical:

  • Tenants use vSphere Cluster-managed VKS cluster bound to Projects
  • Organization Administrators operate clusters

Key Responsibilities of a Guest Cluster

Workload Execution

This is where:

  • CPU, memory, storage are consumed
  • Network traffic flows
  • Availability and performance matter most

Isolation & Blast Radius

Each Guest Cluster provides:

  • Fault isolation
  • Upgrade isolation
  • Security boundaries

A problem in one cluster should not affect others.

Lifecycle Boundaries

Clusters are upgraded:

  • Under SDDC Manager control
  • In a controlled, validated sequence
  • Independently of other clusters (within constraints)

What a Guest Cluster Is Not

It is not:

  • A management plane
  • A place for SDDC Manager or NSX Managers
  • A tenant abstraction by itself
  • A free-for-all environment

A Guest Cluster is shared infrastructure, not a tenant.

Guest Cluster vs Management Cluster

Dimension Management Cluster Guest Cluster
Purpose Run the platform Run applications
Tenant workloads ❌ No ✅ Yes
Owner Platform Admin Shared
Blast radius Global Scoped

Relationship to Multi-Tenancy

There is an important nuance here:

  • Tenancy does not equal cluster
  • Multiple tenants can share:
    • The same Guest Cluster
    • While remaining isolated via:
      • Organizations
      • Projects
      • Namespaces
      • NSX Microsegmentation

Guest Clusters are capacity and fault boundaries, not business boundaries.

Why Does VCF Use Guest Clusters

Pros

  • Strong fault isolation
  • Predictable lifecycle
  • Flexible placement strategies
  • Supports mixed workload types

Cons

  • More moving parts
  • Requires capacity planning
  • Requires clear cluster placement policy

VCF optimizes for enterprise-scale control, not minimal topology.

Summary

  • Guest Clusters are where applications run
  • They live inside Workload (VI) Domains
  • They are operated by Organization Administrators
  • They can host VM and Kubernetes workloads
  • They are shared but strongly governed

A VCF Domain is a core VMware Cloud Foundation platform construct.

  • It is defined and lifecycle-managed by VCF itself (via SDDC Manager).
  • VCF Automation consumes it for tenant self-service and provisioning.
  • VCF Operations observes it for monitoring, health, and capacity.

A VCF Domain is an isolated SDDC unit inside VCF with its own full stack:

Layer Included
Compute One or more ESX clusters
Storage vSAN (or supported external storage)
Networking NSX (T0/T1, segments, firewall)
Management vCenter Server
Lifecycle Patching, upgrades, drift control

Each domain is independently lifecycle-managed (upgrade cadence, patch windows, scaling).

A VCF Domain in a VCF Instance consists of the following components:

  • One vCenter instance.
  • One or more vSphere Clusters with vSphere HA and vSphere DRS enabled.
  • At least one vSphere Distributed Switches per cluster for system traffic and NSX Segments for workloads.
  • Dedicated NSX Manager instance, either specifically for the VCF Domain or shared between VCF Domains, for configuring and implementing software-defined networking.
  • Optional: One or more NSX Edge Clusters, added after you create the Workload Domain, that connects the workloads in the Workload Domain for logical switching, logical dynamic routing, and load balancing.
  • One or more shared storage allocations as principal or supplemental storage.

Types of VCF Domains

There are two types of VCF Domains:

  • Management Domain is created during the deployment or convergence process.
  • Workload Domain runs consumer workloads on Day 2.

VCF Fleet, VCF Instance, and VCF Domains

#architecture

In VMware Cloud Foundation, these three terms describe different scopes of control, stacked hierarchically from largest (fleet) to smallest (domain).

If you only remember one thing:

Fleet → Instance → Domain
Management across clouds → One cloud → Parts of that cloud

VCF Fleet—“All my clouds”

A VCF Fleet is a logical grouping of multiple VCF Instances that are managed, observed, or governed together.

It is a meta-layer, not a runtime construct.

Scope

  • Multiple data centers
  • Multiple regions
  • Core + Edge
  • Possibly different purposes (prod, DR, sovereign, etc.)

What Happens at the Fleet Level

  • Fleet-wide visibility
  • Cross-instance posture (versions, compliance, drift)
  • Centralized operations and Governance
  • Consistent policies and standards

Fleet = “My private cloud estate”

VCF Instance—“One Cloud”

Scope

A VCF Instance is one deployed VCF system:

  • One SDDC Manager
  • One Management Domain
  • One lifecycle authority

This is the unit of deployment and lifecycle.

Characteristics

  • Independently installed
  • Independently upgraded
  • Independently failed / recovered
  • Self-contained control plane

Examples:

  • One on-prem private cloud
  • One regional cloud
  • One large edge aggregation site

VCF Domains—“Functional Partitions Inside a Cloud”

Scope

A Domain is a logical partition inside a VCF Instance, used to separate management concerns, workload types, or failure domains.

There are two major kinds:

Management Domain

  • Created first
  • Hosts:
    • SDDC Manager
    • Management vCenter
    • NSX Managers
  • Runs the platform itself

Workload Domains

  • One or more per instance
  • Host:
    • Tenant workloads
    • Application clusters (i.e., Guest Clusters)
  • Independently scalable and isolatable

Domain = “A major subsystem within a Cloud”

A Canonical Example

VCF Fleet
│
├── VCF Instance (DC-West)
│   ├── Management Domain
│   ├── Workload Domain – Prod
│   └── Workload Domain – Dev
│
├── VCF Instance (DC-East)
│   ├── Management Domain
│   └── Workload Domain – Prod
│
└── VCF Instance (Edge-Region-1)
    ├── Management Domain
    └── Workload Domain – Edge

Each level has different responsibilities and blast radius.

Responsibility and Lifecycle Boundaries

Concept Lifecycle Authority Failure Scope Typical Owner
Fleet Coordinated / Observational None (meta) Central IT / Cloud Ops
Instance SDDC Manager One cloud Platform Admin
Domain SDDC Manager (scoped) Subset of cloud Organization Administrator

Kubernetes Analogy

  • Fleet → Multiple clusters managed centrally
  • Instance → One cluster + control plane
  • Domain → Node pools / system vs workload separation

Why Did VMware Introduce This Terminology

Because customers now have:

  • Many private clouds
  • Many edge clouds
  • A need for consistent Governance at scale

So VMware had to separate:

  • Where lifecycle happens (instance)
  • Where workloads run (domains)
  • Where governance is coordinated (fleet)

Summary

  • VCF Fleet = multi-cloud / multi-site management scope
  • VCF Instance = one complete VCF deployment
  • VCF Domains = functional partitions inside an instance
  • Lifecycle authority lives at the instance
  • Isolation and scale happen via domains
  • Governance and visibility roll up to the fleet

HA

aliases:
  - High Availability

#architecture

HA (High Availability) in VCF is the vSphere-native mechanism that automatically restarts workloads when a physical host fails.

It is not a single VCF feature; rather, VCF relies on and standardizes HA across its managed infrastructure.

Where HA lives in the VCF stack

HA is not invented by VCF. It comes from vSphere and is enforced consistently by VCF.

VMware Cloud Foundation
└── vSphere
    └── Cluster
        └── HA (enabled at cluster level)
            └── VM restart on host failure
  • Implemented by: vSphere
  • Configured in: vCenter Server
  • Consumed by: All VMs running in VCF-managed clusters

VCF ensures HA is correctly configured, validated, and lifecycle-managed as part of its opinionated design.

What Does HA Do

Core Responsibility

Detect host failure → restart affected VMs on surviving hosts

High-Level Workflow

  1. ESX hosts exchange heartbeats
  2. A host stops responding
  3. Remaining hosts declare it failed
  4. vCenter selects new hosts
  5. VMs are restarted from disk

Note that VMs are restarted, not live-migrated

What HA protects Against

✅ Protects Against

  • Physical ESX host failure
  • Power loss to a host
  • PSOD / host crash
  • Host isolation scenarios

❌ Does NOT Protect Against

  • Guest OS crashes
  • Application-level failures
  • Storage corruption
  • Network partition inside the VM
  • Data loss between last write and crash

HA answers “Is the VM running?”, not “Is the app healthy?”

HA vs related VCF Concepts

Feature What it does Relationship to HA
HA Restarts VMs after host failure Baseline resilience
vMotion Live-migrates running VMs Avoids downtime
DRS Load-balances VMs Performance optimization
Fault Tolerance Zero-downtime failover Much stricter, limited
VMware Live Recovery Site-level disaster recovery Cross-site

HA Inside a VCF Workload Domain

In VCF, every Workload Domain cluster is typically:

  • HA-enabled
  • DRS-enabled
  • Validated against design rules

Example:

Workload Domain
└── Cluster
    ├── HA: Enabled
    ├── DRS: Fully Automated
    └── Admission Control: Enabled

VCF expects HA to be on; disabling it usually breaks design assumptions.

Admission Control

HA includes Admission Control, which prevents you from powering on too many VMs.

Purpose:

Ensure enough spare capacity exists to restart VMs after a failure.

If Admission Control says “no”:

  • Nothing is broken
  • You are overcommitted beyond your failure tolerance

This is HA doing its job, not being restrictive.

HA vs Kubernetes-Style HA

VMware HA Kubernetes HA
Restarts VMs Restarts Pods
Minutes-scale recovery Seconds-scale recovery
Infrastructure-level Application-level
Blind to app health Health-probe driven

VCF HA is infrastructure HA, not cloud-native app HA.

Anti-Patterns

❌ Turning off HA to “fit more VMs”
→ Silent loss of failure guarantees

❌ Expecting HA to replace app clustering
→ Wrong layer

❌ Confusing HA with zero downtime
→ There is always a restart

Summary

  • HA in VCF = vSphere HA, standardized and enforced
  • It restarts VMs after host failure
  • It does not provide zero-downtime or app-level resilience
  • It is a non-optional baseline for VCF design assumptions

Fault Tolerance

aliases:
  - FT

#architecture

Fault Tolerance (FT) is a vSphere feature that provides zero-downtime protection for a single VM by running a lockstep secondary copy of that VM on another host.

If the primary host fails, the secondary immediately takes over with no reboot and no data loss.

VCF inherits FT from vSphere; it does not redefine it.

Where FT lives in the VCF stack

VMware Cloud Foundation
└── vSphere
    └── Cluster
        └── Fault Tolerance (per-VM feature)
            ├── Primary VM
            └── Secondary VM (lockstep)
  • Implemented by: vSphere
  • Configured in: vCenter Server
  • Applies to: Individual VMs (opt-in)

What Does FT Do

Core Behavior

  • Executes the same CPU instructions on two VMs simultaneously
  • Streams execution state over a dedicated FT logging network
  • On host failure → instant switchover, no restart

HA restarts VMs.
FT never restarts them.

FT vs HA vs DRS

Feature Failure tolerance Downtime Scope
FT Host failure Zero Single VM
HA Host failure Minutes All VMs
DRS Resource imbalance None Cluster optimization
  • HA: “If the server crashes, reboot it elsewhere”
  • DRS: “Move servers around to keep things fair”
  • FT: “Run two brains in sync at all times”

FT is about continuity, not balancing or recovery speed.

Constraints

FT is powerful but expensive.

Constraints (by design)

  • Limited VM sizes (CPU / memory caps)
  • No snapshots
  • No suspend
  • No Storage vMotion
  • Requires identical CPU features across hosts
  • Heavy network and CPU overhead

Because of this, FT is typically reserved for:

  • Legacy apps that cannot tolerate any downtime
  • Single-instance workloads with no clustering option

FT in a Practical VCF World

VCF environments usually prefer:

  • HA + DRS for infrastructure resilience
  • *Application-level HA (*clusters, replicas, quorum())
  • Kubernetes controllers for pod-level restart
  • Site-level DR (disaster recovery) for disasters

FT remains available, but it is not a default design assumption in VCF.

FT is like live brain mirroring: effective, but costly.

Anti-Patterns

❌ Using FT as a substitute for application HA
❌ Enabling FT broadly “just in case”
❌ Forgetting FT network requirements
❌ Expecting FT to scale like Kubernetes replicas

Summary

  • FT provides true zero-downtime VM failover
  • It is a vSphere feature, inherited by VCF
  • High overhead and strict limits
  • In modern VCF designs, FT is the exception, not the rule

IaaS (infrastructure as a service) is a cloud computing model where you rent fundamental IT infrastructure—instead of owning and managing physical hardware.

IaaS provides fundamental computing resources where the consumer can deploy and run arbitrary software, including operating systems and applications.

  • What software to install
  • How to secure and scale it

Analogy:

  • Owning hardware → buying a house
  • IaaS → renting an empty apartment (walls and plumbing are there; furniture and rules are up to you)

Common IaaS examples

  • Amazon Web Services → EC2
  • Microsoft Azure → Virtual Machines
  • Google Cloud → Compute Engine

All of these give you VMs, networks, and storage—but no application opinion.

How Does IaaS Compare to PaaS and SaaS

Model What you control Typical use
IaaS OS → app → data Max flexibility, infra-heavy systems
PaaS App & data only Developers want speed
SaaS Almost nothing End-user software

Example:

  • IaaS → “Give me a Linux box”
  • PaaS → “Run my app”
  • SaaS → “Just let me use the app”

Why Do people Choose IaaS

Pros

  • Maximum control and flexibility
  • Works for legacy apps
  • Fine-grained networking & security
  • Familiar mental model for infra engineers

Cons

  • You manage OS patching and security
  • Higher operational burden
  • Easier to misconfigure
  • Less opinionated guardrails

In VMware / VCF Terms

  • vSphere-style VM workloads map very naturally to IaaS
  • VCF abstracts hardware, but still exposes infra
  • Kubernetes-on-VCF starts moving you away from pure IaaS toward PaaS-like models

That’s why a workload is a workload is a workload: From the platform’s perspective, VMs and pods are both just scheduled consumers of infrastructure.

Summary

  • IaaS = rented infrastructure
  • You control the OS and above
  • Most flexible, most responsibility
  • Foundation layer beneath PaaS and SaaS
  • Conceptually closest to traditional virtualization

VCF SDKs, APIs, and CLIs

#lifecycle

See also: Gated SDK

In VMware Cloud Foundation, “VCF SDK APIs and CLIs” are programmable control surfaces that expose VCF capabilities for automation, integration, and at-scale operations. They sit above the infrastructure components (vCenter, NSX, vSAN) and below higher-level portals, giving you deterministic, supportable ways to drive the platform.

┌──────────────────────────────────────────────┐
│ Portals / UIs                                │
│  - VCF Automation UI                         │
│  - SDDC Manager UI                           │
├──────────────────────────────────────────────┤
│ SDK APIs & CLIs  ← YOU ARE HERE              │
│  - REST APIs                                 │
│  - Official CLIs                             │
│  - Language SDKs (where provided)            │
├──────────────────────────────────────────────┤
│ Platform Control Planes                      │
│  - SDDC Manager                              │
│  - VCF Automation                            │
├──────────────────────────────────────────────┤
│ Product APIs                                 │
│  - vCenter / NSX / vSAN                      │
└──────────────────────────────────────────────┘

VCF APIs abstract platform intent (*domains, lifecycle, tenancy**), not raw hypervisor knobs.

SDDC Manager APIs

SDDC Manager REST APIs expose everything SDDC Manager knows and controls:

  • Bring-up and expansion
  • Workload Domain lifecycle
  • Host add/remove
  • Cluster creation
  • End-to-end upgrades (LCM)
  • Inventory, versions, and health

These APIs are the authoritative way to automate VCF without breaking supportability.

If it changes VCF topology or lifecycle, it goes through SDDC Manager APIs.

Consumers

  • CI/CD pipelines
  • Infrastructure-as-Code tools
  • Day-2 automation frameworks
  • Internal platforms and runbooks

VCF Automation APIs

These are about consumption and Governance.

APIs for tenant-facing operations such as:

  • Organizations and Projects
  • Policies and approvals
  • Blueprints / templates
  • Workload provisioning (VMs, Kubernetes-backed services)

These APIs sit closer to business intent than infrastructure detail.

  • SDDC Manager APIs → Build and maintain the cloud
  • VCF Automation APIs → Consume the cloud

Official CLIs

These are both human and also script friendly.

SDDC Manager CLI

Used for:

  • Diagnostics
  • Operational workflows
  • Automation-friendly scripting
  • Break-glass scenarios

It is a thin wrapper over SDDC Manager APIs, with guardrails.

This is:

  • A faster iteration than REST during ops
  • An easier integration into shell scripts
  • Provides consistent auth and context handling

CLIs are not separate systems — they are API clients.

SDKs

These are language bindings.

Where available, VMware/Broadcom provides SDKs that wrap REST APIs in native language constructs:

  • Object models
  • Typed requests/responses
  • Auth helpers
  • Pagination, retries, error handling

These SDKs are ideal when:

  • You are building a platform or product
  • You need maintainable, testable code
  • Shell scripts are no longer enough

What VCF APIs Intentionally Do NOT Do

They do not:

  • Replace vCenter or NSX APIs
  • Provide low-level tuning knobs
  • Allow unsupported lifecycle paths
  • Let you bypass validation logic

This is by design. VCF optimizes for predictability and supportability over raw flexibility.

Common Usage Patterns

GitOps-Style Infrastructure

  • Git declares desired domains / versions
  • Pipeline calls SDDC Manager APIs
  • Platform converges safely

Enterprise Portals

  • Internal UI talks to VCF Automation APIs
  • End users never see infrastructure details

Day-2 Ops Automation

  • Health checks
  • Controlled upgrades
  • Capacity expansion

All API-driven, no UI clicks.

Summary

  • VCF SDK APIs are the supported automation surface
  • SDDC Manager APIs control lifecycle and topology
  • VCF Automation APIs control consumption and governance
  • CLIs are just API clients
  • Using these APIs preserves VCF invariants and supportability

VCF Installer

#lifecycle

VCF Installer is the one-time (or rare-use) bootstrap and bring-up tool for VMware Cloud Foundation.

VCF Installer’s sole purpose is to stand up the very first VCF environment and hand control over to SDDC Manager.

If SDDC Manager is the factory control system, VCF Installer is the factory construction crew that builds the factory in the first place.

Core Purpose

VCF Installer exists to answer a very specific question:

“Given raw hardware, how do I get to a fully functional VCF Management Domain with SDDC Manager running?”

Once that answer is executed successfully, VCF Installer is done.

Primary Responsibilities

1. Day-0 Bring-Up (Bootstrap)

VCF Installer is responsible for:

  • Validating hardware readiness
  • Validating network configuration
  • Validating DNS / NTP / IPAM
  • Validating credentials and certificates
  • Orchestrating the initial deployment of:
    • ESX on hosts
    • vCenter Server
    • NSX
    • SDDC Manager itself
    • Management Domain components

This is Day-0 only work.

2. Environment Validation and Guardrails

Before anything is deployed, VCF Installer runs extensive prechecks, including:

  • Hardware compatibility (HCL alignment)
  • Network reachability
  • MTU consistency
  • IP pool correctness
  • Time synchronization
  • DNS forward/reverse lookup
  • Credential access

Think of it as a fail-fast safety gate: Better to stop early than build a broken cloud.

3. Declarative Input, Imperative Execution

VCF Installer typically operates with:

  • Structured input (JSON / guided wizard)
  • Explicit topology definitions
  • Explicit IP assignments

You declare what you want, and the installer executes a deterministic build.

There is no concept of “incremental evolution” here: That belongs to SDDC Manager.

What VCF Installer Is Not

It is not:

  • A lifecycle manager
  • A long-running control plane
  • A tenant or consumption interface
  • A policy engine
  • An operations tool

After bring-up:

  • It is not used daily
  • It does not manage upgrades
  • It does not manage workload domains

Those responsibilities move entirely to the SDDC Manager.

Relationship to SDDC Manager

Phase Tool
Day-0 (Initial Build) VCF Installer
Day-1+ (Operate & Expand) SDDC Manager

Once installation completes:

  • SDDC Manager becomes the authoritative owner
  • VCF Installer steps out of the picture

You can think of it as:

VCF Installer creates SDDC Manager; SDDC Manager creates and manages everything else.

Why VCF Installer Exists Separately

A natural question is: “Why not just use SDDC Manager from the start?”

Because:

  • SDDC Manager itself must be deployed
  • That deployment requires validated sequencing
  • That sequencing must run before SDDC Manager exists

VCF Installer solves this bootstrapping paradox.

Kubernetes Analogy

  • VCF Installer → kubeadm init
  • SDDC Manager → Kubernetes control plane (after init)
  • VCF Automation → Higher-level UX / APIs for consumers

You don’t run kubeadm init every day, and you don’t run VCF Installer every day either.

Summary

  • VCF Installer is a Day-0 tool
  • It bootstraps the Management Domain
  • It deploys the SDDC Manager itself
  • It performs strict prechecks and validation
  • After success, it is effectively retired

VCF Automation

#tenancy #governance

VCF Automation enables IT teams and cloud service providers to deliver a self-service private cloud for AI, Kubernetes, and VM-based applications.

The solution simplifies the process of provisioning and scaling a multi-tenant private cloud with out-of-the-box Infrastructure as a Service (IaaS) offerings, accelerating the time to market for applications while maintaining control using policy-based governance.

VCF Automation provides Governance at the request and lifecycle level—controlling who can deploy which patterns, under what constraints and approvals—while runtime security, drift detection, and continuous enforcement are handled by downstream platforms.

VCF Automation has the following capabilities:

  • Cloud Services: Provides a set of cloud services to meet the needs of application teams
  • Provider Management: Supports IT teams and cloud service providers to effectively manage, deliver, and scale cloud services across multiple vCenter and VCF instances.
  • Organization Management: Provides capabilities that enable the organization administrator of a Line of Business or a tenant to further organize and govern resources allocated to them among application teams.
  • vSphere Supervisor: Provides the capability to run Kubernetes workloads directly on ESX hosts and to create upstream Kubernetes clusters within dedicated namespaces.

VCF Operations

#tenancy

VCF Operations helps your organization build, manage, operate and secure your private cloud infrastructure by deploying and maintaining its fleet-level components, providing visibility and enhanced performance across the workload and infrastructure stack, and helping your organization stay compliant with regulatory standards and organizational guidelines.

VCF Operations has these functional areas:

  • Fleet Management
    • Enables operational consistency and efficient resource management of the VCF infrastructure at scale.
  • Operations Management
    • Provides monitoring and optimization of performance, cost and capacity, and faster troubleshooting.
  • Workload Operations
    •  Ensures that critical applications are running as expected.
  • Performance Monitoring
    • Ensures applications have continuous access to resources
  • FinOps and Capacity
    • Helps you analyze your infrastructure expenses and optimize capacity usage.
  • VCF Operations HCX
    • Provides support for migrating and interconnecting workloads within and across VCF private cloud.
  • Security Management
    • Provides security management to ensure that your VCF private cloud is operationally secure.

VCF Operations does not:

  • Own tenancy boundaries
  • Gate workload creation
  • Enforce business approval flows
  • Define org/project constructs
    Those remain in:
  • VCF Automation
  • Supervisor
  • Upstream automation layers

VCF Operations observes those constructs after the fact.

Day 0 / Day 1 → VCF-A
  - Who can create?
  - What can be created?
  - Under what policies?

Day 2 → VCF-O
  - Is it healthy?
  - Is it over/under-utilized?
  - Is it behaving strangely?

Here is a very high-level look at VCF’s control plane:

VCF (the platform)
├── VCF Automation  (formerly Aria Automation)
├── VCF Operations  (formerly Aria Operations)
├── Supervisor / vSphere
└── Workloads

Here is how things looks from each layer’s point of view:

Layer What exists
VCF Automation Organization, Project
vSphere Supervisor Supervisor Namespace
Guest Cluster Cluster, namespaces, nodes

Once a workload cluster is instantiated in a Supervisor Namespace, it has no awareness of VCF Automation constructs such as Organization or Project—Those concepts exist only at the governance layer (VCF Automation) and are not part of the runtime execution model (Supervisor).

SDDC Manager

#lifecycle #operations

See also: SDDC

SDDC Manager is the foundational lifecycle-management and orchestration engine of VMware Cloud Foundation.

It is responsible for deploying, configuring, upgrading, and maintaining the entire VCF stack as a single, validated system rather than as independent products.

If you think of VCF as a factory-built private cloud, SDDC Manager is the factory control system that ensures every component is installed correctly, stays in a supported state, and evolves safely over time.

UI Deprecation Warning

In VCF 9.0+, Broadcom’s docs state that SDDC Manager UI is being deprecated and that SDDC Manager workflows can now be found in VCF Operations and vSphere Client—so many “platform lifecycle” workflows show up there now. ([

Core Responsibilities

1. Lifecycle Management (LCM)

This is SDDC Manager’s primary job.

SDDC Manager performs end-to-end lifecycle operations across the VCF stack:

  • ESX (hypervisor)
  • vCenter Server
  • NSX
  • vSAN
  • Other optional components.

Key characteristics:

  • Uses validated upgrade bundles
  • Enforces component compatibility
  • Executes sequenced, automated upgrades
  • Provides rollback and precheck logic

This is not just patching: It is system-level lifecycle orchestration.

2. Bring-Up and Expansion

SDDC Manager is responsible for:

  • Initial VCF bring-up
  • Creating:
    • Management Domain
    • VI (Workload) Domains
  • Adding:
    • Hosts
    • Clusters
    • Workload Domains
  • Ensuring:
    • Correct networking
    • Correct certificate trust
    • Correct inventory alignment

Without SDDC Manager, you do not have VCF: That simple!—You will only have a loosely assembled VMware products.

3. Inventory and State Authority

SDDC Manager acts as the system-of-record for:

  • What components exist
  • Their versions
  • Their relationships
  • Their lifecycle state

vCenter manages virtualization.
SDDC Manager manages the cloud system itself.

4. Guardrails and Drift Prevention

SDDC Manager enforces:

  • Supported topologies
  • Supported versions
  • Supported upgrade paths

If you try to:

  • Upgrade NSX independently
  • Modify components outside supported flows
    You risk breaking VCF supportability.

This is intentional: VCF favors predictability over freedom.

What SDDC Manager Is Not

It is not:

  • A tenant portal
  • A developer platform
  • A policy engine
  • A workload orchestrator
  • A replacement for vCenter, NSX Manager, or Kubernetes APIs

Those functions live elsewhere (VCF Automation, vSphere Supervisor, vCenter Server, NSX, etc.).

Component Role
SDDC Manager System lifecycle, topology, version control
vCenter Server Virtualization and cluster management
NSX Manager Networking & security
vSAN Storage
VCF Automation Consumption, policy, approvals
vSphere Supervisor / Kubernetes Workload execution

SDDC Manager manages the cloud itself; everything else manages what runs inside that cloud.

Without SDDC Manager:

  • Enterprises would manually assemble components
  • Version skew would be common
  • Upgrades would be risky and expensive
  • Supportability would collapse

With SDDC Manager:

  • VCF becomes repeatable
  • Lifecycle becomes deterministic
  • VMware/Broadcom can guarantee outcomes

This is why SDDC Manager is non-optional in VCF.

Summary

  • SDDC Manager is the backbone of VCF
  • It owns lifecycle, topology, and compatibility
  • It enforces validated, supported states
  • It is infrastructure-centric, not tenant-centric
  • Without it, VCF does not exist as a product
