vSphere Trust Authority

#security

vSphere Trust Authority (vTA) is a hardware-rooted security feature that separates “who can decrypt and attest” from “who runs the workload.”

It uses trusted execution and attestation so that even infrastructure administrators cannot access protected secrets or plaintext data.

In VCF, vTA is an optional, high-assurance security capability, not a default baseline.

VTA in the VCF Stack

VMware Cloud Foundation
├── Trust Authority Cluster (isolated)
│   ├── Key Provider / Attestation Services
│   └── Hardware-backed trust (TPM, TEE)
│
└── Workload Domains
    └── Protected VMs / workloads
  • Implemented by: vSphere
  • Integrated with: Key Management Interoperability Protocol (KMIP) providers
  • Managed through: vCenter Server

What Does VTA Solve

Classic virtualization problem:

Infrastructure admins can ultimately see everything.

vTA introduces cryptographic separation of duties:

  • Infra admins manage hosts and clusters
  • They do NOT control keys
  • Decryption happens only after hardware attestation

This is importan in:

  • Regulated environments
  • Confidential Computing
  • Insider-threat models
  • Sovereign cloud designs

VTA at a High Level

  1. Trust Authority Cluster boots with TPM-backed identity
  2. Hosts and workloads are attested
  3. Only if attestation passes:
    • Encryption keys are released
    • Protected workloads decrypt and run
  4. If trust breaks:
    • Keys are withheld
    • Workloads remain encrypted / suspended

Access to infrastructure ≠ access to data

Key Components

1. Trust Authority Cluster

  • Separate ESX cluster
  • Minimal admin surface
  • Hosts key and attestation services

2. Trusted Execution

  • Uses CPU security features (e.g., SEV, TDX)
  • Prevents memory inspection

3. External Key Management

  • Keys never live with workload admins
  • Often backed by HSMs

vTA vs Related VCF Security Features

Feature What it protects Threat model
vSAN Encryption Data at rest Disk theft
VM Encryption VM files Storage admins
vSphere Trust Authority Keys and Memory Infra admins
Secure Boot Boot integrity Rootkits
TPM Hardware trust Firmware attacks

vTA is the strongest isolation boundary in vSphere.

vTA in Real VCF Designs

Typical usage:

  • Confidential VMs
  • Telco and Financial Workloads
  • Government / Sovereign Clouds

Typical non-usage:

  • General enterprise clusters
  • Cost-sensitive environments
  • Dev/Test

VTA is deliberately heavyweight.

You can think of vTA as:

“Even if someone owns the data center, they still cannot read the data.”

It is zero-trust applied to virtualization itself.

What vTA is not

  • ❌ Not a secrets manager
  • ❌ Not an identity provider
  • ❌ Not Kubernetes confidential computing (though complementary)
  • ❌ Not required for VCF

Benefits and Liabilities

Pros

  • Strongest insider-threat protection
  • Hardware-rooted trust
  • Clear separation of duties

Cons

  • Additional clusters and hardware
  • Operational complexity
  • Limited workload compatibility

Summary

  • vSphere Trust Authority brings confidential computing to VCF
  • It cryptographically separates infra admins from data access
  • Optional, niche, but extremely powerful
  • Used when “root access is not enough” must be enforced
