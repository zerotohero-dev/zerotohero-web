vSphere Supervisor

#tenancy #kubernetes #architecture

vSphere Supervisor is a control plane built in vSphere that integrates desired state and a single consistent API directly in the hypervisor.

vSphere Supervisor enables native deployment and management of Kubernetes clusters, declarative virtual machines, and  vSphere Pods alongside traditional workloads. It transforms vSphere to an extendable platform by providing a declarative API for self-service provisioning of modern infrastructure services in a resource boundaries enabled by vSphere Namespaces

The vSphere Supervisor lives as a set of Kubernetes control-plane VMs running on ESX hosts, instantiated, wired, and governed by vCenter Server.

vSphere Supervisor:

  • Runs on: ESX
  • Created and managed by: vCenter Server
  • Implemented as:
    • Kubernetes control-plane VMs
    • Distributed across ESX hosts
    • Backed by vSAN / datastores
    • Networked via NSX (or VDS in simpler setups)

When you “enable Supervisor” in vCenter Server:
vCenter Server:

  1. Creates a special-purpose Kubernetes cluster
  2. Instantiates its control plane as VMs
  3. Pins it tightly to:
    • Cluster membership
    • Networking
    • Storage
  4. Treats it as part of the infrastructure, not a tenant workload

So the Supervisor is:

  • Not a workload cluster
  • Not user-managed Kubernetes
  • Not a vSphere product you install

It is an infrastructure service owned by vCenter.

Physical Hosts
└─ ESX
   └─ vCenter Server (control plane)
      └─ Supervisor Cluster (Kubernetes control plane VMs)
         ├─ Supervisor Namespaces
         ├─ vSphere Pods
         └─ Guest / Workload Clusters

The vSphere Supervisor is a Kubernetes control plane deployed as system-managed VMs on ESX, whose lifecycle is owned entirely by vCenter Server and whose purpose is to expose Kubernetes semantics directly on vSphere infrastructure.

The vSphere Supervisor physically lives as a vCenter-managed Kubernetes control-plane VMs running on ESX hosts inside a vSphere Cluster.

vSphere Supervisor is the foundation of VCF consumption and can be enabled during workload domain creation or directly in vCenter.

It transforms vSphere Clusters into modern application platforms by embedding a platform-level Kubernetes control plane directly into ESX hosts, exposing a unified declarative API surface.

This integration allows end users to provision and manage workloads, including VMs, containers via vSphere Pods, full Kubernetes clusters through VKS, OCI image registries, and data services through a consistent API interface.

vSphere Supervisor maintains enterprise-grade Governance, security, and operational consistency through vCenter, VCF Operations, and VCF Automation.

vSphere Supervisor's namespaces provide the foundation for multi-tenancy, enabling GitOps-based workflows, and delivering cloud-like self-service in private data centers.

Why Does Supervisor Exist

Kubernetes didn’t replace VMs.
It replaced application scheduling semantics.

VMware’s bet was:

"Instead of forcing Kubernetes to run on top of vSphere,
we will make Kubernetes a native execution mode of vSphere."

Supervisor is that bet made concrete.

vSphere = substrate
Supervisor = Kubernetes-native personality of vSphere
VCF Automation = enterprise automation layer on top

Supervisor <140 Chars

vSphere Supervisor is vSphere running in a special Kubernetes-enabled mode.

VCF Automation consumes what vSphere Supervisor exposes.

  • It is vSphere running in special Kubernetes-enabled mode.
  • VCF Automation consumes what Supervisor exposes.

You can think of Supervisor as vSphere with Kubernetes fused into its control plane.

When you enable Supervisor on a vSphere Cluster, you are activating a capability inside vSphere.

  • Supervisor runs inside ESX.
  • Uses vCenter as the control plane.
  • Creates Supervisor Namespaces.
  • Exposes Kubernetes APIs, backed by native vSphere primitives.

So conceptually, Supervisor is vSphere operating in "Kubernetes-native mode".

Supervisor Namespaces are not Kubernetes namespaces.

Run Anything

If classic vSphere is "run VMs", vSphere Supervisor is "Run anything (VMs, Pods, services)" with the same primitives.

Where Does Supervisor Run

  • Supervisor runs on ESX.
  • It enables Kubernetes semantics backed by vSphere primitives.
  • VCF Automation is optional and sits above Supervisor.

Supervisor Namespaces look like Kubernetes namespace on purpose.

vSphere (authoritative substrate)
└── Supervisor
    ├── Kubernetes API (authoritative for some things)
    └── vSphere API/UI (authoritative for others)

Things Can Be Declaratively Done via k8s

Kubernetes manifest → vSphere policy enforcement.

Workload Lifecycle (VMs and Pods)

  • Create VMs via VirtualMachine CRDs (VM Operator)
  • Create Pods / Deployments
  • Attach storage
  • Attach networks (within guardrails)

Storage Consumption

  • PVCs backed by vSphere storage policies
  • Datastore selection via StorageClasses
  • Policy-based placement

Secret Consumption

  • Declare “I need this secret”
  • Injector materializes it
  • VCF Secret Service enforces access

Network Attachment (within limits)

  • Pods/VMs attach to NSX-backed segments
  • Load balancers exposed via K8s Service abstractions
  • You don’t manage NSX topology in YAML, but you consume it declaratively.

What You Cannot Do in K8s Declaratively

  • Infrastructure topology
  • Tenancy and Governance constructs (Organization, Projects, Quoatas at Provider level, Approval, Chargeback)
    • Those live in VCF Automation / vSphere UI, not Kubernetes.

So: vSphere Supervisor exposes a Kubernetes-native API surface for workload consumption (pods, VMs, storage, networking), while infrastructure topology, tenancy, and governance remain vSphere-and VCF Automation-controlled.

Supervisor Inventory Model

A vSphere Supervisor extends the vCenter inventory by introducing additional objects that represent Kubernetes-based workloads and their supporting infrastructure.
Within the vCenter inventory, a Supervisor exposes and manages the following objects:

1. vSphere Namespaces

vSphere Namespaces represent logical application environments within the Supervisor. They define where workloads can run and what resources they can consume.

Each namespace:

  • Represents an isolated execution boundary
  • Is backed by a dedicated resource pool
  • Is governed by quotas and policies
  • Is associated with specific users and permissions

2. Resource Pools

For every vSphere Namespace, the Supervisor creates a corresponding resource pool.

  • In a single-cluster Supervisor, one Resource Pool is created per namespace.
  • In a multi-zone (multi-cluster) Supervisor:
    • A Resource Pool is created in each participating cluster.
    • Resource consumption is distributed across all clusters in the zone.

These Resource Pools enforce CPU and memory limits defined at the namespace level.

3. Workloads Within a Namespace

Each vSphere Namespace can contain multiple types of workloads:

  • vSphere Pods
    Lightweight VM-based pods managed by the Supervisor.
  • Supervisor Services
    Platform services deployed into the Supervisor environment.
  • VKS Clusters
    Kubernetes clusters provisioned and managed by VKS.
  • Virtual Machines
    Traditional VMs deployed directly into the namespace.

4. Networking and Storage Resources

Namespaces also encapsulate:

  • Network configuration and connectivity
  • Storage policies and persistent volumes
  • Access to shared or dedicated datastores

All networking and storage enforcement is handled by the underlying vSphere and NSX infrastructure.

5. Permissions and Access Control

Each namespace includes role-based access control assignments that determine:

  • Which users or groups can access the namespace
  • What actions they can perform (deploy, modify, delete)
  • Which resources they can consume

Permissions are enforced by vCenter and integrated identity providers.

Supervisor—VCF Automation Relationship

┌────────────────────────────┐
│   VCF Automation (ex-vRA)  │  ← lifecycle, blueprints, polic
├────────────────────────────┤
│   vSphere Supervisor       │  ← Kubernetes-native control plane
├────────────────────────────┤
│   vSphere (ESX + vCenter)  │  ← compute, storage, networking
├────────────────────────────┤
│   Hardware                 │
└────────────────────────────┘
  • Supervisor:
    • Exposes Kubernetes + VM APIs
    • Owns runtime enforcement
  • VCF Automation:
    • Consumes Supervisor APIs
    • Provides self-service, governance, orchestration

Supervisor enables capabilities
VCF Automation operationalizes them
