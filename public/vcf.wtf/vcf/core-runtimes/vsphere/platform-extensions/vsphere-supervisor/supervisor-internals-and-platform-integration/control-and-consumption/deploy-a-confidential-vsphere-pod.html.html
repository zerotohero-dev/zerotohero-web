Deploy a Confidential vSphere Pod

#kubernetes #security

You can run confidential vSphere Pods on a vSphere Supervisor cluster. These pods use hardware-backed memory encryption to protect guest memory from being accessed by the hypervisor or other privileged components.

This capability is powered by AMD Secure Encrypted Virtualization–Encrypted State (SEV-ES).

Overview

A confidential vSphere Pod is a Kubernetes Pod that runs inside a virtual machine configured with SEV-ES: This ensures that:

  • Memory contents are encrypted
  • CPU register state is protected
  • The hypervisor cannot read or tamper with guest memory

This is especially useful for workloads that require strong isolation guarantees, such as handling sensitive data or running regulated workloads.

Prerequisites for Using SEV-ES

To enable confidential vSphere Pods, the following requirements must be met:

1. Supported Hardware

  • The ESX host must run on hardware that supports AMD SEV-ES.

2. ESX Version

  • ESX 7.0 Update 2 or later is required.

3. BIOS Configuration

SEV-ES must be enabled in the system BIOS.

When enabling SEV-ES, configure the following:

  • Minimum SEV non-ES ASID count
    This value must be set to:

    number of SEV-ES VMs + number of confidential vSphere Pods + 1

    Example:

    • 100 SEV-ES VMs
    • 128 confidential vSphere Pods
      → Minimum value = 229

    The maximum allowed value is typically 500, depending on hardware.

Creating a Confidential vSphere Pod

To deploy a confidential pod, you enable SEV-ES at the pod specification level.

1. Define the Pod Manifest

Create a YAML manifest with the following key elements:

  • Enable confidential pod support via annotation
  • Explicitly set memory requests and limits (they must match)
apiVersion: v1
kind: Pod
metadata:
  name: photon-pod
  namespace: my-podvm-ns
  annotations:
    vmware/confidential-pod: enabled
spec:
  restartPolicy: Never
  containers:
  - name: photon
    image: wcp-docker-ci.artifactory.eng.vmware.com/vmware/photon:1.0
    command: ["/bin/sh"]
    args: ["-c", "while true; do echo hello, world!; sleep 1; done"]
    resources:
      requests:
        memory: "512Mi"
      limits:
        memory: "512Mi"

Memory requests and limits must be identical for confidential pods.

2. Deploy the Pod

Apply the manifest:

kubectl apply -f <yaml-file>.yaml

Once submitted:

  • The Scheduler selects an ESX host that supports SEV-ES
  • If no compatible host is available, the Pod remains unscheduled
  • If scheduled successfully, the Pod runs with encrypted memory

3. Verify the Pod

You can verify the pod deployment using:

kubectl describe pod <pod-name>

From the vSphere side, administrators can see the pod listed with:

Encryption Mode: Confidential Compute

Summary

  • Confidential vSphere Pods use AMD SEV-ES to protect memory contents.
  • Encryption is enforced at the hardware level and is transparent to workloads.
  • The feature requires:
    • Compatible hardware
    • ESX 7.0 U2 or later
    • BIOS configuration for SEV-ES
  • Configuration is done declaratively using Kubernetes YAML.
  • The Supervisor ensures scheduling only onto capable hosts.
