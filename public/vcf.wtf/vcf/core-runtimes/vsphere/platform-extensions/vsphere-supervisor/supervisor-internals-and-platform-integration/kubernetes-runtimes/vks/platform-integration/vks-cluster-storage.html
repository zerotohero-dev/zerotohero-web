VKS Cluster Storage

VKS clusters, like other workloads running in a Supervisor Namespace, require persistent storage.

Persistent storage for VKS clusters is provided through integration with vSphere storage services, using storage policies and CSI-based provisioning.

Storage Policies for VKS Clusters

To provide persistent storage to VKS clusters, a vSphere administrator defines storage policies that describe required storage characteristics (such as performance, availability, and placement rules).

These storage policies are then:

  1. Assigned to a vSphere Namespace
  2. Exposed to VKS clusters deployed within that namespace

The storage policies determine:

  • Which datastores can be used
  • How volumes are provisioned
  • How placement and availability rules are enforced

Storage Class Creation

Based on the assigned storage policies:

  • The vSphere Supervisor automatically creates corresponding Kubernetes StorageClasses
  • These StorageClasses are made available inside the VKS cluster

Each storage policy results in two StorageClass variants:

  • One with Immediate volume binding
  • One with WaitForFirstConsumer volume binding

This allows DevOps teams to choose the appropriate provisioning behavior based on workload requirements.

How VKS Clusters Integrate with vSphere Storage

VKS clusters integrate with vSphere storage using Paravirtual CSI (pvCSI).

Paravirtual CSI (pvCSI)

  • pvCSI is a modified version of the standard vSphere CNS-CSI driver
  • It runs inside the VKS cluster
  • It handles all storage-related operations initiated from the cluster

The pvCSI driver does not communicate directly with vCenter or CNS. Instead:

  1. The pvCSI driver forwards storage requests to the Supervisor
  2. The Supervisor routes the request to CNS (Cloud Native Storage)
  3. CNS performs the actual storage operation in vSphere

Unlike CNS-CSI, pvCSI:

  • Does not require vCenter credentials
  • Operates using a service account within the namespace
  • Relies on the Supervisor for secure mediation

Persistent Volume Lifecycle in VKS

The following sequence describes how a persistent volume is created in a VKS cluster:

  1. A DevOps engineer creates a PersistentVolumeClaim (PVC) in the VKS cluster.
  2. The request is received by the pvCSI driver.
  3. pvCSI forwards the request to the Supervisor.
  4. The Supervisor invokes the CNS API in vCenter to create the volume.
  5. Once the volume is created, the information propagates back through the Supervisor.
  6. The PVC and corresponding PV appear as Bound in both:
    • The VKS cluster
    • The Supervisor view

This ensures consistent visibility and lifecycle tracking across all layers.

Summary

  • VKS clusters rely on vSphere storage policies for persistent storage.
  • Storage policies determine where and how volumes are provisioned.
  • The Supervisor mediates all storage operations between VKS and vSphere.
  • pvCSI enables secure, credential-free storage operations from inside the cluster.
  • Storage resources appear consistently in both the VKS cluster and Supervisor views.
