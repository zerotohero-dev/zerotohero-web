Supervisor Namespace

A 

vSphere Namespace

 sets the resource boundaries where 

vSphere Pods

, VMs, and 

VKS

 clusters, and 

Supervisor Services

 can run. As a vSphere administrator, you create and configure 

vSphere Namespaces

 through the vSphere Client.

When initially created, a 

vSphere Namespace

 has unlimited resources within the 

Supervisor

. As a vSphere administrator, you can set limits for CPU, memory, storage, as well as the number of Kubernetes objects that can run within the 

vSphere Namespace

. Storage limitations are represented as storage quotas in Kubernetes. A resource pool is created in vSphere per each 

vSphere Namespace

 on the 

Supervisor

You can map up to three 

vSphere Zones

 to a 

vSphere Namespace

 for running workloads on them. The 

vSphere Namespace

 spreads across all vSphere clusters that are part of the 

vSphere Zones

. The resources utilized to a 

vSphere Namespace

 are taken from all the underlying vSphere clusters on equal parts. For example, if you dedicate 300 MHz of CPU, 100 MHz are taken from each vSphere cluster.

To provide workload isolation, you should use 

vSphere Zones

 marked as management only for running the 

Supervisor

 conrol plane on these zones.

Differences Between a 

vSphere Namespace

 and Kubernetes Namespace

Although in its core a 

vSphere Namespace

 serves the same function as a Kubernetes namespace, a 

vSphere Namespace

 is specific to the 

vSphere Supervisor

. You should not confuse a 

vSphere Namespace

 with a Kubernetes namespace.

A 

vSphere Namespace

 is implemented as an extension to a vSphere resource pool and its function is to provide resources to workloads running in the 

Supervisor

. A 

vSphere Namespace

 has a direct mapping to a Kubernetes namespace through which object and storage quotas are enforced upon workloads.

Another difference with a regular Kubernetes namespace is that the vSphere administrator manages the user access to 

vSphere Namespaces

, as mentioned from above. The vSphere administrator can also associate VM classes and Content Libraries containing VM templates that DevOps engineers can use to self-service VMs.

These are not Kubernetes namespaces in the usual sense.
They are:

  • Resource-governed containers
  • Backed by:
    • vSphere Networking (NSX)
    • vSphere Storage (vSAN)
    • vSphere RBAC

A tenant can have more than one Supervisor Namespace; however, those namespaces are not individual tenants.

Tenancy is a business/trust concept. Supervisor Namespaces are execution slices that implement tenancy.

vCenter / API / UI
  ↓
Supervisor Namespace created
  ↓
Namespace appears in Kubernetes
  ↓
kubectl can now interact with it

^ This is the opposite of upstream Kubernetes.

In a vSphere Supervisor cluster, tenant namespaces are created and owned by vSphere as Supervisor Namespaces. Kubernetes users cannot create namespaces directly; kubectl is used only to interact with namespaces after vSphere has provisioned them.

  • Supervisor Namespaces are created by vSphere
  • kubectl create ns is not how tenancy is established. This will be explicitly denies on the Supervisor Cluster.
  • There is no concept of a regular tenant namespace on Supervisor
  • Kubernetes reflects what vSphere creates
  • Namespace lifecycle is deliberately centralized for safety
Capability Normal K8s Supervisor
Namespace = scheduling boundary ❌ ✅
Namespace creation allocates capacity ❌ ✅
Cluster-admin can drain cluster ✅ ❌
K8s is resource authority ✅ ❌
vSphere enforces hard limits ❌ ✅

In a Supervisor cluster, Kubernetes cluster-admin privileges do not grant authority over physical resource allocation. Namespace creation and resource envelopes are owned by vSphere, and Kubernetes operates strictly within those bounds.

A Workload is a Workload

From vSphere's perspective, a Pod, VM, or VM-based Pod is just scheduled compute.

Projects are Automation Constructs

Project is an automation construct.

  • Defined by Automation Admin persona.
  • Mapped to one or more Supervisor Namespaces.
  • Invisible to vSphere / Kubernetes.
  • Their purpose is to govern requests, not execution.

From the Guest Cluster’s perspective, Organizations and Projects do not exist; similarly there is no concept of an organization or project inside the Supervisor Cluster either.

Comparing to Kubernetes Namespaces

Supervisor namespaces intentionally mimic Kubernetes semantics:

Behavior Works like K8s?
kubectl get ns ✅
RBAC via kubeconfig ✅
Deploy Pods ✅
CRDs, controllers ✅
ServiceAccounts ✅

From the developer's laptop, they feel like ordinary Kubernetes namespaces.

Here is where they differ though (mostly around ownership and LCM)

Aspect Kubernetes namespace Supervisor namespace
Created by kubectl create ns vSphere / vCenter
Deleted by Kubernetes vSphere
Backed by etcd vSphere inventory objects

Resource Governance

in Supervisor namespaces, resource governance is outside Kubernetes.

In Kubernetes:

  • CPU/memory limits = ResourceQuota
  • Storage = StorageClass + PVC
  • Network = CNI rules

In Supervisor:

  • CPU/memory = vSphere resource pools
  • Storage = datastore policies
  • Network = NSX segments
  • Limits are hard boundaries, not advisory

So when you say:

"This namespace gets 8 vCPU and 64 GB RAM"

That's not a Kubernetes promise---that's vSphere enforcing physics.

Security Model

Security model is in vSphere, not in Kubernetes

Layer Authority
Identity vSphere SSO / LDAP
AuthZ vCenter roles
K8s RBAC Derived / scoped
Admission vSphere + K8s

Kubernetes RBAC applies inside the box vSphere already drew.

That’s why certain Kubernetes admin actions are silently blocked even when RBAC says “allowed”.

Network is not Just CNI

From Kubernetes’ POV:

  • It looks like a CNI plugin

In reality:

  • NSX owns:
    • L2/L3 segments
    • Firewall rules
    • Load balancers

Kubernetes asks politely. NSX decides.

This matters for:

  • NetworkPolicy
  • Ingress behavior
  • Multi-tenant isolation

Storage Semantics are Policy-First, not PVC-first

In vanilla K8s:

  • PVC requests storage
  • Cluster tries to satisfy it

In Supervisor:

  • PVC must map to pre-approved storage policies
  • Datastore capabilities are enforced externally

StorageClass ≈ view of vSphere policy, not a free-form construct.

What Is the Overall Goal

The goal of this abstraction is to:

  • Let Kubernetes users feel at home
  • Let infrastructure teams retain determinism
  • Avoid “Kubernetes ate my datacenter” chaos

So Supervisor namespaces are:

Kubernetes-compatible interfaces to vSphere-owned sandboxes

Summary

  • Supervisor namespaces intentionally impersonate Kubernetes namespaces
  • Kubernetes sees them as native; operations are externally enforced
  • vSphere owns:
    • lifecycle
    • resource limits
    • security boundaries
  • Kubernetes operates within those constraints

Uniqueness

Supervisor Namespace names are unique only within a Supervisor cluster, and every Supervisor Namespace is governed by exactly one Project; Organizations do not directly own namespaces.

Supervisor Namespace names are not globally unique, nor organization-wide unique.

They are unique per Supervisor Cluster.

Supervisor Namespaces are always associated with a Project.

Organization
└── Project
    └── Supervisor Namespace(s)

Uniqueness

  • Namespace names are:
    • ❌ not global
    • ❌ not org-unique
    • ❌ not project-unique
    • ✅ unique per Supervisor cluster

Ownership

  • Namespace:
    • belongs to exactly one Project
    • indirectly belongs to one Organization
    • never floats independently

Here’s a concrete example:

Org: Verizon
Project: West-Coast
  - Supervisor Cluster: sjc-01
    - Namespace: edge-prod
  - Supervisor Cluster: lax-01
    - Namespace: edge-prod   ← same name, totally fine

Project: East-Coast
  - Supervisor Cluster: iad-01
    - Namespace: edge-prod   ← also fine
  • Namespace names are Supervisor Cluster-scoped
  • Projects are mandatory governance owners
  • Organizations are containers of projects, not namespaces
  • Any apparent contradiction is UI simplification, not platform behavior
