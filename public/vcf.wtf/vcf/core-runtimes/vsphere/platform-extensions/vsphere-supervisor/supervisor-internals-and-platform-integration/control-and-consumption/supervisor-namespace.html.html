Supervisor Namespace

#tenancy #kubernetes

A Supervisor Namespace (vSphere Namespace) defines the resource and security boundary within which workloads—such as vSphere Pods, virtual machines, and VKS clusters—run on a vSphere Supervisor.

A namespace is the fundamental construct that binds together:

  • Resource allocation
  • Access control
  • Workload isolation
  • Policy enforcement

All workloads deployed through the Supervisor operate within a Supervisor Namespace (vSphere Namespace).

Purpose of a Supervisor Namespace

A Supervisor Namespace:

  • Defines where workloads can run
  • Controls how much compute, memory, and storage they can consume
  • Establishes administrative and security boundaries
  • Acts as the bridge between vSphere infrastructure and Kubernetes workloads

Namespaces are created and managed by Organization Administrators, not by Kubernetes users.

Namespace Resource Management

When a Supervisor Namespace is created:

  • A corresponding Resource Pool is created in vSphere
  • CPU, memory, and storage limits can be defined
  • Quotas apply to all workloads within the namespace
  • Limits are enforced at the infrastructure layer

If the vSphere Supervisor spans multiple vSphere clusters (zones), the namespace’s resource allocation is distributed across all participating clusters.

Example:
If a namespace is assigned 300 MHz CPU across a three-zone Supervisor:

  • Each cluster contributes 100 MHz
  • Scheduling is balanced across zones

Namespace and Access Control

Access to a Supervisor Namespace is controlled through:

  • vCenter Single Sign-On (SSO)
  • External identity providers (OIDC)

Administrators assign permissions to users or groups, defining what operations they can perform inside the namespace.

This access model governs:

  • Who can deploy workloads
  • Who can view or modify resources
  • Who can manage storage and policies

What Runs Inside a Supervisor Namespace

A Supervisor Namespace can host:

  • vSphere Pods (VM-backed Kubernetes pods)
  • Virtual Machines
  • Supervisor Services
  • VKS cluster components

All workloads deployed within a namespace inherit its resource constraints and access rules.

Relationship to Kubernetes Namespaces

A Supervisor Namespace is not the same as a Kubernetes namespace, even though it appears similar at first glance.

Aspect Kubernetes Namespace Supervisor Namespace
Created by kubectl Organization Administrator
Purpose Logical separation Resource and Governance boundary
Ownership Kubernetes vSphere
Backed by etcd vSphere Resource Pools
Enforced by Kubernetes vSphere + NSX
Visibility Cluster-local Supervisor-wide

A Supervisor Namespace contains Kubernetes namespaces, but is not itself one.

Namespace Lifecycle and Control Plane

Creation Flow

  1. Organization Administrator creates a namespace in the Supervisor.
  2. vSphere allocates backing resources and policies.
  3. Namespace becomes visible to Kubernetes users.
  4. Workloads can be deployed using standard Kubernetes tooling.

Important Constraints

  • Kubernetes users cannot create or delete vSphere namespaces.
  • Namespace lifecycle is owned by vSphere Supervisor, not Kubernetes.
  • Namespace limits cannot be overridden from inside the cluster.

Namespace Uniqueness and Scope

  • Namespace names are unique per Supervisor Cluster, not globally.
  • The same namespace name may exist on multiple Supervisors.
  • A namespace belongs to exactly one Project.
  • Projects belong to a single Organization.

Example:

Organization: AcmeCorp
├── Project: Retail
│   ├── Namespace: prod
│   └── Namespace: stage
└── Project: Logistics
    └── Namespace: prod

Here, prod appears multiple times but in different Supervisor contexts.

Projects and Governance

A Project is a governance construct used to group and manage namespaces.

  • Projects define ownership and intent
  • They map to one or more Supervisor Namespaces
  • They are not visible inside Kubernetes
  • They exist only in the VCF Automation control plane

Projects allow administrators to:

  • Apply policies
  • Enforce quotas
  • Track ownership
  • Control access

Supervisor Namespaces are Kubernetes-compatible, but not Kubernetes-native.

They look and behave like Kubernetes namespaces to workloads, but their lifecycle, limits, and permissions are governed entirely by vSphere.

Summary

  • Supervisor Namespaces are the execution boundary for workloads.
  • They are created and managed by vSphere Supervisor, not Kubernetes.
  • They enforce compute, storage, and security limits.
  • Kubernetes operates inside these boundaries.
  • Projects and Organizations exist above namespaces to provide Governance.
  • Supervisor Namespaces are unique per Supervisor Cluster and not globally unique.
