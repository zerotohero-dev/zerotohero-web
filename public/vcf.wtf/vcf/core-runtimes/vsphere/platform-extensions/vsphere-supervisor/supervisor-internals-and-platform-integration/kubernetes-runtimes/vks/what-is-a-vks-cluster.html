What is a VKS Cluster

A VKS cluster is a Kubernetes cluster that is built and supported by VMware and that runs application workloads.

VKS clusters are part of a vSphere Kubernetes Service (VKS) deployment. You use VKS cluster management to manage VKS clusters. Their lifecycle, however, is handled by the native vSphere Supervisor APIs.

A VKS cluster is a Kubernetes cluster provisioned and managed through VKS.

Characteristics:

  • Built on vSphere
  • Managed via the vSphere Supervisor
  • Created and governed through VCF Automation
  • Runs standard Kubernetes workloads

VKS clusters are where actual workloads execute, but they do not contain awareness of organizational hierarchy (i.e. they don’t know about Organizations, Projects, or Supervisor Namespaces).

Relationship Between VKS and VCF Automation

VKS provides self-service lifecycle management of Kubernetes workload clusters.

VKS is an implementation of the open-source Cluster API project, which defines a set of custom resources and controllers used to manage the lifecycle of Kubernetes clusters.

VKS is a component of the vSphere Supervisor. The Supervisor acts as a source of compute resources that can be registered with VCF Automation.

System administrators allocate Supervisor resources to organizations in VCF Automation. The Supervisor itself does not belong to any organization.

Project Administrators request Supervisor Namespaces for their projects, and VCF Automation allocates the appropriate namespaces to match those requests.

A single vSphere Supervisor can provide resources to multiple VCF Automation organizations.

Within a single organization, Supervisor Namespaces can be moved between projects.

Agents and Extensions Installed on the VKS Cluster

To function properly, VKS cluster management relies on a set of agents and extensions that run in every VKS cluster.

When a VKS cluster is attached to VKS Cluster Management, the vSphere Supervisor auto-attach service installs:

  • Required extensions
  • Cluster resource definitions
  • Agent components on each node

The agent configuration includes:

  • The VKS cluster management endpoint
  • The container image registry used by the cluster
  • Proxy configuration details

Cluster Health

The overall health of a cluster is determined by aggregating the health of its nodes and components.

Cluster Health States

  • Healthy
    All nodes and components are healthy, and a heartbeat is received every minute.
  • Unhealthy
    One or more of the following are unhealthy:
    • One or more control plane nodes
    • One or more cluster components
  • Warning
    A cluster enters a warning state when:
    • One or more worker nodes are unhealthy or unknown, or
    • One or more nodes (control plane or worker) report a warning condition
  • Unknown
    The cluster is marked as unknown if:
    • One or more control plane nodes are in an unknown state, or
    • No heartbeat has been received for more than three minutes

Node Health

Node health is derived from the NodeReady condition and other node-level signals such as:

  • MemoryPressure
  • DiskPressure
  • OutOfDisk

The node’s health state is evaluated as follows:

Healthy

  • NodeReady is True
  • All other conditions are healthy

Unhealthy

  • NodeReady is False, or
  • NodeReady is True, but more than half of the other conditions are unhealthy

Warning

  • NodeReady is True
  • Fewer than half of the other conditions are unhealthy

Unknown

  • NodeReady is neither True nor False, or
  • No heartbeat has been received from the node for more than three minutes

Component Health

VKS cluster management monitors the health of the following control plane components:

  • kube-apiserver
  • scheduler
  • controller-manager
  • etcd components (etcd-0, etcd-1, etcd-2, etc.)

Each component periodically reports its health status.

Component Health States

  • Healthy
    The last reported health condition is True.
  • Unhealthy
    The last reported health condition is False.
  • Unknown
    The last reported health condition is neither True nor False, or no heartbeat has been received for more than three minutes.

Resource Hierarchy Overview

VKS (vSphere Kubernetes Service) provides managed Kubernetes clusters running on vSphere.
It operates within the VCF Automation framework and relies on the Supervisor to provision and manage resources.

VKS cluster management introduces a hierarchical model that separates:

  • Control-plane governance (VCF Automation)
  • Cluster lifecycle management (VKS)
  • Runtime execution (vSphere Supervisor + Kubernetes)

Resource Hierarchy in VKS Cluster Management

VKS cluster management organizes resources using the following hierarchy:

  1. Organization
  2. Project
  3. VKS Cluster
  4. Kubernetes Namespaces (inside the cluster, these are NOT Supervisor Namespaces)

This hierarchy exists in the VCF Automation control plane, not inside Kubernetes itself.

VKS Cluster Namespaces (Workload Namespaces)

Each VKS cluster contains standard Kubernetes namespaces:

  • These are normal Kubernetes namespaces
  • Managed using kubectl
  • Used for application isolation and RBAC within the cluster

These namespaces are not the same as Supervisor Namespaces or VCF Automation Projects.

Supervisor Namespaces (Supervisor Level)

A Supervisor Namespace exists at the Supervisor layer and is used to:

  • Allocate CPU, memory, and storage
  • Bind resource pools to consumers
  • Host VKS cluster resources

Important distinctions:

  • Supervisor Namespaces are not Kubernetes namespaces (*but they are queries using kubectl, so they —look like– Kubernetes namespaces)
  • They are managed through the Supervisor API
  • They serve as the execution boundary for VKS clusters

Conceptual Model Summary

Layer Responsibility Visible via kubectl
VCF Automation Org / Project / Policy / Quotas ❌
Supervisor Resource orchestration ✅ (Supervisor API)
VKS Cluster Workload execution ✅
Kubernetes Namespaces Application isolation ✅

Summary

VCF Automation defines intent.
VKS enforces that intent.
Kubernetes executes the result.

The organizational model exists above Kubernetes and never becomes part of the Kubernetes API surface.
