CNS

#kubernetes

CNS (Cloud Native Storage) is the vSphere capability that makes Kubernetes storage first-class and lifecycle-aware.
It bridges Kubernetes PersistentVolumes with vSphere storage (typically vSAN) so that Kubernetes, not admins, drives volume lifecycle, while vSphere retains visibility, Policy, and Governance.

VCF relies on CNS wherever Kubernetes (Supervisor or guest clusters) consumes vSphere storage.

CNS in the VCF Stack

VCF
├── vSphere
│   ├── vCenter Server
│   │   └── CNS (storage awareness & lifecycle)
│   ├── vSAN / VMFS / NFS
│   └── ESX
│
└── Kubernetes
    ├── Supervisor Cluster
    └── Guest Clusters (VKS)
        └── PVC → PV → vSphere Volume
  • Implemented by: vSphere
  • Integrated via: vSphere CSI Driver
  • Observed/managed in: vCenter Server

What Problem Does CNS Solve

Without CNS:

  • Kubernetes creates volumes
  • vSphere sees “random disks”
  • No linkage, no lifecycle alignment

With CNS:

  • Kubernetes objects ↔ vSphere volumes are linked
  • vSphere understands which PVC belongs to which pod
  • Storage policy, health, and cleanup are automatic

CNS makes Kubernetes storage visible, governable, and safe in vSphere.

What Does CNS Do

Core Responsibilities

  • Lifecycle Mapping
    • PVC create/delete ↔ vSphere volume create/delete
  • Topology Awareness
    • Knows which cluster, namespace, pod owns a volume
  • Policy Enforcement
    • Applies vSphere Storage Policies (SPBM)
  • Operational Visibility
    • Shows Kubernetes volumes in vCenter UI & APIs

How Does the Plumbing Work

  1. Kubernetes creates a PVC
  2. CSI driver requests a volume
  3. vSphere provisions storage (vSAN)
  4. CNS records:
    • Kubernetes metadata
    • Volume ownership
  5. On PVC delete → CNS ensures correct cleanup

Kubernetes stays declarative; vSphere stays authoritative.

CNS vs Traditional VM Storage

Aspect VM Disk CNS-backed Volume
Owner vSphere admin Kubernetes
Lifecycle Manual Declarative (PVC)
Visibility VM-centric Pod / namespace-aware
Cleanup Error-prone Automatic
Policy Optional Enforced

CNS in VCF Kubernetes Flavors

Supervisor Cluster

  • CNS is always on
  • StorageClasses map directly to vSphere policies

Guest Clusters (VKS)

  • CNS operates via CSI
  • Still centrally visible in vCenter

Same mechanism, different consumption layers.

What CNS is NOT

  • ❌ Not a storage array
  • ❌ Not a CSI driver itself
  • ❌ Not Kubernetes-only
  • ❌ Not a replacement for vSAN

CNS is the glue, not the storage.

If CSI is:

“The driver that talks to storage”

Then CNS is:

“The brain that understands Kubernetes intent inside vSphere”

Why CNS Matters in VCF

  • Enables policy-driven storage for containers
  • Prevents orphaned volumes
  • Makes Kubernetes operationally sane for infra teams
  • Required for supported VCF Kubernetes designs

Summary

  • CNS = Cloud Native Storage awareness in vSphere
  • Aligns Kubernetes volume lifecycle with vSphere
  • Essential for Supervisor & TKG clusters in VCF- Turns storage into a first-class Kubernetes citizen
