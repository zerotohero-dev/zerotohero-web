Distributed Virtual Switch

In VMware Cloud Foundation, a Distributed Virtual Switch (DVS) is the standard, centralized virtual networking fabric used across ESX hosts in a Workload Domain.

Distributed Virtual Switch is a cluster-wide virtual switch that provides consistent networking configuration across all ESX hosts.
It centralizes network policy (port groups, uplinks, security, QoS) instead of configuring each host individually.

VCF standardizes on VDS as the baseline virtual networking fabric.

In VCF, a Distributed Virtual Switch is the mandatory, centrally managed L2 networking fabric that ensures every ESX host behaves identically so that VCF Automation, lifecycle management, and NSX can work reliably at scale.

DVS provides consistent networking configuration, policy enforcement, and operational control at scale—something VCF depends on to behave predictably.

DVS can be seen as:

One logical network switch that spans many ESX hosts,
configured once, and applied everywhere.

Instead of configuring networking host-by-host, you configure it centrally, and every participating ESX host follows that definition.

How DVS fits into the VCF stack

VCF is opinionated and automated. Networking must be:

  • Deterministic
  • Centrally managed
  • Compatible with lifecycle automation
  • Consistent across hosts and clusters

That’s why VCF mandates DVS (not standard vSwitches) for system and workload networking.

ESX hosts do not own the switch config — vCenter does.

VMware Cloud Foundation
└── vSphere
    └── vCenter Server
        └── vSphere Distributed Switch (VDS)
            ├── ESX Host A
            ├── ESX Host B
            └── ESX Host C
                └── Port Groups → VM vNICs

What Does a DVS Do

Core Responsibilities

  • Centralized L2 switching
  • Consistent port group configuration
  • NIC teaming & uplinks
  • Traffic shaping, security, VLAN tagging
  • Host lifecycle safety (no per-host drift)

“One virtual switch, shared by the whole cluster.”

Port Groups on a DVS

A Distributed Port Group defines:

  • VLAN ID(s)
  • Security policies (promiscuous mode, MAC changes)
  • Traffic shaping
  • Teaming / failover

VM vNICs attach to port groups, not directly to the switch.

What DVS is NOT

  • ❌ Not a router (no L3)
  • ❌ Not an NSX Segment
  • ❌ Not a tenant boundary
  • ❌ Not optional in VCF

Why VCF “Specifically” needs DVS

VCF automation (bring-up, scaling, patching, upgrades) assumes:

Requirement Why DVS matters
Host consistency All hosts must expose identical port groups and policies
Zero-touch expansion New hosts inherit networking immediately
Predictable upgrades NSX / ESX upgrades assume known switch topology
Advanced features LACP, NIOC, NetFlow, PVLANs, traffic shaping

A standard vSwitch breaks all of this because it is host-local and imperative.

In VCF:

  • Management Domain has a predefined VDS
  • Workload Domains get their own VDS
  • NSX transport zones bind to VDS uplinks
  • Lifecycle (patching, expansion) assumes VDS presence

You rarely “design” VDS in VCF—it is opinionated and enforced.

What Runs on a DVS in VCF

Typical VCF DVS port groups include:

System Traffic

  • Management
  • vMotion
  • vSAN
  • NSX Host TEPs

Workload Traffic

  • VM networks (VLAN-backed)
  • Supervisor / Kubernetes traffic (when applicable)

VCF may create multiple DVS instances, for example:

  • One for system domain
  • One per workload domain

DVS vs Standard vSwitch

Feature Standard vSwitch Distributed Virtual Switch
Scope Single ESX host Entire cluster / domain
Managed by ESX vCenter
Automation-ready ❌ ✅
LACP ❌ ✅
Network I/O Control ❌ ✅
Required by VCF ❌ ✅

Relationship to NSX

  • DVS ≠ NSX
  • DVS provides L2 switching
  • NSX builds overlay networking (Geneve tunnels, logical segments) on top of DVS

You can have:

  • DVS without NSX (VLAN-backed networking)
  • However, NSX requires DVS underneath

So DVS is the foundational substrate, not the SDN layer.

Key Distinction

  • DVS = underlay L2 switch
  • NSX = overlay (L3) networking

How Do They Work Together

  • NSX creates overlay segments
  • Overlay traffic is encapsulated (GENEVE)
  • Encapsulated packets ride over VDS uplinks

NSX does not replace VDS—it depends on it.

If DVS is:

“A shared Ethernet switch for ESX hosts”

Then NSX Segment is:

“A logical network built on top of that switch.”

Different layers, different jobs.

Operational Implications

  • DVS is a control-plane object
    • ESX caches its config locally, but authority lives in vCenter
  • If vCenter is temporarily unavailable
    • Traffic keeps flowing
    • You just can’t change switch config
  • VCF LCM expects DVS invariants
    • Port groups, uplinks, MTU, and teaming must match what SDDC Manager expects

Summary

  • DVS is not optional in VCF
    • VDS is the foundational virtual switch in VCF
  • It is owned by vCenter, not hosts
  • It enables automation, scale, and consistency
  • Provides centralized, cluster-wide L2 networking
  • NSX depends on it, but it is not NSX itself
    • NSX overlays run on top of VDS, not instead of it
  • Standard vSwitches are fundamentally incompatible with VCF’s design goals
