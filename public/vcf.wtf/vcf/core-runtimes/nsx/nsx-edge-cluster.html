NSX Edge Cluster

NSX Edge Cluster — and did vSphere ever have an “edge” concept?

NSX Edge Cluster in VMware Cloud Foundation

Short definition

An NSX Edge Cluster is a pool of dedicated NSX Edge Nodes that handle north–south traffic and advanced network services (routing, NAT, load balancing, VPN, firewall at the perimeter).

It is part of VMware NSX, not vSphere.

What an NSX Edge Cluster actually does

Primary responsibilities

  • North–south routing

    • Tier-0 Gateways (uplink to physical network)

    • Tier-1 Gateways (tenant / app routing)

  • Perimeter services

    • NAT

    • L4–L7 Load Balancing

    • VPN (IPsec, L2VPN)

    • Gateway firewalling

  • High throughput datapath

    • Optimized for packet forwarding

    • Separate from ESXi host dataplane

What it does not do

  • ❌ It does not run workloads

  • ❌ It does not manage policy (that’s NSX Manager)

  • ❌ It does not replace distributed firewall (DFW)

Where NSX Edge fits in the stack

VCF
├── vSphere Cluster (compute)
│   └── Workload VMs
│       └── East–West traffic (DFW on ESXi)
│
├── NSX Edge Cluster
│   └── North–South traffic
│       ├── Tier-0 Gateway
│       └── Services (NAT, LB, VPN)
│
└── NSX Manager
    └── Policy & control plane

A clean rule of thumb:

ESXi hosts = east–west
NSX Edge nodes = north–south

High availability & scale

  • Edge nodes are deployed as VMs

  • Grouped into Edge Clusters

  • Services can be:

    • Active/Active (ECMP)

    • Active/Standby

  • Protected by vSphere HA

  • Scaled horizontally by adding Edge Nodes

Now the second question:

Did vSphere have an “edge” concept too?

Short answer

No — vSphere itself does not have an edge concept.

But the confusion is understandable.

Why this confusion exists (important nuance)

1. vSphere networking before NSX

Classic vSphere had:

  • vSwitch / vDS

  • Physical routers & firewalls outside vSphere

No virtual routing tier.
No virtual perimeter.
No “edge” construct.

2. The historical name: NSX Edge

Earlier NSX versions (NSX-V era) had a product literally called:

NSX Edge Services Gateway

People shortened this to “the edge”, and the name stuck.

But it was always NSX, never vSphere.

3. Modern terminology (NSX-T / current NSX)

Today:

  • NSX Edge Node

  • NSX Edge Cluster

  • Tier-0 / Tier-1 Gateways

Still not vSphere-native.

What vSphere does have that people mislabel as “edge”

Thing people call “edge” What it really is
ESXi at a remote site Just an ESXi host
Management Domain Control plane, not edge
Supervisor Cluster Kubernetes control plane
vSAN Edge / ROBO Deployment model, not network edge
Physical router next to ESXi Physical edge, not vSphere

There is no vSphere Edge Node equivalent.

Mental model (clean separation)

  • vSphere: compute scheduling & virtualization

  • NSX: software-defined networking

  • NSX Edge: virtual perimeter routers & services

  • VCF: opinionated lifecycle glue tying them together

If you remove NSX:

  • vSphere has no edge

  • Networking reverts to physical devices

Key takeaways

  • NSX Edge Cluster = NSX’s north–south dataplane

  • It handles routing, NAT, LB, VPN, perimeter firewalling

  • vSphere never had an edge concept

  • The term “edge” is historical NSX vocabulary, not vSphere

  • In VCF, Edge Clusters are foundational for routed, multi-tenant designs
