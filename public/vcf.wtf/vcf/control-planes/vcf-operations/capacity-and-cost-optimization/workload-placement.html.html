Workload Placement

#operations

Workload Placement in VCF is the coordinated outcome of automation intent, vSphere scheduling (DRS), networking availability (NSX), storage policy (CNS), and runtime mobility (vMotion).

vMotion is the mechanism, not the decision-maker.

Workload Placement Pipeline

Workload Placement is a pipeline, not a switch—shaken, but not stirred:

Intent → Eligibility → Initial Placement → Continuous Optimization → Recovery

Each stage is owned by a different layer of VCF.

1. Intent layer

Answers: “what am I allowed to run, and where?”

Owner

  • VCF Automation (formerly Aria Automation)

What Happens

  • The user requests a workload (VM, Kubernetes cluster, pod)
  • Constraints are applied:
    • Organization / Project
    • Quotas and approvals
    • Target workload domain / cluster
    • Storage policy
    • Network intent

There is no vMotion here: This is just pre-placement filtering.

2. Eligibility Filtering

Answers: “Which clusters can even accept this?”

Owners

  • vSphere
  • NSX
  • vSAN

Checks

  • Cluster compatibility
  • HA admission control
  • Resource Pool limits
  • Storage policy compatibility
  • NSX Segment & transport availability

Anything that fails here is never considered for placement.

3. Initial placement

This is where DRS takes the helm.

Owner

  • DRS

What Does DRS Do

  • Chooses the best host at power-on time
  • Considers:
    • CPU & memory headroom
    • NUMA alignment
    • Resource Pool shares
    • Affinity / anti-affinity rules

This happens before the VM ever runs.

This is placement — not vMotion.

4. Runtime Optimization

This is where vMotion appears.

Owners

  • DRS + vMotion

Mechanism

  • vMotion

Triggered When

  • Load imbalance
  • Hotspots
  • Policy drift
  • Host entering maintenance

vMotion is:

  • How VMs move
  • NOT *why they move

5️. Failure-Driven Placement

Owner

  • HA

What Happens

  • Host fails
  • HA re-places VMs on surviving hosts
  • DRS may later rebalance

This is reactive placement, NOT optimization.

Networking Constraints

NSX matters more than what’s seen on the surface.

Owner

  • NSX Manager / VCF Edge

Placement is invalid if:

  • Segment is not available on target host
  • Transport zone mismatch
  • Edge capacity insufficient (north–south)

Networking can veto otherwise valid placements.

Storage Constraints

Especially for Kubernetes (CNS)

Owners

  • vSAN
  • CNS (for K8s)

Placement requires:

  • Policy-compatible datastore
  • Accessibility from host
  • Topology alignment (for StatefulSets, etc.)

For Kubernetes:

  • PVC → StorageClass → Storage Policy
  • CNS enforces this during placement

The Pipeline in a Nutshell

Layer Role in placement
VCF Automation Intent & constraints
vCenter Execution authority
DRS Initial + ongoing placement decisions
vMotion Movement mechanism
HA Failure recovery placement
NSX Network feasibility
Storage / CNS Data locality & policy

So things are a tad bit more involved than “VCF places workloads using vMotion”.

VCF places workloads using DRS decisions; vMotion is only the transport.

Summary

  • Workload placement in VCF is layered and collaborative
  • DRS decides, vMotion moves
  • HA re-places after failure
  • NSX and storage can block or shape placement
  • VCF Automation governs who can ask for what
