Supervisor Administrator manages the platform that hosts tenants, while Organization Administrator manages the tenants themselves, and Project Administrator manages what runs inside those tenants.

The Three Different Control Planes in VCF

VCF separates responsibilities cleanly into three distinct administrative planes, each with its own roles:

1. Infrastructure / Supervisor Plane

→ “How the Kubernetes substrate itself is operated”

This is where the Supervisor Cluster lives.

2. Consumption / Tenancy Plane

→ “Who consumes the infrastructure and how”

This is where Organizations, Projects, and quotas live.

3. Workload Plane

→ “What runs on top of it”

This is where Kubernetes workloads, namespaces, and apps live.

Where Does Supervisor Administrator Fit

Supervisor Administrator

Scope: Supervisor Cluster (platform level)

Primary responsibility:

  • Operates and governs the Supervisor Cluster itself
  • Manages cluster-level capabilities that all tenants depend on

Typical responsibilities:

  • Configure and manage Supervisor Cluster lifecycle
  • Control Supervisor services (e.g., Kubernetes control plane, CSI, CNI, storage integrations)
  • Define which Organizations can exist
  • Enable or disable platform features available to Organizations
  • Maintain host, networking, and storage alignment required for Kubernetes workloads

Supervisor Administrator does not manage application workloads or tenant namespaces.

From the Supervisor Administrator’s point of view:

“I run the Kubernetes platform that everyone else builds on.”

Organization Administrator

Organization Administrator is the tenant-level authority.

Organization Administrator

Scope: Organization (tenant boundary)

Role type: Consumer / tenant administrator

Responsibilities:

  • Create and manage Projects
  • Assign users to projects and roles
  • Define quotas and usage limits
  • Control access within their organization
  • Govern how their organization consumes platform capacity

They do not manage the Supervisor Cluster itself.

From the Organization Administrator’s perspective:

“I manage my organization’s slice of the platform.”

Project Administrator

Project Administrator has workload-level control over the project they have access to.

Project Administrator

Scope: Single Project within an Organization

Responsibilities:

  • Manage Kubernetes workloads (clusters, namespaces, services)
  • Manage users within the project
  • Configure project-level policies and resources

They operate inside boundaries defined by the Organization Administrator and enforced by the Supervisor.

From Project Administrator’s point of view:

“I manage workloads, not the platform.”

Hierarchical Relationship Summary

Supervisor Administrator
│
├── Organization Administrator
│   ├── Project Administrator
│   │   └── Application / Workload Operators
│   └── Project Users
│
└── (Multiple Organizations)
  • Supervisor Admin ≠ Super Org Admin
  • They operate at different abstraction layers

Historically, roles like this were blurry or implicit (vSphere Admin, Cloud Admin, etc.).

VCF introduces clear separation of concerns:

Problem Before Now
Platform admins had too much tenant visibility Supervisor Admin scoped to infrastructure
Tenants needed more autonomy Organization + Project roles
Mixed responsibilities Explicit role boundaries

This aligns VCF with cloud provider models (for, i.e., AWS control plane vs account admins).

Role Scope Controls
Supervisor Administrator Platform Supervisor cluster, infrastructure services
Organization Administrator Tenant Org configuration, projects, access
Project Administrator Workload Namespaces, workloads, app lifecycle
