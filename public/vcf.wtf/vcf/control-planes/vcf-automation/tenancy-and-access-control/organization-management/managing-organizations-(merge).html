Managing Organizations (merge)

With 

VMware Cloud Foundation Automation

 (

VCF Automation

), you, as an organization administrator, and your application teams can view and consume infrastructure that the provider administrator made available to your organization. As an organization administrator, you create projects, manage users, and define resource limits for running workloads.

The underlying infrastructure is based on vSphere Supervisors, which serve as the foundational structure for resource allocation. Supervisor-based infrastructure supports modern application use cases for provisioning containers and VMs. Provider administrators allocate infrastructure resources to organizations using regional quotas, while organization administrators distribute that allocated infrastructure to different projects within the organization using namespaces.

Infrastructure Concepts

VCF Automation

 enables a private cloud experience by providing abstraction layers on vSphere infrastructure. Each abstraction serves a specific purpose.

Regions.

 A region in 

VCF Automation

 aggregates compute, storage, memory, and networking resources from one or more zones. Regions are defined and managed by the provider administrator.

Zones.

 A zone represents a vSphere cluster that provides compute, memory, and storage resources for the organization.

Region Quota.

 A region quota defines the compute and storage resources that are allocated to an organization for a given region. The organization can receive quotas from multiple regions. The provider administrator decides from which regions to give a quota to an organization. The region quota can be split across more than one vSphere zone in a region. A region quota includes the following:

  • CPU and memory utilization limits which represent the upper bound of quota in a region.

  • CPU and memory reservations which represent the guaranteed capacity that is allocated in a region.

  • VM classes which represent the T-shirt-sized classes of compute that are available to the organization.

  • Storage classes which represent the storage policies and the quota that is assigned in a region.

Resources for Application Teams

Resources allocated to the organization in the form of a region quota can be further subdivided and allocated to application teams.

Project.

 Projects are logical representations of users and user groups and the resource pools that those users are entitled to. The resource pools that projects use are called namespaces. Resources, like VMs and Kubernetes clusters, are provisioned on namespaces.

Namespace.

 A vSphere Namespace is a resource pool that comprises CPU, memory, and storage resources that are assigned to projects in

VCF Automation

. Namespaces derive their quota from the overall region quota of the organization and come with controls such CPU, memory, and storage usage. Each namespace is bound to the scope of the project that it is associated with. Each project can contain multiple namespaces. Because the region quota can be split across multiple vSphere zones, the resource pool of a namespace can also be split across multiple vSphere zones. As an organization administrator, you can configure which zones a namespace can derive its resource limits from.

Namespace Class.

 Namespace classes are templates for defining guardrails on parameters that are used to create a namespace, like CPU limits, storage limits, permissible VM and storage classes, and others. Namespace classes are a prerequisite for creating namespaces. In the namespace class, you can set default values for CPU and memory limits and reservations. If a namespace class has a zone defined, default values for namespace limits and reservations are used from the zone section. As an organization administrator, you create namespace classes that fit your project needs. You can create one or more namespaces using the default namespace classes that come preconfigured in your organization. You can also override those defaults in a given project to suit specific configurations.

Networking Concepts

VPC.

 Virtual private clouds (VPCs) are the foundational networking element used in 

VCF Automation

 to enable applications to communicate with each other and the outside world. A VPC provides an isolated domain of networking resources that can be associated with one or more namespaces. As the organization administrator, you create VPCs for your organization. In addition, organization administrators and advanced project users can use the Networking Service in the IaaS Console to create, update, and manage resources within a given VPC.

Connectivity Profile.

 A connectivity profile is a template for VPC configuration. You can provide one or more external and private Transit Gateway IP address blocks that are used by VPCs configured with a specific connectivity profile. When the provider administrator allocates the regional networking configuration to the organization, Ð° default connectivity profile is created. As an organization administrator, you can create additional connectivity profiles to use in your organization.

Consumption Concepts

Application teams can provision resources on a self-service basis in 

VCF Automation

 using the catalog or the IaaS Services console. The catalog contains blueprints that are curated and maintained by the organization administrator, while cloud services in the IaaS Services console can be accessed directly without blueprints. Both methods of provisioning resources require content libraries that host the images.

Blueprints and Catalog.

 Blueprints are infrastructure templates that define the resources and cloud services you want to create to run an application. Organization administrators create the blueprints and publish them as catalog items for end users. End users interact with form entries of catalog items to request resources. These forms can be customized to hide unnecessary fields or to create dynamic entries.

Self-service IaaS Console.

 The self-service IaaS Services Console in 

VCF Automation

 allows users to provision cloud services directly on namespaces from the 

VCF Automation

 user interface, the 

VCF Automation

 APIs, and CLIs without the need for blueprints and the catalog.

Content Library.

 Content libraries host VM images that application teams use when deploying resources. Organizations can either use content libraries that are provided by the provider administrator or create their own organization-specific content libraries.

Policies.

 Policies provide governance on resources like VMs and VKS clusters that are provisioned from cloud services. Policies can be applied to the whole organization or to individual projects. 

VCF Automation

 supports two types of policies:

  • Policies that are applied to resources provisioned from the catalog, such as Approval, Day 2, and Lease policies.

  • Policies that are applied to resources provisioned through the self-service IaaS console.
