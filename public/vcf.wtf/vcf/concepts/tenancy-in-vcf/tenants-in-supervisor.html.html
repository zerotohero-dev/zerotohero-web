Tenants in Supervisor

#tenancy #governance

See also: Multi-Tenancy

  • In vSphere Supervisor a tenant is namespace-bound.
  • There is no separate tenant object inside Kubernetes.
  • A Supervisor Namespace is the tenant boundary at the vSphere-Kubernetes interface.

If you assume:

  • Tenant = Project → you’ll design the wrong thing
  • Tenant = Kubernetes namespace → you’ll under-secure
  • Tenant = Supervisor Namespace → things line up

“tenant namespace” is an informal shorthand for supervisor namespace. Organizations and Projects are Governance overlays. They do not define the tenant.

Kubernetes namespaces inside Guest Clusters are secondary scopes.

In vSphere Supervisor each tenant is realized as one (or more) Supervisor Namespaces.
VCF Automation sits above, provides governance (Organizations, Projects, quotas) and self-service, but does not define tenancy by itself.

This is sometimes “informally” expressed as “tenant namespaces”. What is meant by it is a Supervisor Namespace.

Or more precisely: In a Supervisor Cluster tenant == supervisor namespace (there is a 1-1 relationship).

In vSphere Supervisor, tenancy is realized through Supervisor Namespaces, which serve as the execution and isolation boundary for tenants.

When people say:

  • “tenant namespace”
  • “project namespace”
  • “customer namespace”

They almost always mean:

A Supervisor Namespace allocated to that tenant.

Kubernetes does not have a separate construct for this: It is a vSphere concept, projected as a (Supervisor) namespace.

Boundaries

A Supervisor Namespace can contain:

  • Pods
  • VMs
  • Entire guest Kubernetes clusters
Tenant
└── Supervisor Namespace
    ├── VM workloads
    ├── Pod workloads
    └── Guest K8s Cluster
        └── many Kubernetes namespaces
            ^ these are distinct from the "Supervisor Namespace"
              they are regular 
              Kubernetes `Kind:Namespace` resources.
              (/api/v1/namespaces) 
              --
              i.e., a cluster-scoped Kubernetes API resource 
              of kind "Namespace", defined in the 
              core (v1) API group, 
              used to provide a logical scope 
              for namespaced resources.
              --
              i.e. "a good old Kubernetes namespace";
              NOT a Supervisor Namespace.
              --
              vSphere Supervisor does not see, or care
              about the namespaces in this guest cluster.
              vSphere Supervisor see the "Guest Cluster"
              as a resource; that's the end of the level
              of its granularity.
              To be more precise:
              The Supervisor is not aware of, 
              and does not manage, Kubernetes namespaces 
              inside the Guest Cluster.
              ---
              * These namespaces exist only inside the 
                Guest Cluster
              * They are not visible to the Supervisor
              * They are not related to Supervisor Namespaces 
                in any structural way
              * They are ordinary Kubernetes namespaces, 
                no different from what you’d find in 
                a kind/minikube/EKS/AKS cluster

Infrastructure

This is the vSphere Layer of tenancy.

VCF Environment
└── vSphere (vCenter)
    └── Supervisor Cluster
        └── Supervisor Namespace  ← TENANT BOUNDARY

At this layer:

  • Supervisor Namespace
    • Has:
      • Resource limits
      • Storage policies
      • Network segments
      • RBAC bindings
    • Is created and owned by vCenter
    • Maps to:
      • Resource Pools
      • NSX constructs
      • Datastore permissions

This is where hard multitenancy lives.

VCF Automation

This is logical tenancy, not execution tenancy.

It is the automation and consumption layer.

Organization
└── Project
    └── Deployment / Blueprint
        └── (Targets a Supervisor Namespace)
  • Organization
    • Business / administrative grouping
  • Project
    • Quota + governance container

These map downwards, but they don’t execute workloads themselves.

Concept Where it lives Enforces isolation?
Organization VCF Automation ❌
Project VCF Automation ❌
Supervisor Namespace vSphere ✅
Kubernetes Namespace (guest cluster) Kubernetes ✅ (within cluster)

Supervisor Namespace defines a tenant from a runtime and security perspective.

At VCF Automation, Organizations and Projects define:

  • Ownership
  • Billing
  • Governance
  • Access workflows
    But not execution boundaries.

Concreate Example With Real Names

VCF Environment (Rackspace)
└── Supervisor Cluster
    ├── ns-cocacola      ← Tenant: Coca-Cola
    ├── ns-pepsi         ← Tenant: Pepsi
    └── ns-pizzahut      ← Tenant: Pizza Hut

Each tenant is a supervisor namespace.

What Does Each Supervisor Namespace Contain

For Coca-Cola (example):

  • Fixed CPU / memory quota
  • Dedicated or isolated NSX segment
  • Approved storage policies
  • RBAC scoped to Coca-Cola admins
  • Ability to run:
    • Pods
    • VMs
    • Guest Kubernetes clusters

That namespace is the tenant boundary.

VCF Automation Layer

Layer VCF Automation on top of this:

Organization: Coca-Cola
└── Project: Coke-Prod
    └── Deployments → targets ns-cocacola

Organization: Pepsi
└── Project: Pepsi-Shared
    └── Deployments → targets ns-pepsi

Organizations and Projects map to Supervisor Namespaces—they DO NOT replace them.

Automation gives:

  • Self-service catalogs
  • Approval workflows
  • Policy as code
  • Lifecycle orchestration

But the tenant still “lives” at the Supervisor Namespace.

Admin Roles and Personas

Note that the company and product names used in this section are for illustrative purposes—to give realistic analogies—only.

Rackspace Platform Admin

Scope

  • vCenter
  • Supervisor Cluster
  • NSX
  • Storage policies

Responsibilities

  • Create Supervisor Namespaces
  • Set quotas
  • Attach networks
  • Bind tenant identities
  • Enforce platform guardrails

Power level

  • God-mode
  • Not visible to customers

Tenant Platform Admin (Coca Cola IT)

Scope

  • One Supervisor Namespace (e.g., ns-cocacola)
  • Kubernetes API within that namespace

Responsibilities

  • Manage workloads
  • Deploy guest clusters
  • Configure service
  • Delegate access internally

Cannot

  • See other tenants
  • Modify quotas
  • Escape namespace
    This is who you hand the kubeconfig to.

Tenant App Team / DevOps

Scope

  • Kubernetes RBAC inside tenant (supervisor) namespace
  • Possibly inside guest clusters

Responsibilities

  • Deploy apps
  • Manage services
  • Consume secrets

No infrastructure privileges

VCF Automation Admin

Scope

  • Automation policies
  • Catalogs
  • Projects
  • Approvals

This role:

  • Often held by Rackspace
  • Sometimes delegated partially to tenants

Trust Boundaries

  • Secrets do not cross Supervisor Namespaces
  • Identity trust roots often terminate there
  • Network policy enforcement happens there
  • Audit logs are scoped there

So when you say tenant == Supervisor Namespace,
then:

  • Security
  • Compliance
  • Billing
  • Automation

all align clearly.
