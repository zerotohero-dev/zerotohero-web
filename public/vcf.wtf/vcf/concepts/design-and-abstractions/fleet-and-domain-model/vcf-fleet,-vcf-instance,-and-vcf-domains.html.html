VCF Fleet, VCF Instance, and VCF Domains

#architecture

In VMware Cloud Foundation, these three terms describe different scopes of control, stacked hierarchically from largest (fleet) to smallest (domain).

If you only remember one thing:

Fleet → Instance → Domain
Management across clouds → One cloud → Parts of that cloud

VCF Fleet—“All my clouds”

A VCF Fleet is a logical grouping of multiple VCF Instances that are managed, observed, or governed together.

It is a meta-layer, not a runtime construct.

Scope

  • Multiple data centers
  • Multiple regions
  • Core + Edge
  • Possibly different purposes (prod, DR, sovereign, etc.)

What Happens at the Fleet Level

  • Fleet-wide visibility
  • Cross-instance posture (versions, compliance, drift)
  • Centralized operations and Governance
  • Consistent policies and standards

Fleet = “My private cloud estate”

VCF Instance—“One Cloud”

Scope

A VCF Instance is one deployed VCF system:

  • One SDDC Manager
  • One Management Domain
  • One lifecycle authority

This is the unit of deployment and lifecycle.

Characteristics

  • Independently installed
  • Independently upgraded
  • Independently failed / recovered
  • Self-contained control plane

Examples:

  • One on-prem private cloud
  • One regional cloud
  • One large edge aggregation site

VCF Domains—“Functional Partitions Inside a Cloud”

Scope

A Domain is a logical partition inside a VCF Instance, used to separate management concerns, workload types, or failure domains.

There are two major kinds:

Management Domain

  • Created first
  • Hosts:
    • SDDC Manager
    • Management vCenter
    • NSX Managers
  • Runs the platform itself

Workload Domains

  • One or more per instance
  • Host:
    • Tenant workloads
    • Application clusters (i.e., Guest Clusters)
  • Independently scalable and isolatable

Domain = “A major subsystem within a Cloud”

A Canonical Example

VCF Fleet
│
├── VCF Instance (DC-West)
│   ├── Management Domain
│   ├── Workload Domain – Prod
│   └── Workload Domain – Dev
│
├── VCF Instance (DC-East)
│   ├── Management Domain
│   └── Workload Domain – Prod
│
└── VCF Instance (Edge-Region-1)
    ├── Management Domain
    └── Workload Domain – Edge

Each level has different responsibilities and blast radius.

Responsibility and Lifecycle Boundaries

Concept Lifecycle Authority Failure Scope Typical Owner
Fleet Coordinated / Observational None (meta) Central IT / Cloud Ops
Instance SDDC Manager One cloud Platform Admin
Domain SDDC Manager (scoped) Subset of cloud Organization Administrator

Kubernetes Analogy

  • Fleet → Multiple clusters managed centrally
  • Instance → One cluster + control plane
  • Domain → Node pools / system vs workload separation

Why Did VMware Introduce This Terminology

Because customers now have:

  • Many private clouds
  • Many edge clouds
  • A need for consistent Governance at scale

So VMware had to separate:

  • Where lifecycle happens (instance)
  • Where workloads run (domains)
  • Where governance is coordinated (fleet)

Summary

  • VCF Fleet = multi-cloud / multi-site management scope
  • VCF Instance = one complete VCF deployment
  • VCF Domains = functional partitions inside an instance
  • Lifecycle authority lives at the instance
  • Isolation and scale happen via domains
  • Governance and visibility roll up to the fleet
