Workload Domain

You create Workload Domains in VCF Operations to run consumer workloads.

For the first workload domain in your environment, VCF Operations deploys a vCenter instance and an NSX Manager instance in the Management Domain. For each additional workload domain, VCF Operations deploys an additional vCenter instance.

New Workload Domains can share the same NSX Manager cluster with an existing Workload Domain or you can deploy a new NSX Manager cluster.

Workload domains cannot use the NSX Manager cluster for the Management Domain.

A Workload Domain (or VI (virtual infrastructure) Workload Domain)

  • is created after Management Domain
  • hosts customer workloads
  • Typical separation:
    • Prod vs non-prod
    • Regulated vs unregulated
    • Performance-sensitive vs general

What Do VCF Domains Solve

A VCF Domain is to VCF what a Kubernetes cluster is to Kubernetes-based platforms.

It gives you:

  • Failure domain isolation
  • Security boundary
  • Upgrade blast-radius control
  • Operational independence

Without VCF Domains, everything would be patched, upgraded, and potentially broken together.

How VCF Automation sees a VCF Domain

In VCF Automation:

  • A VCF Domain is a backing infrastructure boundary
  • It is mapped to:
    • Organizations
    • Projects
    • Supervisor Namespaces

VCF Automation never creates or upgrades domain: It only consumes them.

How VCF Operations sees a VCF Domain

In VCF Operations:

  • A VCF Domain is an observed object
  • Used for:
    • Capacity planning
    • Health scoring
    • Performance analytics
    • Alerting

Operations has zero authority over its lifecycle.

Summary

  • VCF Domain is a foundational VCF concept
  • Defined and lifecycle-managed by SDDC Manager
  • Not a VCF Automation concept
  • Not a VCF Operations concept
  • BUT both products depend on it, yet neither owns it
  • Primary purpose: isolation, lifecycle control, and scalability
