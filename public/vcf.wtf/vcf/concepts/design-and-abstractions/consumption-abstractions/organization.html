Organization

In VCF Automation, an Organization is not a technical runtime boundary—It is a business-facing logical entity that defines who is allowed to consume cloud capabilities, under which rules, and with which visibility.

It is the outermost “contract boundary” for consumption, not execution.

An Organization is a logical tenant boundary that groups identity, policy, quotas, visibility, and approval flows for consuming infrastructure and platform services.

It answers:

  • Who is consuming?
  • What are they allowed to consume?
  • Under which business rules?
    —not how workloads actually run.

Organization is “Logical”, Namespace is the Real Boundary

Because Supervisor Namespaces are the actual enforcement boundary, a single Organization can define multiple tenants.

But those “tenants” are operational tenants, not consumption tenants.

There are two different kinds of tenancy in VCF:

Layer Kind of tenant Enforced by
Consumption layer Business / Contract tenant Organization
Runtime layer Technical / Isolation tenant Namespace

They are deliberately decoupled.

Namespace is the “real” Tenant Boundary

A Supervisor Namespace enforces:

  • RBAC
  • Resource quotas
  • Network isolation
  • Storage policies
  • Pod security boundaries
  • Runtime blast radius
    This is where:
  • Kubernetes says “no”
  • vSphere enforces limits
  • Workloads are actually prevented from crossing boundaries

Namespace = hard enforcement boundary

Organizations define intent and Governance.
An Organization:

  • Defines who is allowed to create tenants
  • Defines how many tenants
  • Defines under which rules
  • Defines approval and lifecycle policies

It’s closer to:

  • “Who is allowed to mint isolation boundaries?”
    not:
  • “Where isolation happens.”

An Organization Can Have Multiple Tenants

A single Organization can own:

  • Multiple Projects
  • Each Project can map to:
    • One Supervisor Namespace
    • Or several namespaces (depending on policy / model)
Organization: Acme Corp
├── Project: Acme-Prod     → ns-acme-prod
├── Project: Acme-Stage    → ns-acme-stage
├── Project: Acme-Dev      → ns-acme-dev
└── Project: Acme-Analytics → ns-acme-analytics

Each namespace is a real tenant boundary.

But from a business standpoint:

  • Acme Corp is still one customer:
    • One billing entity
    • One governance envelope
    • One policy domain

This construct enables the following scenarios (which would have been impossible otherwise).

1. One Customer, Many Isolated Environments

Classic need:

  • Dev / Stage / Prod
  • Regulated vs non-regulated
  • Different trust zones

All under:

  • One contract
  • One identity plane
  • One catalog

2. Platform Teams Acting as Internal Providers

Example:

  • Organization = “Platform Team”
  • Projects = internal product teams
  • Namespaces = per-team isolation

This would break if Organization == Namespace.

3. Managed Services / MSP model

Example:

  • Organization = External customer
  • Projects = customer environments
  • Namespaces = actual enforcement boundaries

Same model works for:

  • Internal IT
  • SaaS providers
  • Telco multi-tenant platforms

Namespace is the True Invariant

If something must be isolated, it must be isolated at the namespace level.

Anything above that:

  • Is governance
  • Is visibility
  • Is policy
  • Is economy

Organizations define who may create tenants; namespaces are the tenants.

Organization ≈ Cloud Account with Governance

Similar to:

  • AWS Account
  • Azure Subscription (with Management Groups layered above)
  • GCP Project (with Org/Folder hierarchy)

But in VCF, it’s explicitly framed as consumption-first, not infrastructure-first.

What Is an Organization

1. A Tenant Boundary

  • Separates consumers (e.g., companies, business units, external customers)
  • Enforces isolation in:
    • Identity and Access
    • Visibility
    • Quotas and Limits
    • Chargeback / showback

2. A Policy Anchor

Organizations are where policies attach, such as:

  • Who can request what
  • Approval workflows
  • Budget limits
  • Lease durations
  • Allowed regions / environments
  • Allowed workload types (VM, Kubernetes, services)

These policies then flow downward.

3. A Consumption Namespace

Everything inside an Organization is:

  • Visible only to that Organization (unless shared)
  • Governed consistently

This includes:

  • Projects
  • Blueprints / templates
  • Catalog items
  • Requests
  • Cost views
Organization
├── Projects
│   ├── Policies
│   ├── Quotas
│   ├── Users / Groups
│   └── Workload requests
│
└── (maps indirectly to)
    └── Cloud Zones
        └── Infrastructure (vSphere, Supervisor, VKS, etc.)
  • Infrastructure never belongs to an Organization
  • Consumption always does

Consumption Abstraction

Organization deliberately hides infrastructure topology.

From the Organization’s point of view:

  • There is no ESX
  • No cluster
  • No datastore
  • No vSphere Supervisor

There is only:

  • “What can I request?”
  • “What am I allowed to use?”
  • “Who approves this?”
  • “How much does it cost?”

This abstraction enables:

  • Multi-tenant platforms
  • Internal private clouds
  • Managed services
  • SaaS-like experience on top of VCF

Organization and Projects

If Organization is “who is the customer”,
then Project is “what they are working on”.

Typical mapping:

Organization: Acme Corp
├── Project: Acme-Prod
├── Project: Acme-Stage
└── Project: Acme-Dev

Projects:

  • Scope quotas
  • Bind to cloud zones
  • Eventually map to Supervisor Namespaces or runtime constructs

Organizations never do.

Summary

  • Organization = Governance + visibility
  • It exists to model consumption, not execution
  • It owns policy, identity, and economics
  • It deliberately abstracts away infrastructure
  • Everything technical happens below it, via Projects and Cloud Zones
