Policy

#tenancy #governance

A Policy in VCF is a Governance Rule that defines what is allowed, constrained, or enforced across infrastructure and workloads.
It exists across multiple layers of VCF, but it becomes a first-class, user-visible construct in VCF Automation.

If you think of VMware Cloud Foundation as a city:

  • Infrastructure → roads, power, zoning
  • Policies → zoning laws, speed limits, building codes
  • VCF Automation → city planning office that enforces them before you build
  • VCF Operations → inspectors that check compliance after things exist

1. Infrastructure-Level Policies

These are foundational, implicit, and exist below VCF Automation, often enforced automatically.

Examples:

  • Storage policies (SPBM): RAID, availability, performance tiers
  • Network policies: NSX security groups, microsegmentation rules
  • Host policies: DRS, HA, admission control

These are:

  • Defined in vSphere / NSX
  • Consumed transparently by higher layers
  • Not tenant-facing by default

2. **VCF Automation Policies

This is usually what people mean when they say “policy”.

This is where Policy becomes a named, explicit, governable object.

In VCF Automation, policies define constraints and guardrails for consumers.

Examples:

  • Who can deploy what
  • Where workloads can land
  • What sizes, images, or networks are allowed
  • Lease durations and approval requirements

Typical policy categories:

  • Placement policies: Which clusters, regions, zones
  • Sizing policies: CPU, memory, storage limits
  • Image policies: Approved OS images only
  • Network policies: allowed networks / isolation
  • Approval policies: manual gates for certain actions
  • Lease policies: TTL and reclamation rules

These policies are evaluated before provisioning.

3. Organization and Project Scoping

In VCF Automation, policies are usually scoped to:
- Organizations
- or Projects

They act as guardrails, not implementation details

Example:

“In Project Prod, only Gold storage, max 16 vCPU, and approval required.”

This is where policy intersects with consumption abstraction.

4. VCF Operations

In VCF Operations, policy ≠ enforcement.

VCF Operations uses policy indirectly.

In VCF Operations:

  • Policies become:
    • Alerts
    • Compliance checks
    • Drift detection
    • Optimization rules

VCF Operations answers:

“Is reality still compliant with the policy?”

VCF Automation answers:

“Should this action be allowed at all, in the first place?”

Summary

  • A Policy in VMware Cloud Foundation is a governance rule, not a single product feature
  • VCF Automation is where policies are:
    • Explicit
    • Named
    • Scoped
    • Evaluated before action
  • VCF Operations observes and reports compliance
  • Infrastructure layers actually enforce the low-level mechanics
